## 상세 요약

LLM API의 등장으로 데이터 과학자와 MLE의 역할이 줄어들었다는 인식이 퍼졌지만, Hamel Husain은 정반대의 주장을 펼친다. 모델 훈련은 업무의 핵심이 아니었으며, 진짜 핵심 업무인 "실험 설계, 확률적 시스템 디버깅, 좋은 메트릭 설계"는 LLM API를 쓴다고 사라지지 않는다.

**OpenAI의 하네스 엔지니어링과 데이터 과학:**
OpenAI가 발표한 Codex의 하네스(테스트+스펙+관찰 가능성)에는 메트릭이 핵심 구성 요소로 포함되어 있다. Andrej Karpathy의 auto-research도 같은 패턴이다. 하네스의 상당 부분이 데이터 과학이다.

**5가지 평가(Eval) 함정:**
1. **일반적 메트릭** — "유용성 점수", "일관성 점수" 같은 기성 메트릭은 진단에 쓸모없다. 애플리케이션별 메트릭("캘린더 스케줄링 실패", "인간 에스컬레이션 실패")이 필요하다.
2. **검증 안 된 판정자** — LLM-as-a-judge를 쓸 때 인간 라벨과 비교해 분류기처럼 검증해야 한다. 정확도 대신 precision/recall을 사용해야 한다.
3. **잘못된 실험 설계** — LLM에 "테스트 쿼리 50개 만들어줘" 하면 비대표적 데이터가 나온다. 실제 프로덕션 데이터에서 시작해야 한다. Likert 5점 척도 대신 이진 pass/fail을 사용해야 한다.
4. **나쁜 데이터와 라벨** — 도메인 전문가가 직접 라벨링해야 한다. "기준 표류(criteria drift)" 현상: 사용자는 LLM 출력을 실제로 보기 전까지 자신이 원하는 것을 모른다.
5. **과도한 자동화** — LLM이 배관 작업은 도와줄 수 있지만, 데이터를 들여다보는 작업을 대신할 수는 없다.

**핵심 메시지:**
모든 함정의 근본 원인은 데이터 과학의 기초를 건너뛰는 것이다. 트레이스 읽기 = EDA, 판정자 검증 = 모델 평가, 대표 테스트셋 구축 = 실험 설계, 도메인 전문가 라벨링 = 데이터 수집. 이름만 바뀌었지 일 자체는 동일하다.

## Soo에게 의미 있는 이유

AInD 컨설팅에서 가장 자주 마주칠 고객 과제가 바로 "LLM 평가를 어떻게 하느냐"이다. 이 글은 그 해답의 프레임워크를 제공한다. "데이터를 직접 봐라"라는 단순하지만 강력한 원칙, 그리고 LLM judge 검증 방법론은 PoC 단계에서 즉시 적용 가능한 실무 지식이다.
