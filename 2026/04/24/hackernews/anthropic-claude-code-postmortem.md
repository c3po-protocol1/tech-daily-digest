# Anthropic의 Claude Code 품질 이슈 포스트모템

> **출처:** [anthropic.com](https://www.anthropic.com/engineering/april-23-postmortem)
> **HN 토론:** [Hacker News](https://news.ycombinator.com/item?id=47878905) (491 points)
> **태그:** `#AI` `#ClaudeCode` `#Anthropic` `#Postmortem`

## 요약

지난 한 달간 Claude Code 응답 품질 저하 보고가 이어졌고, Anthropic이 원인을 추적한 결과 **3개의 별도 변경사항**이 동시에 영향을 미치고 있었음을 밝혔다. API와 추론 레이어는 영향받지 않았다. 모든 이슈는 4월 20일(v2.1.116) 기준으로 해결되었다.

### 이슈 1: 추론 노력도(reasoning effort) 기본값 변경 (3월 4일)
- `high`에서 `medium`으로 변경 → UI가 멈춘 것처럼 보이는 긴 레이턴시를 줄이려는 의도
- **잘못된 트레이드오프**였다. 사용자들이 "더 높은 지능을 원하고, 단순 작업에서만 낮은 노력도를 선택하겠다"고 피드백
- 4월 7일 되돌림. 현재 Opus 4.7은 `xhigh`, 다른 모델은 `high`가 기본값

### 이슈 2: 캐싱 최적화 버그 (3월 26일)
- 1시간 이상 유휴 세션에서 이전 추론(thinking)을 정리하여 비용을 줄이려는 의도
- **버그**: 한 번만 정리해야 하는데, **세션의 나머지 모든 턴에서 계속 추론을 삭제**
- 결과: Claude가 **건망증이 심하고, 반복적이며, 이상한 도구 선택**을 하게 됨
- 캐시 미스가 연쇄적으로 발생하여 사용량 한도도 예상보다 빨리 소진
- 내부에서도 재현이 어려웠음 (두 개의 관련 없는 실험이 버그를 가림)
- 4월 10일 수정. **Opus 4.7이 해당 PR에서 버그를 발견했고, Opus 4.6은 못 찾았다**

### 이슈 3: 시스템 프롬프트 변경 (4월 16일)
- Opus 4.7의 장황함을 줄이기 위해 추가: "도구 호출 사이 텍스트는 ≤25단어, 최종 응답은 ≤100단어"
- 여러 주 내부 테스트에서 회귀가 없어 자신감을 갖고 출시했지만, 더 넓은 평가 세트에서 **3% 하락** 발견
- 4월 20일 되돌림

### 재발 방지 대책
- 더 많은 내부 직원이 정확히 공개 빌드를 사용하도록 변경
- 모든 시스템 프롬프트 변경에 모델별 넓은 평가 세트 실행
- soak 기간, 점진적 롤아웃 추가
- Code Review 도구에 추가 레포지토리 컨텍스트 지원 추가
- **모든 구독자의 사용량 한도 리셋**

## Soo에게 의미 있는 이유

이 포스트모템은 **AI 도구 품질 관리의 복잡성**을 적나라하게 보여준다. 세 가지 독립적인 변경이 서로 다른 트래픽 슬라이스에서 서로 다른 스케줄로 영향을 미쳐 "넓고 일관성 없는 저하"로 나타났다. AInD 환경에서 에이전트 도구를 운영할 때, 이런 복합 장애의 디버깅 어려움을 미리 인식하고, 카나리 배포와 넓은 평가 세트의 중요성을 고객에게 전달해야 한다.
