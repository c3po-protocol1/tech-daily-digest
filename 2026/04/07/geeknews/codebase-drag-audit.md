## 요약

수년간의 코드베이스 감사(Audit) 결과를 바탕으로, 팀 속도 저하의 원인이 사람이 아닌 코드베이스임을 증명하는 5가지 신호와 진단 루브릭.

### 코드베이스 드래그(Codebase Drag)란

- 모든 작업이 필요 이상으로 오래 걸리게 만드는 코드베이스 상태
- 대시보드나 스프린트 리포트에 드러나지 않아 리더십이 "사람 문제"로 오해

### 5가지 신호 (각 0~2점 채점)

1. **사과식 예측**: 3일짜리 기능에 2주 견적 → 모듈 간 결합으로 blast radius 예측 불가
2. **배포 공포**: 금요일 배포 기피, 큰 단위 드문 릴리즈 → 롤백 전략 부재, 불안정 테스트
3. **'손대지 마' 파일**: 팀에서 건드리지 말라는 파일 존재 → 코드가 죽은 구역 중심으로 기형 성장
4. **커버리지 거짓말**: 80% 커버리지지만 핵심 모델 3개 테스트 전무 → "테스트가 통과해도 프로덕션 깨짐"
5. **첫 커밋까지의 시간**: 건강한 코드베이스 1~2일 vs 드래그 중 2주+ → setup rot

### 역설적 현상

> 숙련된 엔지니어일수록 위험을 더 잘 인식해 더 신중히 → 오히려 더 느려 보임 → 리더십이 인재 문제로 오인 → 프로세스만 추가 → 악순환

### 처방

- **4점 이상**: 다른 조치보다 코드베이스 직접 투자 선행
- **7점 이상**: 즉각적 구조적 개입 필요
- 가장 높은 점수 신호 **하나부터** 시작

## Soo에게 의미 있는 이유

AInD 컨설팅 현장에서 "AI 도구 도입 전에 코드베이스 건강도부터 진단해야 한다"는 논거로 활용 가능. AI 에이전트도 나쁜 코드베이스에서는 생산성이 떨어진다는 METR 연구 결과와 연결된다.
