## 요약

Andrej Karpathy가 "코드보다 개인 지식 저장소 구축에 토큰을 더 많이 쓰고 있다"며 공개한 LLM 기반 위키 아이디어 가이드.

### RAG vs LLM-Wiki의 핵심 차이

- **RAG**: 매 쿼리마다 원문에서 정보 재추출. 지식의 누적 없음
- **LLM-Wiki**: LLM이 영속적 위키를 점진적으로 구축·유지. 교차 참조가 이미 구성되어 있고, 모순은 이미 표시된 상태

### 3계층 아키텍처

1. **원문 소스**: 변경 불가(immutable), LLM은 읽기만. 진실의 원천
2. **위키**: LLM이 생성하는 마크다운 디렉터리. LLM이 완전 소유, 사용자는 읽기만
3. **스키마**: CLAUDE.md / AGENTS.md에 해당하는 설정 문서. 위키 구조와 컨벤션 정의

### 핵심 작업

- **인제스트(Ingest)**: 단일 소스 처리 시 10~15개 위키 페이지 업데이트 가능
- **쿼리(Query)**: 위키 대상 질문 → 인용 포함 답변 합성. 좋은 답변은 위키에 재저장
- **린트(Lint)**: 주기적 위키 상태 점검 — 모순, 고아 페이지, 누락 교차참조 탐지

### Karpathy가 말한 4가지 장점

- **명시성(Explicit)**: AI가 아는 것/모르는 것이 눈에 보임
- **데이터 소유권**: 로컬 컴퓨터에 저장, 특정 업체에 잠기지 않음
- **파일 우선**: 마크다운/이미지 등 범용 포맷. Unix 도구 전체 적용 가능
- **AI 선택 자유(BYOAI)**: 원하는 AI 자유롭게 연결

### 실제 사례 — Farzapedia

- 일기, Apple Notes, iMessage 대화 2,500건 → 400개 상세 위키 문서 자동 생성
- 에이전트가 index.md를 진입점으로 필요한 페이지 직접 탐색

## Soo에게 의미 있는 이유

OpenClaw 에이전트 아키텍처와 직접적으로 연결되는 개념. MEMORY.md, daily notes 시스템이 이미 LLM-Wiki의 축소판이며, 이를 Obsidian + 에이전트로 확장하면 AInD 컨설팅의 지식 관리 시스템으로 활용 가능. "에이전트 활용 능력이 21세기 핵심 스킬"이라는 Karpathy의 결론도 주목.
