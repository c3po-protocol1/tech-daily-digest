## TL;DR

- 2022~2026년, AI 개발 패러다임이 세 번 전환: **Prompt Engineering → Context Engineering → Harness Engineering**
- 각 전환의 동인은 "이전 패러다임이 약속한 것을 지키지 못했기 때문"
- Chad Fowler의 "Relocating Rigor" — **엔지니어링의 엄밀함은 사라지지 않는다, 이동할 뿐이다**
- 2026년의 핵심 메트릭: 프롬프트 품질이 아니라 **KV-cache hit rate**와 **하네스 복잡도**

## 1. 왜 지금 이 연대기인가

Epsilla의 메타포가 4년을 정확히 포착한다:
- **2022년**: 완벽한 이메일 작성법을 연구
- **2025년**: 받은편지함 관리를 학습
- **2026년**: 이메일 시스템 자체를 설계

핵심 논지는 하나. **엔지니어링의 엄밀함은 사라지지 않았다. 이동했을 뿐이다.** Chad Fowler가 "Relocating Rigor"에서 짚었듯, XP 운동이 "설계 문서 대신 테스트 코드"를 주장했을 때도, 동적 언어가 "컴파일러의 타입 검사 없이" 배포하겠다고 했을 때도, 엄밀함은 포기된 게 아니라 **더 높은 추상화 수준으로 이동**한 것이었다. 이번에도 마찬가지.

## 2. 프롬프트 엔지니어링의 시대 (2022-2024)

### 2.1 Copilot이 문을 열다

GitHub Copilot (2022.06) — 역사상 최초의 상업용 AI 코딩 어시스턴트:
- OpenAI Codex(GPT-3 기반), 에디터에서 "고스트 텍스트" 자동완성
- 개발자 88% 생산성 향상 체감, 반복 코드 작성 시간 절반 감소
- 2024년 초: 2,000만+ 사용자, 470만 유료 구독자, Fortune 100의 90% 도입

**Copilot의 진화가 세 시대의 축소판:**

| 시기 | 버전 | 핵심 변화 | 대응 시대 |
|------|------|-----------|-----------|
| 2022.06 | 초기 자동완성 | 현재 파일 기반 다음 줄 제안 | 프롬프트 시대 |
| 2023.11 | Chat + GPT-4 | 대화형 코드 질의·리팩토링 | 컨텍스트 전환 시작 |
| 2025.02 | Agent Mode | 멀티파일 편집, 터미널 실행, 린트-수정 루프 | 하네스 시대 |
| 2025.05 | Coding Agent | Issue → 코드 → 테스트 → PR 자동 생성 | 하네스 심화 |

### 2.2 ChatGPT와 "영어가 곧 프로그래밍 언어다"

- 2022.11 ChatGPT 공개 — 5일 만에 100만, 2개월 만에 1억 사용자
- Karpathy의 "Software 3.0": 자연어 지시문 자체가 프로그램인 패러다임
- 모든 것이 **"어떻게 말하느냐"** 의 문제로 보였던 시기

### 2.3 학문적 토대: 추론을 프롬프트로 유도하다

**Chain-of-Thought (Wei et al., 2022)**
- "단계별로 생각하라(Let's think step by step)" → PaLM 540B 수학 정확도 **17.9% → 58.1%**
- 프롬프트 한 줄의 위력이 증명된 순간

**ReAct (Yao et al., 2022)**
- 생각(Thought) + 행동(Action) + 관찰(Observation) 루프
- ALFWorld 34% 절대 향상, WebShop 10% 향상
- **오늘날 모든 AI 에이전트의 원형** — Claude Code, Cursor Agent, Copilot Coding Agent 모두 이 루프의 변형

**Tree-of-Thought (Yao et al., 2023)**
- 여러 추론 경로를 동시 탐색 + 백트래킹 → 매혹적이나 **비용 폭발**

**Self-Refine & Reflexion (2023)**
- 모델이 자기 출력을 비판·개선하는 루프
- 근본 한계: **피드백 품질이 모델 자신의 능력에 종속** (실력 없는 학생이 자기 답을 채점)
- 이 문제가 2026년 Anthropic의 3-에이전트 아키텍처에서 다시 등장 → "채점은 다른 사람이 해야 한다"는 결론에 **3년**이 걸림

### 2.4 Andrew Ng의 4대 에이전틱 디자인 패턴 (2024.03)

Sequoia AI Ascent 발표. 그간의 연구를 실무 프레임워크로 종합:

1. **Reflection (반성)** — 가장 안정적. 같은 모델이 다른 페르소나로 검토
2. **Tool Use (도구 사용)** — ReAct의 실전판. 챗봇과 에이전트의 결정적 차이
3. **Planning (계획)** — 강력하나 가장 불안정. "디테일에 악마가 있다"
4. **Multi-Agent Collaboration** — 2024년 시점 가장 초기, 가장 큰 가능성

**핵심 발견**: "GPT-3.5에 에이전틱 워크플로우를 씌우면 특정 벤치마크에서 **GPT-4 제로샷보다 낫다**." → 모델 바깥의 시스템이 중요하다는 힌트

Anthropic "Building Effective Agents" (2024.12)에서 워크플로우(흐름 고정)와 에이전트(자율 결정) 구분. 핵심 메시지: **"가능한 한 단순하게 시작하라."**

### 2.5 프롬프트의 벽

프롬프트가 "기존 코드를 재사용하라"고 써놓아도, **컨텍스트 윈도우에 해당 유틸리티 파일이 없으면** 에이전트는 그 존재 자체를 모른다. Mitchell Hashimoto가 "Blind Prompting"이라 명명 — 측정·테스트 없이 시행착오에만 의존하는 프롬프트 작성. **연금술에 가까웠다.**

**프롬프트 시대의 사인(死因)**: 엄밀함이 있어야 할 곳은 프롬프트 텍스트가 아니라, **프롬프트가 소비하는 컨텍스트 전체**였다.

### 3.3 Vibe Coding — 프롬프트 엔지니어링의 논리적 극한

Karpathy (2025.02): "diff도 거의 안 봅니다." → **"Vibe coding"** 명명

- YC Winter 2025 배치 25%가 코드베이스 95%를 AI 생성
- Collins 사전 2025 올해의 단어 선정

**숙취 (2025.09, Fast Company)**:
- 3개월 전 AI로 만든 MVP → 버그 수정하려니 **아무도 코드를 이해 못 함**

**Chad Fowler의 "Relocating Rigor" 3원칙:**
1. "반드시 지켜져야 하는 규칙"은 **코드로 강제하라** (주석이 아닌 타입/assertion)
2. 실패는 **즉각적이고 시끄러워야** 한다
3. 엔지니어의 역할: 코드 타이핑 → **의도 명세 + 결과 검증**

Vibe coding은 이 세 가지를 모두 무시한 결과.

## 4. 컨텍스트 엔지니어링의 시대 (2025 중반~)

### 4.1 기원: 2025년 6월

- **Tobi Lütke (2025.06.19)**: "프롬프트 엔지니어링보다 컨텍스트 엔지니어링이 핵심 역량을 더 잘 설명"
- **Karpathy**: "컨텍스트 엔지니어링은 다음 단계에 딱 필요한 정보로 컨텍스트 윈도우를 채우는 기술이자 과학" + **"두꺼운 레이어의 작은 한 조각일 뿐"** (이 표현이 8개월 뒤 하네스 엔지니어링으로 돌아옴)

핵심 질문의 전환: **"어떤 말을 해야 하나" → "어떤 정보를 넣어야 하나"**

### 4.2 LLM-as-OS: 운영체제 비유

| 기존 OS | LLM OS 대응 |
|---------|-------------|
| 커널 | LLM 추론 엔진 |
| **RAM** | **컨텍스트 윈도우** |
| 파일 시스템 | RAG / 벡터 DB |
| 시스템 호출 | Tool Call / API |
| 프로세스 관리 | 멀티 에이전트 오케스트레이션 |

프롬프트 = OS에 입력하는 명령어 한 줄. 진짜 성능 = **RAM(컨텍스트 윈도우)에 무엇을 올려놓는가**.

### 4.3 핵심 원칙

#### Anthropic의 4대 전략: Write / Select / Compress / Isolate

1. **Write (작성)**: 시스템 프롬프트를 명확하고 구조화된 형태로. "무엇을 말할 것인가"가 아니라 **"어떻게 구조화할 것인가"**
2. **Select (선택)**: 전체 FAQ가 아니라 관련 문서만 골라 컨텍스트에. "Lost-in-the-Middle" 방지, **신호 대 잡음비** 최적화
3. **Compress (압축)**: 20턴 대화를 요약. 정보 보존율 80%+ 유지, 토큰 사용량 대폭 감소
4. **Isolate (격리)**: 진단 서브에이전트에게 별도 위임. 메인 컨텍스트 오염 방지

#### Google ADK의 컨텍스트 스택 아키텍처

3대 설계 원칙:
1. **저장과 표현의 분리** — 원본(Session)은 보존, API에 보내는 건 최적화된 "작업 컨텍스트(Working Context)"
2. **명시적 변환 파이프라인** — "시스템 프롬프트 삽입 → 관련 히스토리 선별 → 도구 결과 요약 → 토큰 예산 확인" 순서
3. **기본 스코핑** — 서브에이전트에게 전체 대화가 아닌 관련 컨텍스트만 전달 (최소 권한)

컨텍스트 윈도우 이분법:
- **안정 접두어 (Stable Prefix)**: 시스템 프롬프트, 에이전트 정체성, 도구 정의 — 자주 변하지 않는 것을 **앞쪽**에
- **동적 접미어 (Variable Suffix)**: 최신 입력, 도구 출력 — 자주 변하는 것을 **뒤쪽**에

### 4.4 KV-cache: 프로덕션의 핵심 메트릭

Manus 팀이 **에이전트 프레임워크를 네 번 갈아엎은** 끝에 도달한 결론: 진짜 병목은 **컨텍스트 관리**.

**KV-cache 작동 원리**: 이전 요청의 프롬프트 접두어가 현재 요청과 동일하면, 어텐션 Key-Value를 재계산 없이 캐시 재사용. Claude Sonnet 기준 **캐시 히트 시 비용 1/10**. **접두어 토큰 하나만 바뀌어도 이후 전체 캐시 무효화.**

→ 프롬프트의 "품질"보다 프롬프트의 **"안정성"** 이 프로덕션에서 훨씬 중요. 2년간 프롬프트를 다듬었는데, 정작 중요한 건 **프롬프트를 건드리지 않는 것**.

**Manus 팀 5대 실전 원칙:**
1. **KV-cache 최적화와 접두어 안정성 최우선**
2. **도구 목록을 중간에 바꾸지 말 것** — 활성/비활성 전환으로 대체
3. **통제된 다양성** — 접두어 안정성 범위 내 변주로 루프 방지
4. **프로덕션 우선** — fine-tuning보다 컨텍스트 엔지니어링이 반복 속도 빠름
5. **모델 교체해도 하네스 유지** — 특정 모델 비종속 설계

**HumanLayer의 12-Factor Agents 핵심 3가지:**
1. 컨텍스트 **40% 이상 차면 "dumb zone"** — 지시 따르기 능력 급락
2. 자연어보다 **구조화된 출력** (JSON)
3. **Human-in-the-loop를 설계 시점에** 포함

### 4.5 에이전틱 인프라

**MCP (Model Context Protocol, Anthropic 2024.11)**
- LLM ↔ 외부 도구 통신 표준. USB 이전 vs 이후의 차이
- 2025.03 OpenAI 도입, 2025.04 Google 지원, 2025.12 Linux Foundation 기부
- 1년 만에 월간 SDK 다운로드 **9,700만 건**, **10,000+ 커뮤니티 서버**
- A2A (Agent-to-Agent Protocol, Google): MCP가 세로축(에이전트↔도구)이면 A2A는 가로축(에이전트↔에이전트)

**스킬(Skills)**: 도구가 함수라면, 스킬은 모듈. **지연 로딩** — 목록만 안정 접두어에, 상세 정의는 필요 시 동적 접미어에 로딩 → KV-cache 보존

**서브에이전트**: "Isolate" 전략의 구현체. 메인 컨텍스트 깨끗하게 유지

**Swarm (OpenAI 2024.10)**: 루틴(routine) + 핸드오프(handoff). 중앙 오케스트레이터 없이 자율 위임

**Context Hub (Andrew Ng)**: "전향성 기억상실증" 해결. `chub get stripe --lang python`으로 최신 API 문서 주입
- 큐레이션된 문서 (검증됨, 68개 API로 시작)
- 스택 특화 (`--lang py/js`)
- 어노테이션 지속성 (세션 간 교훈 유지)

**메모리**: 파일 기반(CLAUDE.md), 벡터 DB, 구조화 JSON. Anthropic의 "컨텍스트 컴팩션" — 다단계 압축 파이프라인

### 4.6 컨텍스트의 벽 — 그래도 부족했다

1. **단일 턴의 한계**: 15번째 턴에서 3번째 턴의 도구 결과가 압축되어 사라졌다면? **컨텍스트가 아니라 시스템의 문제**
2. **에러 복구 부재**: 도구 실패, 환각, 비용 폭주 → 비용 인식 에러 복구, 수확 체감 감지, 서킷 브레이커 필요
3. **보안**: Willison의 "Lethal Trifecta" — 프롬프트 인젝션 한 방에 회사 데이터 유출 가능

**컨텍스트 시대의 사인**: 완벽한 컨텍스트를 구성해도, **소비하는 시스템의 설계가 허술하면** 여전히 실패. 필요 조건이었지 충분 조건이 아니었다.

## 5. 하네스 엔지니어링의 시대 (2026~)

### 5.1 기원: 2026년 2월의 동시 발견

Mitchell Hashimoto "My AI Adoption Journey":
> **"에이전트가 실수할 때마다, 그 실수가 구조적으로 다시 발생할 수 없도록 시스템을 변경한다."**

2주 내에 **OpenAI, Martin Fowler/Birgitta Böckeler, Ethan Mollick이 독립적으로 같은 결론** → 과학사의 "multiple discovery". 같은 벽에 부딪히고 있었기 때문.

**핵심 공식 (Fowler/Böckeler)**:
> **에이전트 = 모델 + 하네스**

- Philipp Schmid: 모델 = CPU, **하네스 = 운영체제**
- Mollick: 말(horse)이 아무리 강해도 **고삐·안장·굴레(하네스)** 없이는 밭을 갈 수 없다
- Willison: **"에이전트 = 루프 안에서 도구를 실행하는 LLM"** — 루프를 제어하는 것이 하네스

### 5.2 하네스의 해부학: 네 사분면

|  | 피드포워드 (사전 유도) | 피드백 (사후 교정) |
|--|----------------------|-------------------|
| **결정론적** | **가이드**: AGENTS.md, .cursorrules, 코딩 컨벤션 | **연산적**: 컴파일러, 타입 체커, 린터 |
| **비결정론적** | **시스템 프롬프트**: 역할 정의, 행동 제약, few-shot | **추론적**: LLM-as-a-judge, 의미론적 코드 리뷰 |

- **가이드 (좌상)**: 비용 ≈ 0, 강제성 없음. AGENTS.md에 "React 대신 Vanilla JS"
- **연산적 (우상)**: 기계적으로 잡는다. 컴파일 에러 → 자동 수정. **OpenAI Codex 팀 특히 강조**
- **시스템 프롬프트 (좌하)**: 결정론적 규칙으로 못 잡는 뉘앙스. "확실하지 않으면 확인을 구하라"
- **추론적 (우하)**: "컴파일되지만 의미적으로 틀린" 경우. Anthropic Evaluator

프로덕션 하네스 = **네 사분면 레이어링 = 방어의 깊이(defense in depth)**

### 5.3 실전 사례 세 개

#### 사례 1: Anthropic 3-에이전트 아키텍처

핵심 발견: **에이전트는 자기 작업을 정확히 평가할 수 없다.** GAN에서 영감 → 만드는 놈과 평가하는 놈 분리:

1. **Planner**: 간단한 프롬프트 → 상세 제품 스펙 확장. 기술 세부사항이 아닌 야심찬 범위에 집중. 너무 구체적인 지시 → 캐스케이딩 에러
2. **Generator**: 한 번에 하나의 기능씩 구현 (React, Vite, FastAPI, SQLite). 스프린트 단위 → 컨텍스트 주기적 리셋
3. **Evaluator**: Playwright E2E 테스트. 제품 깊이·기능·디자인·코드 품질 기준 채점. 미달 시 구체적 피드백과 함께 반환

비용: 단독(20분, $9) vs 풀 하네스(**6시간, $200** — 22배). 하지만 **비용의 위치 이동** — 사후 사람 수정 비용 → 사전 시스템 검증 비용

#### 사례 2: OpenAI Codex 실험

규모: **5개월, 3→7명, 수동 코드 0줄, 생성 코드 ~100만줄, PR ~1,500개, 수동 대비 ~10배 속도**

엔지니어들이 한 세 가지:
1. **저장소 지식의 시스템화** — "에이전트에게 보이지 않는 지식은 존재하지 않는 것과 같다." Slack 합의, 머릿속 원칙 → 마크다운과 코드로 문서화
2. **기계적 강제성** — 커스텀 린터 + 구조적 테스트로 아키텍처 규칙 강제. 인간 리뷰어 → 기계적 강제로 대체
3. **점진적 공개** — 1,000페이지 매뉴얼이 아니라 **지도(map)를 주고** 필요한 정보를 직접 찾아가게

핵심: **실수가 나면 에이전트를 탓하지 않고, 하네스를 개선한다.**

#### 사례 3: Ralph 패턴 — 밤새 돌아가는 에이전트

Geoffrey Huntley 제안. PRD 정의 → AI 코딩 도구를 루프 안에서 반복 실행 → PRD 모든 항목 완료까지:
- 매 이터레이션 **클린 컨텍스트로 새로 시작** (윈도우 오염 원천 차단)
- 상태는 **git 히스토리, progress.txt, prd.json**으로 유지
- 에이전트의 기억 = 컨텍스트 윈도우가 아닌 **파일 시스템**
- 2개월 만에 GitHub 스타 12,000+

### 5.4 보안: Lethal Trifecta와 2인 규칙

**Willison의 Lethal Trifecta** — 세 가지가 동시에 있으면 보안 사고 필연:
1. 신뢰할 수 없는 입력 처리 (외부 웹, 이메일)
2. 민감한 시스템/데이터 접근 (개인정보, 내부 API)
3. 상태 변경 능력 (이메일 발송, 파일 삭제)

**Meta AI의 Rule of Two** (Chromium 보안 정책에서 차용):
- 세 가지 중 **최대 두 가지만** 동시 허용
- 세 가지 모두 필요 → **반드시 human-in-the-loop**

## 6. 종합 비교와 교훈

| 차원 | Prompt Engineering | Context Engineering | Harness Engineering |
|------|-------------------|--------------------|--------------------|
| 시대 | 2022-2024 | 2025 | 2026~ |
| 핵심 질문 | "어떤 말을 해야 하나?" | "어떤 정보를 넣어야 하나?" | "어떤 시스템을 만들어야 하나?" |
| 은유 | 이메일 작성 | 받은편지함 관리 | 이메일 시스템 설계 |
| OS 비유 | 명령어 한 줄 | RAM 관리 | **운영체제 전체** |
| 핵심 메트릭 | 응답 품질 (주관적) | KV-cache hit rate | 태스크 완료율, 비용/태스크 |
| 실패 모드 | Blind prompting | 컨텍스트 오염 | 오케스트레이션 버그, 보안 사고 |
| 엄밀함의 위치 | 프롬프트 텍스트 | 컨텍스트 윈도우 구성 | **전체 시스템 아키텍처** |
| 필요 역량 | 언어 감각 + 도메인 지식 | 정보 아키텍처 | **시스템 설계 + 보안** |

### 핵심 교훈

**각 시대는 이전 시대를 대체하지 않고 포함(subsume)한다.** 하네스 안에 컨텍스트가, 그 안에 프롬프트가 있다. "프롬프트 엔지니어링이 죽었다"는 틀렸다 — **죽은 게 아니라 승진한 것이다.**

**하네스는 뜯어낼 수 있어야(rippable) 한다.** 모델이 발전하면 하네스의 일부가 불필요해진다. "무엇을 만들 것인가"만큼 **"무엇을 쉽게 제거할 수 있게 만들 것인가"** 가 중요.

### 전망: 엄밀함의 다음 정류장

1. **Guardian Agent** — 에이전트를 감시하는 에이전트. 엄밀함이 "실행"에서 "감시"로 이동
2. **평가 엔지니어링** — "behavior beats benchmarks". 검증 불가능한 보상(글 품질, 디자인 미감) 평가가 핵심 난제
3. **지식 엔진** — 벡터 검색 → 코드 그래프 + 커밋 히스토리 + 메모리 통합. **"왜 이 아키텍처를 선택했는가"** 를 에이전트가 이해
