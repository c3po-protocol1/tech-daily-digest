## 상세 요약

Hamel Husain이 PyAI Conf에서 발표한 "데이터 사이언티스트의 역습"을 기반으로 한 글로, LLM API 시대에 데이터 사이언티스트/MLE가 AI 출시 핵심 경로에서 밀려났지만, 실제로 가장 중요한 작업은 여전히 데이터 과학의 영역이라는 주장을 펼친다.

**5가지 핵심 함정:**
1. **범용 메트릭:** 오프더셸 eval 프레임워크의 도움성/일관성/환각 점수는 범용적이라 쓸모없다. 실제 트레이스를 읽고, 커스텀 뷰어를 만들어 데이터를 직접 봐야 한다. 애플리케이션 특화 메트릭("캘린더 스케줄링 실패", "사람에게 에스컬레이션 실패" 등)이 필요.
2. **검증되지 않은 판사:** LLM을 판사로 사용할 때 판사의 신뢰성을 검증하지 않는 팀이 대부분. 인간 레이블과 대조하여 train/dev/test 분할로 분류기처럼 평가해야 한다. 정확도보다 정밀도와 재현율을 사용할 것.
3. **잘못된 실험 설계:** LLM으로 합성 데이터 50개 생성하는 대신, 실제 프로덕션 데이터 기반으로 차원을 식별하고 엣지 케이스를 주입해야 한다. Likert 5점 척도 대신 범위가 한정된 이진 합격/불합격 기준 사용.
4. **나쁜 데이터와 레이블:** 도메인 전문가가 직접 레이블링해야 한다. 레이블링 과정 자체가 "무엇이 중요한가"를 발견하는 과정(criteria drift).
5. **과도한 자동화:** LLM이 배관 작업은 도와줄 수 있지만, 데이터를 직접 보는 것은 대체할 수 없다.

**핵심 매핑:** 트레이스 분석 = EDA, LLM 판사 검증 = 모델 평가, 대표 테스트셋 구축 = 실험 설계, 도메인 전문가 레이블링 = 데이터 수집, 프로덕션 모니터링 = Production ML.

## Soo에게 의미 있는 이유

AInD 컨설팅에서 가장 빈번하게 만나게 될 문제가 바로 "AI 시스템을 어떻게 평가하고 개선할 것인가"다. 이 글이 제시하는 5가지 함정과 데이터 사이언스 기본기의 중요성은 컨설팅 방법론의 핵심 구성요소가 될 수 있다. 특히 "데이터를 직접 봐야 한다"는 메시지는 아무리 강조해도 지나치지 않다.
