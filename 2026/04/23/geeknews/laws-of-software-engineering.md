## 상세 요약

소프트웨어 시스템, 팀, 의사결정에 영향을 미치는 **56가지 원칙과 패턴**을 한곳에 모은 컬렉션 사이트가 공개되었다. 주니어부터 시니어까지 경력 수준별로 분류되어 있으며, 아키텍처/팀/계획/품질/설계/의사결정 카테고리로 나뉜다.

**주요 법칙들 (카테고리별 하이라이트):**

**팀:**
- Conway's Law: 조직이 설계하는 시스템은 자신의 커뮤니케이션 구조를 반영한다
- Brooks's Law: 지연된 프로젝트에 인력을 추가하면 더 늦어진다
- Dunbar's Number: 한 사람이 유지할 수 있는 안정적 관계는 약 150개
- Price's Law: 전체 참여자의 제곱근이 50%의 작업을 수행한다
- Bus Factor: 프로젝트를 심각한 위기에 빠뜨릴 수 있는 최소 팀원 수

**아키텍처:**
- Hyrum's Law: 충분한 API 사용자가 있으면 모든 관찰 가능한 동작에 누군가가 의존한다
- Gall's Law: 작동하는 복잡한 시스템은 반드시 작동하는 간단한 시스템에서 진화한 것이다
- Tesler's Law: 모든 애플리케이션에는 제거할 수 없는 고유 복잡성이 있다
- CAP Theorem: 분산 시스템은 일관성, 가용성, 파티션 내성 중 두 가지만 보장 가능
- Zawinski's Law: 모든 프로그램은 메일을 읽을 수 있을 때까지 확장하려 한다

**계획:**
- 90-90 Rule: 코드의 첫 90%가 개발 시간의 90%를 차지하고, 나머지 10%가 또 다른 90%를 차지한다
- Hofstadter's Law: 항상 예상보다 오래 걸리며, Hofstadter의 법칙을 고려해도 마찬가지다
- Goodhart's Law: 측정값이 목표가 되면 좋은 측정값이 되지 못한다

**품질:**
- Kernighan's Law: 디버깅은 코드를 작성하는 것보다 두 배 어렵다
- Technical Debt: 소프트웨어 개발을 느리게 만드는 모든 것

각 법칙은 카드 형태로 클릭하면 상세 설명을 볼 수 있으며, 검색과 필터링이 가능하다.

## Soo에게 의미 있는 이유

AInD 컨설팅 TF를 이끌면서 팀 구조, 아키텍처 결정, 프로젝트 계획을 할 때 이 법칙들은 **의사결정의 근거**가 된다. 특히 Conway's Law는 AI 에이전트 팀 구조 설계에, Hyrum's Law는 AI API 설계에, Gall's Law는 점진적 AI 도입 전략에 직접 적용된다.
