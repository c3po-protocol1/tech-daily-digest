## 핵심 요약

AI 에이전트와 외부 API 사이에 위치하는 **보안 프록시 게이트웨이** 프로젝트 onecli가 GitHub에 공개되었다. 이 도구의 핵심 가치는 명확하다 — **API 키를 에이전트에 직접 노출하지 않으면서, 에이전트가 API를 사용할 수 있게 해주는 것**이다.

현재 AI 에이전트 생태계에서 시크릿 관리는 가장 취약한 지점 중 하나이다. 에이전트에게 API를 호출하는 능력을 부여하려면 API 키를 제공해야 하는데, 이 키가 에이전트의 컨텍스트에 노출되면 프롬프트 인젝션 공격을 통해 유출될 수 있다. 환경 변수로 키를 주입하더라도, 에이전트가 `env` 명령이나 프로세스 메모리 접근을 통해 키를 읽을 수 있는 위험이 존재한다.

onecli는 이 문제를 **아키텍처 수준**에서 해결한다. 에이전트는 API 이름(예: "github", "openai", "slack")만 지정하여 요청을 보내면, onecli 게이트웨이가 요청을 가로채고 내부 암호화 저장소에서 해당 API 키를 꺼내 요청 헤더에 자동으로 주입한 후, 대상 API 서버로 전달한다. 에이전트의 관점에서는 키 없이 API를 호출하는 것이고, 실제 키는 게이트웨이 내부에서만 존재한다.

기술적 구현 세부사항도 주목할 만하다. 키는 AES-256-GCM으로 암호화되어 로컬 저장소에 보관된다. 각 API에 대한 접근 정책(어떤 에이전트가 어떤 API에 접근 가능한지, 요청 빈도 제한, 허용되는 엔드포인트 등)을 설정 파일로 관리할 수 있다. 요청/응답 로그를 남겨 감사 추적이 가능하며, 이상 패턴(갑작스런 요청 급증, 비정상적 엔드포인트 접근 등)을 감지하는 기본적인 모니터링도 제공한다.

이 패턴은 AI 에이전트 보안의 **"제로 트러스트(Zero Trust)"** 원칙의 구현이라고 볼 수 있다. 에이전트를 신뢰하지 않되, 필요한 기능은 안전하게 제공하는 것이다. 마치 기업 네트워크에서 직원에게 VPN 접근 권한을 부여하되 모든 트래픽을 모니터링하는 것과 유사하다.

규모가 작은 프로젝트(6포인트)이지만, 이 아키텍처 패턴 자체는 프로덕션 에이전트 배포에서 반드시 고려해야 할 핵심 설계 요소이다. 에이전트에게 시크릿을 직접 제공하는 현재의 일반적 관행은 보안 사고가 터지기 전까지만 유지될 수 있으며, 게이트웨이 패턴이 표준이 될 가능성이 높다.

## Soo에게 의미 있는 이유

AInD 컨설팅에서 기업에 AI 에이전트를 도입할 때, 시크릿 관리는 보안팀이 가장 먼저 제기하는 우려 사항이다. 게이트웨이 패턴의 존재를 알고 있으면 "에이전트에 API 키를 주는 건 위험하지 않나요?"라는 질문에 아키텍처 수준의 해법을 제시할 수 있다.
