## 1. 전체 개발 플로우 (End-to-End Development Flow)

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        AI-NATIVE DEVELOPMENT FLOW                          │
│                                                                             │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌───────┐ │
│  │  PHASE 1  │───▶│  PHASE 2  │───▶│  PHASE 3  │───▶│  PHASE 4  │───▶│PHASE5│ │
│  │  Design   │    │  Prompt   │    │  Agent    │    │  Review   │    │Deploy│ │
│  │  & Plan   │    │  & Task   │    │  Execute  │    │  & QA     │    │& Mon │ │
│  └──────────┘    └──────────┘    └──────────┘    └──────────┘    └───────┘ │
│   [Human]         [Human]         [Agent]         [Agent+Human]  [Agent]   │
└─────────────────────────────────────────────────────────────────────────────┘
```

### Phase 1: 설계 & 계획 (Human-Led)
```
 👤 엔지니어
  │
  ├── 1a. 목표 정의 (High-Level Goal)
  │    └── "사용자 인증 시스템을 구축하라"
  │
  ├── 1b. 깊이 우선 문제 분해 (Depth-First Decomposition)
  │    ├── 목표 → 서브 목표 → 빌딩 블록
  │    ├── "어떤 기능(capability)이 없는가?" 를 반복 질문
  │    └── 의존성 순서대로 정렬
  │
  ├── 1c. 아키텍처 결정 & 문서화
  │    ├── docs/design/ → 설계 문서 (검증 상태 포함)
  │    ├── docs/architecture/ → 레이어링 규칙
  │    └── docs/plans/ → 실행 계획 + 결정 로그
  │
  └── 1d. 골든 원칙(Golden Principles) 정의
       ├── 의존성 방향 규칙
       ├── 네이밍 컨벤션
       ├── 구조화된 로깅 규칙
       └── 파일 크기 제한 등
```

### Phase 2: 프롬프트 & 태스크 생성 (Human-Led)
```
 👤 엔지니어
  │
  ├── 2a. 프롬프트 작성
  │    ├── 빌딩 블록 단위로 프롬프트 구성
  │    ├── AGENTS.md (목차, ~100줄)로 에이전트 안내
  │    └── docs/ 디렉토리의 상세 컨텍스트 참조 지시
  │
  ├── 2b. 필요한 도구/추상화 준비
  │    ├── 에이전트가 사용할 수 있는 도구 확인
  │    ├── 부족한 도구가 있으면 → 도구 구축 프롬프트 먼저 실행
  │    └── 테스트 가능한 환경 확인
  │
  └── 2c. 태스크 큐에 등록
       ├── PR 단위로 분리된 작업
       └── 병렬 실행 가능한 작업 식별
```

### Phase 3: 에이전트 실행 (Agent-Led)
```
 🤖 Codex Agent
  │
  ├── 3a. 컨텍스트 로딩
  │    ├── AGENTS.md → 목차 읽기
  │    ├── docs/ → 관련 설계 문서, 아키텍처 규칙 로딩
  │    └── 기존 코드 패턴 분석
  │
  ├── 3b. 코드 생성
  │    ├── 애플리케이션 로직
  │    ├── 테스트 코드
  │    ├── CI 설정
  │    └── 문서
  │
  ├── 3c. 자체 검증 (Self-Verification)
  │    ├── git worktree에서 앱 부팅
  │    ├── Chrome DevTools Protocol로 UI 검증
  │    │    ├── DOM 스냅샷
  │    │    ├── 스크린샷 캡처
  │    │    └── 네비게이션 테스트
  │    ├── 관측 스택(LogQL/PromQL)으로 성능 검증
  │    └── 린터/구조 테스트 실행
  │
  ├── 3d. PR 생성
  │    └── 단일 실행이 최대 6시간까지 지속 가능
  │
  └── 3e. 실패 시 → "어떤 capability가 부족한가?" → Phase 2로 피드백
```

### Phase 4: 리뷰 & QA (Agent + Human)
```
 🤖 Agent-to-Agent Review          👤 Human Review (선택적)
  │                                  │
  ├── 4a. 자동 코드 리뷰             ├── 4d. 고수준 설계 검토
  │    ├── 아키텍처 규칙 준수         │    └── 아키텍처 일관성 판단
  │    ├── 의존성 방향 검증           │
  │    └── 골든 원칙 준수             ├── 4e. 취향(Taste) 판단
  │                                  │    └── "이것이 올바른 추상화인가?"
  ├── 4b. CI/린터 자동 실행           │
  │    ├── 커스텀 린터 (에이전트용     └── 4f. 골든 원칙 업데이트
  │    │   에러 메시지 포함)               └── 새로운 패턴 발견 시
  │    ├── 구조 테스트                      원칙에 인코딩
  │    └── 통합 테스트
  │
  └── 4c. 테스트 실패 처리
       ├── 플레이크 → 후속 실행으로 해결
       └── 실제 실패 → 수정 PR 자동 생성
```

### Phase 5: 배포 & 지속적 유지보수 (Agent-Led)
```
 🤖 Background Agents (상시 가동)
  │
  ├── 5a. 자동 머지
  │    ├── 최소 차단 머지 게이트
  │    └── 짧은 수명의 PR
  │
  ├── 5b. AI 슬롭 관리 (Entropy GC)
  │    ├── 백그라운드 Codex가 편차 스캔
  │    ├── 품질 등급 업데이트
  │    ├── 타겟 리팩토링 PR 생성
  │    └── 대부분 1분 이내 리뷰 → 자동 머지
  │
  ├── 5c. 문서 정원사 (Doc-Gardening Agent)
  │    ├── 오래된 문서 스캔
  │    ├── 수정 PR 자동 생성
  │    └── 지식 기반의 최신성 유지
  │
  └── 5d. 관측 & 모니터링
       ├── 로그(LogQL) / 메트릭(PromQL)
       └── 성능 저하 감지 → 자동 수정 PR
```

## 2. 각 단계별 핵심 작업 (Critical Tasks)

| 단계 | 핵심 작업 | 담당 | 중요도 | 이유 |
|------|----------|------|--------|------|
| **Phase 1** | 깊이 우선 문제 분해 | 👤 Human | ⭐⭐⭐⭐⭐ | 에이전트는 모호한 목표를 처리할 수 없음. 분해의 품질이 전체 결과를 결정 |
| **Phase 1** | 골든 원칙 정의 | 👤 Human | ⭐⭐⭐⭐⭐ | 이것 없이는 에이전트가 저장소의 나쁜 패턴까지 복제 |
| **Phase 1** | 아키텍처 레이어링 설계 | 👤 Human | ⭐⭐⭐⭐⭐ | "보통 수백명일 때 하는 것을, 에이전트와 함께라면 초기 필수조건" |
| **Phase 2** | AGENTS.md 목차 설계 | 👤 Human | ⭐⭐⭐⭐ | 에이전트의 내비게이션. 100줄 초과 시 효과 급감 |
| **Phase 2** | docs/ 구조화 | 👤 Human | ⭐⭐⭐⭐ | 단일 파일 = 부패. 구조화된 디렉토리 = 기계적 검증 가능 |
| **Phase 3** | git worktree 격리 환경 | 🤖 Agent | ⭐⭐⭐⭐⭐ | 에이전트가 독립적으로 테스트 가능해야 병목 해소 |
| **Phase 3** | Chrome DevTools 연결 | 🤖 Agent | ⭐⭐⭐⭐ | UI 검증 없이는 인간 QA가 병목 |
| **Phase 4** | 에이전트용 린트 에러 메시지 | 🤖+👤 | ⭐⭐⭐⭐⭐ | 에러 메시지에 수정 지침 포함 → 에이전트가 자동 수정 가능 |
| **Phase 4** | 에이전트 간 코드 리뷰 | 🤖 Agent | ⭐⭐⭐⭐ | 인간 리뷰 부하를 제거하는 핵심 |
| **Phase 5** | AI 슬롭 자동 정리 | 🤖 Agent | ⭐⭐⭐⭐⭐ | 금요일 20% 수동 정리 → 확장 불가. 자동화 필수 |
| **Phase 5** | 문서 정원사 에이전트 | 🤖 Agent | ⭐⭐⭐⭐ | 문서 부패 방지. 에이전트의 컨텍스트 품질 유지 |

## 4. 하네스의 핵심 구성 요소별 기능 상세

### 4.1 Knowledge Base Layer (지식 기반 계층)

```
┌─────────────────────────────────────────────────────┐
│                 KNOWLEDGE BASE                       │
│                                                     │
│  AGENTS.md (목차 역할, ~100줄)                      │
│  ┌─────────────────────────────────────────────┐   │
│  │ "이 저장소는 X를 하는 프로젝트이다.          │   │
│  │  설계 → docs/design/ 참조                   │   │
│  │  아키텍처 → docs/architecture/ 참조          │   │
│  │  실행 계획 → docs/plans/ 참조                │   │
│  │  품질 기준 → docs/quality/ 참조              │   │
│  │  기술 부채 → docs/tech-debt/ 참조"           │   │
│  └─────────────────────────────────────────────┘   │
│         │                                           │
│         ▼                                           │
│  docs/                                              │
│  ├── design/          ← 설계 문서 + 검증 상태      │
│  ├── architecture/    ← 도메인/패키지 레이어링      │
│  ├── plans/           ← 실행 계획 + 결정 로그       │
│  ├── quality/         ← 도메인별 품질 등급          │
│  └── tech-debt/       ← 기술 부채 트래커            │
│                                                     │
│  [핵심] 전용 린터 + CI가 문서 구조/최신성 강제     │
│  [핵심] Doc-Gardening Agent가 자동으로 문서 갱신    │
└─────────────────────────────────────────────────────┘
```

**왜 중요한가:**
- 단일 AGENTS.md → 실패 (컨텍스트 부족, 부패, 검증 불가)
- 구조화된 docs/ → 기계적 검증 가능, 분산 관리, 자동 갱신

### 4.2 Tooling & Environment Layer (도구 & 환경 계층)

```
┌─────────────────────────────────────────────────────┐
│           AGENT EYES & EARS (에이전트의 눈과 귀)    │
│                                                     │
│  ┌─────────────────┐    ┌───────────────────────┐  │
│  │   CDP (Chrome   │    │   Observability       │  │
│  │   DevTools      │    │   Stack               │  │
│  │   Protocol)     │    │                       │  │
│  │                 │    │   Logs ──▶ LogQL      │  │
│  │ 기능:           │    │   Metrics ──▶ PromQL  │  │
│  │ • DOM 읽기      │    │                       │  │
│  │ • 스크린샷      │    │   Per-worktree로      │  │
│  │ • 클릭/입력     │    │   격리된 임시 스택    │  │
│  │ • 네비게이션    │    │                       │  │
│  └────────┬────────┘    └───────────┬───────────┘  │
│           │                         │               │
│           ▼                         ▼               │
│  ┌─────────────────────────────────────────────┐   │
│  │        Git Worktree 격리 환경                │   │
│  │                                               │   │
│  │   PR #1 ──▶ worktree/1/ ──▶ app instance #1 │   │
│  │   PR #2 ──▶ worktree/2/ ──▶ app instance #2 │   │
│  │   PR #N ──▶ worktree/N/ ──▶ app instance #N │   │
│  │                                               │   │
│  │   각 인스턴스는 완전히 독립적으로 부팅/테스트 │   │
│  └─────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────┘
```

**왜 중요한가:**
- 에이전트에게 "눈과 귀"가 없으면 → 인간 QA가 병목
- 격리 환경 없으면 → 병렬 실행 불가, 처리량 한계

### 4.3 Architecture Enforcement Layer (아키텍처 강제 계층)

```
┌─────────────────────────────────────────────────────┐
│         ARCHITECTURE ENFORCEMENT                     │
│                                                     │
│  의존성 방향 (단방향만 허용):                       │
│                                                     │
│  Types ──▶ Config ──▶ Repo ──▶ Service ──▶ Runtime ──▶ UI │
│                                                     │
│  ┌──────────────────────────────────────────────┐  │
│  │  Custom Linters (에이전트 친화적)             │  │
│  │                                               │  │
│  │  규칙 #1: 레이어 역방향 의존 금지             │  │
│  │  규칙 #2: 구조화된 로깅만 허용                │  │
│  │  규칙 #3: 네이밍 컨벤션 강제                  │  │
│  │  규칙 #4: 파일 크기 제한                      │  │
│  │  규칙 #5: 플랫폼 신뢰성 요구사항 강제         │  │
│  │                                               │  │
│  │  [핵심] 에러 메시지에 수정 지침 포함          │  │
│  │  예: "Error: Service layer cannot import      │  │
│  │   from UI layer. Move this type to Types      │  │
│  │   layer at src/types/ instead."               │  │
│  │                                               │  │
│  │  → 에이전트가 에러 메시지를 읽고 자동 수정    │  │
│  └──────────────────────────────────────────────┘  │
│                                                     │
│  ┌──────────────────────────────────────────────┐  │
│  │  Cross-Cutting Concerns                       │  │
│  │                                               │  │
│  │  Auth, Logging, Error Handling 등             │  │
│  │  → 단일 명시적 인터페이스를 통해서만 진입     │  │
│  │  → 직접 import 금지, 인터페이스 통해서만      │  │
│  └──────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘
```
