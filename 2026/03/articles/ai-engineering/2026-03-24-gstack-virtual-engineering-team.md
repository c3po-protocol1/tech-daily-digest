## 1. 프로젝트 개요 — "한 명이 20명 팀처럼"

Y Combinator의 CEO Garry Tan이 직접 만들고 매일 사용하는 **AI 기반 소프트웨어 팩토리**이다. Claude Code를 단순한 코딩 도우미가 아닌, **역할이 분화된 가상 엔지니어링 팀**으로 변환한다. CEO가 제품을 재구상하고, 엔지니어링 매니저가 아키텍처를 확정하고, 디자이너가 AI 슬롭(slop)을 잡아내고, 리뷰어가 프로덕션 버그를 찾고, QA 리드가 실제 브라우저를 열어 테스트하고, 보안 담당자가 OWASP + STRIDE 감사를 실행하고, 릴리즈 엔지니어가 PR을 올린다 — 이 모든 역할을 슬래시 커맨드 하나로 호출할 수 있다.

Garry Tan의 실적 수치가 이 도구의 위력을 증명한다. **지난 60일간 600,000줄 이상의 프로덕션 코드**(35%가 테스트), **하루 10,000-20,000줄**, YC CEO 직무를 풀타임으로 수행하면서 파트타임으로 달성한 수치이다. 직전 주간 기록은 140,751줄 추가, 362 커밋, 순 115K LOC이다.

프로젝트의 영감은 Andrej Karpathy의 발언에서 왔다: "아마 12월부터 코드를 한 줄도 직접 타이핑하지 않은 것 같다." 그리고 Peter Steinberger가 OpenClaw(247K GitHub 스타)를 사실상 혼자 AI 에이전트와 함께 구축한 사례도 참고했다.

## 2. 핵심 아키텍처 — "프로세스, 도구 모음이 아니다"

gstack의 가장 중요한 설계 철학은 **"프로세스(process)이지, 도구 모음(collection of tools)이 아니다"**라는 것이다. 스킬들은 실제 스프린트의 순서대로 실행되며, 각 단계의 출력이 다음 단계의 입력이 된다:

**Think → Plan → Build → Review → Test → Ship → Reflect**

구체적으로:
- `/office-hours`가 작성한 설계 문서를 `/plan-ceo-review`가 읽는다
- `/plan-eng-review`가 작성한 테스트 계획을 `/qa`가 활용한다
- `/review`가 발견한 버그를 `/ship`이 수정 여부를 검증한다
- 모든 단계가 이전 단계를 알고 있기 때문에 **아무것도 누락되지 않는다**

이것이 기존의 "프롬프트 모음집"이나 단순한 스킬 패키지와 근본적으로 다른 점이다. gstack은 소프트웨어 개발의 전체 라이프사이클을 구조화된 프로세스로 모델링한 것이다.

## 3. 28개 스킬 — 가상 팀의 역할 분담

### 스프린트 스킬 (Think → Ship)

| 슬래시 커맨드 | 가상 역할 | 하는 일 |
|---|---|---|
| `/office-hours` | YC 오피스 아워 | 6가지 강제 질문으로 제품을 재구상. 전제를 도전하고, 구현 대안을 생성. 설계 문서를 작성하여 하위 스킬에 전달 |
| `/plan-ceo-review` | CEO/파운더 | 문제를 재정의. 요청 안에 숨겨진 "10성급 제품"을 찾아냄. 4개 모드: 확장, 선택적 확장, 범위 유지, 축소 |
| `/plan-eng-review` | 엔지니어링 매니저 | 아키텍처, 데이터 플로우, ASCII 다이어그램, 엣지 케이스, 테스트 계획을 확정. 숨겨진 가정을 표면화 |
| `/plan-design-review` | 시니어 디자이너 | 각 디자인 차원을 0-10점으로 평가하고, 10점이 어떤 모습인지 설명. **AI Slop 감지** 포함 |
| `/design-consultation` | 디자인 파트너 | 완전한 디자인 시스템을 처음부터 구축. 랜드스케이프 리서치, 창의적 리스크 제안, 프로덕트 목업 생성 |
| `/review` | 스태프 엔지니어 | CI는 통과하지만 프로덕션에서 터지는 버그를 찾음. 자명한 것은 자동 수정, 완전성 갭을 플래그 |
| `/qa` | QA 리드 | **실제 브라우저**에서 앱을 테스트. 버그 발견 → 수정 → 재검증. 모든 수정에 회귀 테스트 자동 생성 |
| `/cso` | CISO(보안 책임자) | OWASP Top 10 + STRIDE 위협 모델. 17개 오탐 제외 규칙, 8/10 이상 신뢰도 게이트, 각 발견에 구체적 익스플로잇 시나리오 포함 |
| `/ship` | 릴리즈 엔지니어 | main 동기화, 테스트 실행, 커버리지 감사, 푸시, PR 생성. 테스트 프레임워크가 없으면 부트스트랩 |
| `/land-and-deploy` | 릴리즈 엔지니어 | PR 머지 → CI 대기 → 배포 → 프로덕션 헬스 검증까지 한 커맨드 |
| `/canary` | SRE | 배포 후 모니터링. 콘솔 에러, 성능 퇴보, 페이지 실패 감시 |
| `/retro` | 엔지니어링 매니저 | 팀 인식 주간 회고. 인당 분석, 출시 스트릭, 테스트 건강도, 성장 기회. `global` 모드로 모든 프로젝트와 AI 도구 통합 회고 |

### 파워 도구

| 커맨드 | 기능 |
|---|---|
| `/codex` | OpenAI Codex CLI를 세컨드 오피니언으로 활용. 리뷰, 적대적 도전, 오픈 상담 3개 모드 |
| `/careful` | 안전 가드레일. `rm -rf`, `DROP TABLE`, force-push 등 위험 명령 전 경고 |
| `/freeze` | 편집 잠금. 디버깅 중 특정 디렉토리 외 파일 수정 방지 |
| `/guard` | `/careful` + `/freeze` 결합. 프로덕션 작업 시 최대 안전 모드 |
| `/browse` | 실제 Chromium 브라우저. 실제 클릭, 실제 스크린샷. 명령당 ~100ms |
| `/autoplan` | CEO → 디자인 → 엔지니어링 리뷰를 자동 파이프라인으로 실행. 취향 결정만 인간에게 올림 |

## 4. 실제 사용 예시 — "일일 브리핑 앱"에서 "개인 참모 AI"로

gstack의 README에 포함된 사용 시나리오가 도구의 철학을 잘 보여준다:

1. **사용자:** "캘린더 일일 브리핑 앱을 만들고 싶어"
2. **`/office-hours`:** 구체적 고통 포인트를 질문 (가설이 아닌 실제 경험 기반)
3. **사용자:** "여러 Google 캘린더, 오래된 정보, 잘못된 위치. 준비에 시간이 너무 걸려"
4. **에이전트:** "프레이밍에 반박하겠습니다. 당신은 '일일 브리핑 앱'이라고 했지만, 실제로 설명한 것은 **개인 참모 AI(chief of staff AI)**입니다." → 사용자가 인지하지 못한 5가지 역량을 추출, 4가지 전제에 도전, 3가지 구현 접근법 제시
5. **`/plan-ceo-review`:** 설계 문서를 읽고 범위에 도전, 10개 섹션 리뷰
6. **`/plan-eng-review`:** 데이터 플로우 ASCII 다이어그램, 상태 머신, 에러 경로, 테스트 매트릭스
7. **승인 후:** 11개 파일에 2,400줄을 ~8분에 작성
8. **`/review`:** 2개 이슈 자동 수정, 1개 경합 조건 인간에게 확인
9. **`/qa`:** 실제 브라우저에서 플로우 테스트, 버그 발견 및 수정
10. **`/ship`:** 테스트 42→51(+9 신규). PR 생성 완료

**8개 커맨드로 엔드 투 엔드 완성.** 이것이 코파일럿이 아니라 **팀**이라는 gstack의 핵심 주장이다.

## 5. 병렬 스프린트 — "10개 에이전트가 동시에"

gstack의 가장 야심찬 기능은 **병렬 스프린트**이다. **Conductor**라는 도구가 여러 Claude Code 세션을 병렬로 실행한다 — 각 세션은 자체 격리된 워크스페이스에서 작동한다. 하나는 `/office-hours`를, 다른 하나는 `/review`를, 세 번째는 기능 구현을, 네 번째는 `/qa`를 동시에 수행한다.

스프린트 구조가 병렬 실행을 가능하게 하는 핵심이다 — **프로세스 없이 10개 에이전트를 돌리면 10개의 혼란이지만, 프로세스가 있으면 각 에이전트가 정확히 무엇을 해야 하고 언제 멈춰야 하는지 안다.**

## 6. 크로스 에이전트 호환성

gstack은 Claude Code 전용이 아니다. **SKILL.md 표준을 지원하는 모든 에이전트**에서 작동한다:
- **Claude Code** — 네이티브 지원 (`.claude/skills/`)
- **OpenAI Codex** — `./setup --host codex`로 설치 (`.agents/skills/` 또는 `~/.codex/skills/`)
- **Gemini CLI** — 지원
- **Cursor** — 지원
- `--host auto`로 설치된 에이전트를 자동 감지

28개 전체 스킬이 모든 지원 에이전트에서 작동한다.

## 7. 기술적 세부사항

- **설치:** 30초. `git clone` → `./setup` → 완료
- **의존성:** Claude Code, Git, Bun v1.0+, Node.js(Windows만)
- **브라우저 QA:** Playwright 기반 실제 Chromium. 실제 쿠키 가져오기 지원 (Chrome, Arc, Brave, Edge)
- **텔레메트리:** 기본 OFF. 옵트인 시에도 스킬명/시간/성공여부/OS만 수집. 코드/파일경로/프롬프트 절대 미수집
- **`/codex`:** OpenAI Codex CLI를 세컨드 오피니언으로 활용 — 크로스 모델 분석 가능 (Claude + Codex가 모두 리뷰하면 교차 분석 리포트 생성)
- **팀 공유:** `cp -Rf` 로 레포에 복사하면 `git clone`만으로 팀원도 사용 가능 (서브모듈 아님)

## 8. 왜 이것이 중요한가 — 하네스 엔지니어링의 최전선

gstack은 단순한 프롬프트 모음집이 아니다. 소프트웨어 개발의 **전체 프로세스를 구조화하고, 각 단계에 전문화된 에이전트 페르소나를 배정하며, 단계 간 정보를 자동으로 전달**하는 종합 하네스 시스템이다. 이것이 바로 "하네스 엔지니어링"의 구체적 실현이며, 어제 요약한 여러 글에서 논의된 개념(Skills, Instincts, Memory, 프로세스 구조화)이 하나의 완성된 시스템으로 통합된 형태이다.

핵심 인사이트:
- **프로세스가 도구보다 중요하다** — 10개의 강력한 도구보다, 순서와 연결이 있는 프로세스가 더 강력하다
- **에이전트에게 역할을 부여하면 성능이 달라진다** — 같은 Claude 모델이라도 "스태프 엔지니어로서 리뷰하라"와 "코드를 리뷰하라"는 다른 결과를 낸다
- **단계 간 정보 전달이 핵심이다** — 설계 문서 → 엔지니어링 리뷰 → 코드 생성 → 리뷰 → QA → 배포의 전체 흐름에서 정보가 유실되지 않는 것이 gstack의 진짜 가치
- **안전 장치가 내장되어 있다** — `/careful`, `/freeze`, `/guard`로 위험한 작업을 체계적으로 방지

## 9. Soo에게 의미 있는 이유

gstack은 AInD 컨설팅의 **핵심 데모 소재이자 참고 아키텍처**이다.

첫째, **"하네스 엔지니어링이 무엇인가"를 한 번에 보여줄 수 있다.** 추상적 개념 설명 대신, gstack을 실제로 시연하면 "같은 AI 모델이라도 하네스에 따라 이렇게 다르다"는 것을 직관적으로 증명할 수 있다.

둘째, **YC CEO가 직접 사용한다는 사실이 강력한 사회적 증거**이다. "실리콘밸리 최전선에서 이미 이렇게 일하고 있다"는 메시지는 기업 경영진에게 강한 자극이 된다.

셋째, **기업 맞춤 gstack 구축이 컨설팅 서비스가 될 수 있다.** gstack의 프로세스 구조(Think→Plan→Build→Review→Test→Ship)를 기업의 기존 개발 프로세스에 맞게 커스터마이즈하는 것이 구체적인 컨설팅 딜리버리가 될 수 있다. 예를 들어 금융 기업이면 `/compliance-review` 스킬을 추가하고, 의료 기업이면 `/hipaa-audit` 스킬을 추가하는 식이다.

넷째, **"10,000-20,000줄/일"이라는 수치가 ROI 논의의 기반**이 된다. AI 도입의 ROI를 구체적 숫자로 보여줄 수 있으며, 이 수치를 Padawan이 직접 재현해보면 더욱 설득력 있는 사례가 된다.
