## 0. 전체 구조 이해

Claude Code의 멀티 에이전트는 **두 가지 계층**으로 나뉜다:

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│   1. Sub-agents (서브 에이전트)                                  │
│      = 단일 세션 내에서 특수화된 도우미                          │
│      = "전문가를 불러서 특정 일 맡기기"                         │
│                                                                  │
│   2. Agent Teams (에이전트 팀)                                   │
│      = 여러 Claude 인스턴스가 병렬로 협업                       │
│      = "팀을 꾸려서 동시에 작업하기"                            │
│                                                                  │
│   ┌─────────────────────────────────────────────────────┐       │
│   │  비유:                                               │       │
│   │  Sub-agent = 시니어가 주니어에게 "이것 좀 조사해봐" │       │
│   │  Agent Team = 팀장이 팀원 5명에게 "각자 맡아서 해"  │       │
│   └─────────────────────────────────────────────────────┘       │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 1. Sub-agents (서브 에이전트)

### 1.3 커스텀 서브에이전트 만들기

**파일 위치:**
```
프로젝트 레벨 (팀 공유):
  .claude/agents/code-reviewer.md
  .claude/agents/debugger.md
  .claude/agents/db-reader.md

사용자 레벨 (개인용, 모든 프로젝트에서 사용):
  ~/.claude/agents/my-agent.md
```

**기본 구조 (Markdown + YAML frontmatter):**

```markdown
---
name: code-reviewer
description: "코드 변경사항을 보안, 성능, 가독성 관점에서 리뷰합니다.
              자동으로 git diff를 실행하여 최근 변경사항을 분석합니다."
model: inherit    # haiku, sonnet, opus, 또는 claude-sonnet-4-6 등
tools:
  - Read
  - Grep
  - Glob
  - Bash
disallowedTools:
  - Write
  - Edit
---

# Code Reviewer

You are a senior code reviewer. When invoked:

1. Run `git diff HEAD~1` to see recent changes
2. Analyze each file for:
   - 🔴 Critical: security vulnerabilities, data loss risks
   - 🟡 Warning: performance issues, error handling gaps
   - 🟢 Suggestion: style improvements, readability
3. Provide actionable feedback with specific line references
```

**핵심 필드 설명:**

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│   name          서브에이전트 이름 (호출 시 사용)                │
│                                                                  │
│   description   ⭐ 가장 중요!                                   │
│                 Claude가 이 설명을 읽고 "자동으로" 위임할지    │
│                 결정함. 구체적이고 명확하게 작성해야 함          │
│                                                                  │
│   model         사용할 모델                                     │
│                 • inherit: 메인 세션과 같은 모델                │
│                 • haiku: 빠르고 저렴 (탐색/검색용)              │
│                 • sonnet: 균형 (대부분의 작업)                  │
│                 • opus: 최고 품질 (복잡한 추론)                 │
│                                                                  │
│   tools         허용하는 도구 목록                               │
│                 Read, Write, Edit, Grep, Glob,                  │
│                 Bash, AskUserQuestion 등                        │
│                                                                  │
│   disallowedTools  차단하는 도구 목록                            │
│                    Write, Edit → 읽기 전용 에이전트 생성       │
│                                                                  │
│   skills        사전 로딩할 스킬 (재사용 가능한 프롬프트)       │
│                                                                  │
│   memory        project | user | local                          │
│                 persistent MEMORY.md로 지식 축적 가능           │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 1.4 실전 예시 — 4가지 패턴

#### 패턴 1: 읽기 전용 코드 리뷰어

```markdown
---
name: code-reviewer
description: "코드 변경사항을 리뷰합니다. git diff를 분석하여
              보안, 성능, 가독성 이슈를 찾습니다."
model: inherit
tools:
  - Read
  - Grep
  - Glob
  - Bash
disallowedTools:
  - Write
  - Edit
---

# Code Reviewer

리뷰 시 다음 기준으로 분류:
- 🔴 Critical: 보안 취약점, 데이터 유실 위험
- 🟡 Warning: 성능 이슈, 에러 처리 부족
- 🟢 Suggestion: 스타일 개선, 가독성
```

**사용법:**
```bash
# 자동 위임 (description을 보고 Claude가 판단)
claude "최근 커밋 변경사항 리뷰해줘"

# 명시적 호출
claude "@code-reviewer 인증 모듈 변경사항 확인해줘"
```

#### 패턴 2: 디버거

```markdown
---
name: debugger
description: "버그를 체계적으로 추적하고 수정합니다.
              에러 재현 → 원인 격리 → 수정 → 검증 순서로 진행."
model: inherit
tools:
  - Read
  - Write
  - Edit
  - Bash
  - Grep
---

# Debugger

5단계 디버깅 프로세스:
1. 에러 캡처: 에러 메시지, 스택 트레이스 확인
2. 재현: 동일한 환경에서 버그 재현
3. 격리: 원인 코드 위치 특정
4. 수정: 최소한의 변경으로 버그 수정
5. 검증: 수정 후 재현 테스트 + 회귀 테스트
```

#### 패턴 3: DB 읽기 전용 (Hooks로 안전 보장)

```markdown
---
name: db-reader
description: "데이터베이스를 조회합니다. 읽기만 가능하며
              INSERT, UPDATE, DELETE 등 쓰기 작업은 불가합니다."
model: sonnet
tools:
  - Bash
  - Read
disallowedTools:
  - Write
  - Edit
---

# Database Reader (Read-Only)

You have read-only access to the database.
You can run SELECT queries only.
If asked to INSERT, UPDATE, DELETE, or modify schema,
explain that you only have read access.
```

**안전 장치 — Hook 스크립트:**

```bash
# .claude/hooks/db-guard.sh
#!/bin/bash
# PreToolUse hook: Bash 명령 실행 전에 SQL 검사

INPUT=$(cat)
SQL=$(echo "$INPUT" | grep -iE '(INSERT|UPDATE|DELETE|DROP|ALTER|TRUNCATE)')

if [ -n "$SQL" ]; then
  echo "❌ BLOCKED: 쓰기 SQL이 감지되었습니다."
  echo "이 에이전트는 읽기 전용입니다."
  exit 2  # exit 2 = 실행 차단
fi

exit 0  # exit 0 = 실행 허용
```

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│   방어 심층 (Defense in Depth):                                  │
│                                                                  │
│   Layer 1: 시스템 프롬프트                                      │
│            "읽기만 가능합니다" (행동적 제약)                     │
│                                                                  │
│   Layer 2: Hook 스크립트                                         │
│            SQL 쓰기 명령 차단 (기술적 강제)                     │
│                                                                  │
│   → 프롬프트만으로는 100% 방어 불가                            │
│   → Hook이 기술적 안전망 역할                                   │
│   → 이것이 바로 "Harness"의 구체적 사례!                       │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

#### 패턴 4: MCP 서버 연결 (브라우저 테스터)

```markdown
---
name: browser-tester
description: "Playwright를 사용하여 웹 UI를 테스트합니다.
              브라우저를 자동으로 제어하여 E2E 테스트를 수행합니다."

tools:
  - Read
  - Bash
  - mcp__playwright__*   # Playwright MCP 도구만 허용

  playwright:
    command: npx
    args: ["@anthropic/playwright-mcp"]
---

# Browser Tester

Playwright MCP를 사용하여:
1. 페이지 이동 및 요소 상호작용
2. 스크린샷 캡처 및 시각적 검증
3. E2E 사용자 플로우 테스트
```

### 1.5 서브에이전트 호출 방법 (3가지)

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│   방법 1: 자연어 (자동 위임)                                    │
│   ──────────────────────────                                    │
│   claude "최근 코드 변경사항 보안 이슈 확인해줘"               │
│   → Claude가 description을 보고 code-reviewer에게 자동 위임    │
│                                                                  │
│   방법 2: @멘션 (명시적 위임)                                   │
│   ──────────────────────────                                    │
│   "인증 모듈을 @debugger로 디버깅해줘"                         │
│   → 한 작업에 대해 특정 서브에이전트 보장                      │
│                                                                  │
│   방법 3: --agent 플래그 (전체 세션)                            │
│   ──────────────────────────                                    │
│   claude --agent code-reviewer                                  │
│   → 전체 세션이 해당 서브에이전트 설정으로 실행                │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 1.6 실행 모드 (포그라운드 vs 백그라운드)

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│   포그라운드 (기본):                                            │
│   • 메인 세션을 차단 (블로킹)                                   │
│   • 권한 요청 시 사용자에게 프롬프트 전달                       │
│   • 대화형 작업에 적합                                          │
│                                                                  │
│   백그라운드:                                                   │
│   • 메인 세션과 병렬 실행 (논블로킹)                            │
│   • 권한은 사전에 승인해야 함                                   │
│   • Ctrl+B로 실행 중 작업을 백그라운드로 전환 가능              │
│   • 질문(AskUserQuestion)은 실패하지만 작업은 계속              │
│                                                                  │
│   ┌──────────────────────────────────────────────────────┐     │
│   │  Ctrl+B 활용 팁:                                     │     │
│   │                                                      │     │
│   │  1. 서브에이전트에게 작업 시작                        │     │
│   │  2. 진행 상황 보면서 "이건 오래 걸리겠다" 판단       │     │
│   │  3. Ctrl+B → 백그라운드로 전환                       │     │
│   │  4. 메인 세션에서 다른 작업 수행                      │     │
│   │  5. 완료 시 자동 알림                                │     │
│   └──────────────────────────────────────────────────────┘     │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 1.7 Memory (지속적 기억)

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│   memory: project                                                │
│                                                                  │
│   서브에이전트가 자신만의 MEMORY.md를 가짐                      │
│   → 대화 간에도 지식이 유지됨                                   │
│   → 코드 패턴, 아키텍처 결정, 반복 이슈 등을 축적              │
│                                                                  │
│   예시:                                                         │
│   code-reviewer가 "이 프로젝트에서는 에러 핸들링에             │
│   항상 Result 타입을 쓴다"를 기억 →                            │
│   다음 리뷰 시 자동으로 해당 패턴 위반 감지                    │
│                                                                  │
│   이것은 Harness Engineering의                                  │
│   Knowledge Base Layer의 구체적 구현!                           │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 2. Agent Teams (에이전트 팀)

### 2.1 개념

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│   Sub-agents: 1개의 Claude가 도우미를 호출 (직렬/위임)         │
│   Agent Teams: 여러 Claude가 동시에 병렬로 작업 (팀 협업)      │
│                                                                  │
│   ┌──────────────────────────────────────────────────────┐     │
│   │                                                      │     │
│   │              👤 사용자                               │     │
│   │                │                                     │     │
│   │                ▼                                     │     │
│   │         ┌─────────────┐                              │     │
│   │         │  Lead Agent │ (팀장)                       │     │
│   │         │  작업 분배   │                              │     │
│   │         │  결과 취합   │                              │     │
│   │         └──────┬──────┘                              │     │
│   │           ┌────┼────┬────────┐                       │     │
│   │           ▼    ▼    ▼        ▼                       │     │
│   │         ┌──┐ ┌──┐ ┌──┐   ┌──┐                      │     │
│   │         │T1│ │T2│ │T3│   │T4│  (팀원들)            │     │
│   │         │FE│ │BE│ │테스│  │문서│ (병렬 실행)        │     │
│   │         └──┘ └──┘ └──┘   └──┘                      │     │
│   │                                                      │     │
│   └──────────────────────────────────────────────────────┘     │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 2.2 활성화

```bash
# 환경 변수로 활성화 (실험적 기능)
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1

# 이후 Claude Code 실행
claude
```

### 2.3 사용 시나리오 (4가지)

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│   시나리오 1: 병렬 리서치                                       │
│   ──────────────────────                                        │
│   "인증, 데이터베이스, API 모듈을 각각 다른 에이전트가         │
│    동시에 분석해줘"                                              │
│                                                                  │
│   Lead → Teammate A: "인증 모듈 분석"                           │
│        → Teammate B: "데이터베이스 모듈 분석"                   │
│        → Teammate C: "API 모듈 분석"                            │
│   (3개가 동시에 실행 → 시간 1/3)                                │
│                                                                  │
│   ─────────────────────────────────────────────                  │
│                                                                  │
│   시나리오 2: 모듈 분리 개발                                    │
│   ──────────────────────                                        │
│   "프론트엔드, 백엔드, 테스트를 각각 맡아서 동시에 개발해줘"   │
│                                                                  │
│   Lead → Teammate A: "프론트엔드 구현" (src/frontend/)         │
│        → Teammate B: "백엔드 구현" (src/backend/)              │
│        → Teammate C: "테스트 작성" (tests/)                    │
│   (파일 충돌 없도록 영역 분리 필수!)                            │
│                                                                  │
│   ─────────────────────────────────────────────                  │
│                                                                  │
│   시나리오 3: 경쟁 가설 디버깅                                  │
│   ──────────────────────                                        │
```

### 2.5 커뮤니케이션

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│   팀원 간 소통 방식:                                            │
│                                                                  │
│   1. message (1:1 메시지)                                       │
│      Lead → Teammate A: "인증 모듈 분석 결과 알려줘"           │
│      Teammate A → Lead: "JWT 검증에 취약점 발견"               │
│                                                                  │
│   2. broadcast (전체 방송)                                      │
│      Lead → 모든 Teammate:                                      │
│      "전체 공지: API 스펙이 변경되었습니다. 참고하세요."       │
│      ⚠️ 주의: 토큰 비용이 팀원 수만큼 증가!                   │
│                                                                  │
│   자동 메시지 전달 — 폴링 불필요                                │
│                                                                  │
│   작업 흐름:                                                    │
│   ┌──────────────────────────────────────────────────────┐     │
│   │  Lead: 작업 목록 생성 + 분배                         │     │
│   │     │                                                │     │
│   │     ├── Teammate A: 작업 수행 → 완료 보고            │     │
│   │     │   → 새 작업 자동 할당 (또는 자체 claim)        │     │
│   │     │                                                │     │
│   │     ├── Teammate B: 작업 수행 → 질문 발생            │     │
│   │     │   → Lead에게 질문 → 답변 수신 → 작업 계속     │     │
│   │     │                                                │     │
│   │     └── Teammate C: 작업 수행 → 다른 팀원과 협의     │     │
│   │         → Teammate A에게 메시지 → 협업               │     │
│   └──────────────────────────────────────────────────────┘     │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 2.6 품질 관리 — Hooks

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│   TeammateIdle Hook:                                             │
│   팀원이 작업을 마치고 대기 상태일 때 발동                      │
│                                                                  │
│   exit 0 → 팀원 대기 허용                                       │
│   exit 2 → 피드백 전달 + 작업 계속 ("더 해야 할 게 있어")     │
│                                                                  │
│   ─────────────────────────────────────────────                  │
│                                                                  │
│   TaskCompleted Hook:                                            │
│   팀원이 작업 완료를 보고할 때 발동                              │
│                                                                  │
│   exit 0 → 완료 승인                                            │
│   exit 2 → 완료 거부 ("품질 기준 미달, 다시 해")              │
│                                                                  │
│   ┌──────────────────────────────────────────────────────┐     │
│   │  이것은 Harness Engineering의                        │     │
│   │  Architecture Enforcement Layer!                     │     │
│   │                                                      │     │
│   │  • Hook = 커스텀 린터와 같은 역할                    │     │
│   │  • 에이전트 출력을 기계적으로 검증                   │     │
│   │  • 품질 기준 미달 시 자동 차단                       │     │
│   └──────────────────────────────────────────────────────┘     │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 2.7 실전 사용 예시

```bash
# 1. 환경 변수 설정
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1

# 2. Claude Code 실행
claude

# 3. 팀 구성 요청
> "이 프로젝트에 새로운 결제 시스템을 추가해야 해.
   팀을 꾸려서 작업해줘:
   - 1명: 결제 API 백엔드 개발 (src/api/payment/)
   - 1명: 결제 UI 프론트엔드 개발 (src/components/payment/)
   - 1명: 결제 관련 테스트 작성 (tests/payment/)
   - 1명: 결제 관련 문서 작성 (docs/payment/)
   파일 충돌이 없도록 각자 영역을 지켜줘."

# 4. Lead Agent가 자동으로:
#    - 4명의 Teammate 생성
#    - 각자에게 작업 할당
#    - 진행 상황 모니터링
#    - 결과 취합

# 5. Shift+Down으로 각 팀원 상태 확인

# 6. 특정 팀원에게 직접 지시 가능:
> "Teammate 2에게: 결제 UI에 로딩 스피너 추가해줘"
```

### 2.8 주의사항 & 제약

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│   ⚠️ 비용:                                                     │
│   • 각 팀원이 자체 컨텍스트 윈도우를 가짐                       │
│   • 팀원 수 × 토큰 비용 (선형 증가)                            │
│   • broadcast 시 팀원 수만큼 비용 증가                          │
│   • 5명 팀 = 약 5배 토큰 비용                                   │
│                                                                  │
│   ⚠️ 수확 체감 (Diminishing Returns):                          │
│   • 2-4명: 효과적                                               │
│   • 5-7명: 조정 비용 증가                                       │
│   • 8명+: 비효율적 (소통 오버헤드 > 생산성)                    │
│                                                                  │
│   ⚠️ 기술적 제약:                                              │
│   • 세션 재개(resume) 불가 — 팀원이 사라짐                     │
│   • 팀 내 팀 (sub-team) 불가                                    │
│   • Lead 역할 이전 불가                                         │
│   • 종료 시 팀원이 현재 도구 호출 완료까지 대기                │
│   • Split-pane: VS Code 터미널, Windows Terminal 미지원         │
│                                                                  │
│   📌 Best Practices:                                            │
│   • 파일 충돌 방지: 각 팀원에게 비중복 파일 영역 할당          │
│   • 작업 크기: "함수 하나" ~ "파일 하나" 수준이 적절           │
│   • 시작은 리서치/리뷰부터 → 익숙해지면 병렬 개발로            │
│   • 복잡한 의존성 있는 작업은 Sub-agent로 (직렬 처리)          │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 3. Sub-agents vs Agent Teams 비교

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│   항목            Sub-agents           Agent Teams              │
│   ──────          ──────────           ───────────              │
│   실행 방식       직렬 (위임)          병렬 (동시)             │
│   관계            1:1 (호출자↔도우미)  1:N (리드↔팀원들)      │
│   소통            결과 반환만           상호 메시지 교환        │
│   컨텍스트        메인 세션 내/외       각자 독립               │
│   비용            낮음 (1개 추가)       높음 (N개 추가)         │
│   세션 재개       가능                  불가                    │
│   사용 시점       전문가 의견 필요      동시 작업 필요          │
│   활성화          기본 제공             환경 변수 필요          │
│   안정성          안정                  실험적 (Experimental)   │
│                                                                  │
│   선택 기준:                                                    │
│   ┌──────────────────────────────────────────────────────┐     │
│   │                                                      │     │
│   │  "이 작업에 전문가가 필요해"          → Sub-agent    │     │
│   │  "이 작업을 빨리 끝내야 해 (병렬)"   → Agent Team  │     │
│   │  "이 작업이 서로 의존적이야"          → Sub-agent    │     │
│   │  "이 작업이 서로 독립적이야"          → Agent Team  │     │
│   │  "비용을 아껴야 해"                   → Sub-agent    │     │
│   │  "시간을 아껴야 해"                   → Agent Team  │     │
│   │                                                      │     │
│   └──────────────────────────────────────────────────────┘     │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 4. Harness Engineering과의 연결

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│   Claude Code의 멀티 에이전트 시스템은                          │
│   Harness Engineering의 구체적 구현이다:                         │
│                                                                  │
│   📚 Knowledge Base Layer                                       │
│      = CLAUDE.md + Memory + Skills                              │
│      = 에이전트가 "무엇을 아는가"를 정의                       │
│                                                                  │
│   🔧 Tooling & Environment Layer                                │
│      = tools + MCP Servers + Bash                               │
│      = 에이전트가 "무엇을 할 수 있는가"를 정의                 │
│                                                                  │
│   🏗️ Architecture Enforcement Layer                             │
│      = disallowedTools + Hooks + description                    │
│      = 에이전트가 "어떤 규칙을 따르는가"를 강제                │
│                                                                  │
│   ♻️ Continuous Maintenance Layer                                │
│      = Memory (지속적 학습) + TaskCompleted Hook                │
│      = 에이전트가 "스스로 개선되는가"를 관리                   │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```
