# Anthropic Managed Agents — 두뇌와 손을 분리하는 에이전트 인프라 설계 (상세 요약)

> **출처**: [Anthropic Engineering Blog](https://www.anthropic.com/engineering/managed-agents)

> **저자**: Lance Martin, Gabe Cemaj, Michael Cohen
> **태그**: `#AI에이전트` `#Anthropic` `#아키텍처` `#인프라` `#하네스`

## TL;DR

- Anthropic이 장기 실행 에이전트 호스팅 서비스 **Managed Agents**의 아키텍처를 공개
- 핵심 철학: **"두뇌(brain)와 손(hands)을 분리하라"**
- OS가 하드웨어를 process/file로 가상화한 것처럼, 에이전트 구성 요소(Session, Harness, Sandbox)를 가상화
- **하네스의 가정은 유통기한이 있다** — 인터페이스가 구현보다 오래 지속되도록 설계해야

---

## 1. 배경: 하네스의 가정은 상한다

Anthropic Engineering Blog의 반복 주제: 하네스는 "Claude가 스스로 할 수 없는 것"에 대한 가정을 인코딩한다. 그러나 그 **가정은 모델이 개선되면서 상한다(go stale)**.

**구체적 사례**: Claude Sonnet 4.5에서 컨텍스트 한계가 다가오면 작업을 조기에 마무리하는 "context anxiety" 행동이 있었다. 이를 해결하기 위해 하네스에 context reset을 추가했다. 그런데 같은 하네스를 Opus 4.5에 적용하니 — 그 행동 자체가 사라져 있었다. **Reset 코드가 죽은 무게(dead weight)** 가 된 것이다.

→ 하네스는 계속 진화할 것이므로, **특정 구현이 아니라 인터페이스가 오래 살아남도록** 설계해야 한다. 이것이 Managed Agents의 설계 동기.

---

## 2. "아직 생각하지 못한 프로그램을 위한 시스템"

핵심 설계 문제는 컴퓨팅 역사의 오래된 질문: **"programs as yet unthought of"를 위한 시스템을 어떻게 설계하는가?**

수십 년 전, OS는 하드웨어를 **process, file** 같은 추상화로 가상화하여 해결했다. `read()` 명령은 1970년대 디스크팩이든 현대 SSD든 상관없이 작동한다. **추상화가 하드웨어보다 오래 살아남았다.**

Managed Agents도 같은 패턴:
- **Session**: 일어난 모든 일의 append-only 로그
- **Harness**: Claude를 호출하고 도구 호출을 라우팅하는 루프
- **Sandbox**: 코드 실행 및 파일 편집 환경

각 구성 요소의 구현을 다른 것에 영향 주지 않고 교체 가능. **인터페이스의 형태에 대해서는 의견이 있지만, 그 뒤에서 무엇이 실행되는지에 대해서는 의견이 없다.**

## 3. 반려동물 컨테이너의 문제

### 3.1 초기 설계: 모든 것이 하나의 컨테이너

처음에는 세션, 하네스, 샌드박스를 **하나의 컨테이너**에 넣었다. 장점: 파일 편집이 직접 syscall, 서비스 경계 설계 불필요.

### 3.2 "반려동물"이 되어버린 컨테이너

Pets vs Cattle 비유:
- **Pet**: 이름이 있고, 손으로 돌보는 개체. 잃을 수 없음
- **Cattle**: 교체 가능한 개체

컨테이너가 **pet**이 되었다:

**문제 1 — 세션 유실**: 컨테이너 실패 → 세션 소실. 응답 없는 컨테이너를 간호(nurse)해야 했다.

**문제 2 — 디버깅 불가**: 유일한 관찰 창구가 WebSocket 이벤트 스트림뿐. 하네스 버그인지, 패킷 드롭인지, 컨테이너 오프라인인지 구분 불가. 디버깅하려면 사용자 데이터가 있는 컨테이너에 셸을 열어야 → 실질적으로 **디버깅 능력 상실**.

**문제 3 — 인프라 결합**: 하네스가 "작업 대상이 같은 컨테이너 안에 있다"고 가정. 고객 VPC 연결 시 네트워크 피어링 필요하거나 고객 환경에서 하네스를 실행해야 → 하네스에 구워진 가정이 **인프라 확장의 장벽**.

## 4. 두뇌와 손의 분리

### 4.1 핵심 아키텍처

세 가지를 분리:
- **Brain (두뇌)** = Claude + 하네스
- **Hands (손)** = 샌드박스 + 도구
- **Session (세션)** = 이벤트의 append-only 로그

각 구성 요소가 독립적으로 실패하거나 교체될 수 있다.

### 4.2 하네스가 컨테이너를 떠난다

하네스(두뇌)가 더 이상 컨테이너 안에 살지 않는다. 컨테이너를 **다른 도구와 동일한 방식으로** 호출:

```
execute(name, input) → string
```

컨테이너가 **cattle**이 됨:
- 컨테이너 사망 → 하네스가 tool-call error로 받아서 Claude에게 전달
- Claude가 재시도 결정 → 새 컨테이너를 `provision({resources})`로 초기화
- 더 이상 죽은 컨테이너를 간호할 필요 없음

### 4.3 하네스 실패 복구

하네스도 cattle이 됨. 세션 로그가 하네스 바깥에 존재하므로:

1. 하네스 실패 시 새 하네스를 `wake(sessionId)`로 부팅
2. `getSession(id)`로 이벤트 로그 복구
3. 마지막 이벤트부터 재개
4. 실행 중에는 `emitEvent(id, event)`로 세션에 기록

### 4.4 보안 경계

결합 설계에서: Claude가 생성한 비신뢰 코드가 자격증명이 있는 **같은 컨테이너**에서 실행됨. 프롬프트 인젝션이 자기 환경을 읽도록 Claude를 설득하기만 하면 됨. 토큰을 얻으면 제한 없는 새 세션 생성 가능.

**구조적 해결**: 토큰이 샌드박스에서 **절대 접근 불가**하도록 설계.

두 가지 패턴:
1. **Git**: 접근 토큰으로 샌드박스 초기화 시 repo clone + git remote 연결. 에이전트가 토큰 자체를 핸들링하지 않음
2. **커스텀 도구 (MCP)**: OAuth 토큰을 보안 볼트에 저장. Claude는 전용 프록시를 통해 MCP 도구 호출 → 프록시가 볼트에서 자격증명 가져와서 외부 서비스 호출. **하네스가 어떤 자격증명도 알지 못함**

## 5. 세션 ≠ Claude의 컨텍스트 윈도우

### 5.1 비가역적 결정의 문제

장기 작업은 컨텍스트 윈도우를 초과한다. 기존 해결책: compaction(요약), memory tool(파일에 기록), context trimming(선택적 토큰 제거). 하지만 모두 **비가역적 결정** — 미래 턴이 어떤 토큰을 필요로 할지 알기 어렵다.

### 5.2 세션을 컨텍스트 객체로

Managed Agents에서 세션은 **Claude의 컨텍스트 윈도우 바깥에 존재하는 컨텍스트 객체**.

인터페이스 `getEvents()`:
- 이벤트 스트림의 위치 기반 슬라이스 선택
- 마지막 읽은 위치부터 이어서 읽기
- 특정 순간 직전으로 되감기
- 특정 행동 전 컨텍스트 다시 읽기

가져온 이벤트는 하네스에서 **Claude 컨텍스트 윈도우에 전달하기 전 변환** 가능: 캐시 히트율 최적화, 컨텍스트 엔지니어링 등.

**관심사 분리**: 복구 가능한 컨텍스트 저장(세션)과 임의의 컨텍스트 관리(하네스)를 분리. 미래 모델이 어떤 컨텍스트 엔지니어링을 요구할지 예측 불가하므로, 인터페이스는 **컨텍스트 관리를 하네스에 밀어넣고** 세션은 **내구성과 조회 가능성만 보장**.

## 6. Many Brains, Many Hands

### 6.1 Many Brains — 확장성

초기: 하나의 brain = 하나의 컨테이너. 모든 세션이 컨테이너 프로비저닝 비용(repo clone, 프로세스 부팅, 이벤트 fetch)을 선불로 지불.

**Time-to-First-Token (TTFT)** 문제: 사용자가 가장 절실하게 느끼는 지연.

분리 후: 컨테이너는 brain이 도구 호출로 **필요할 때만** 프로비저닝. 샌드박스가 필요 없는 세션은 컨테이너를 기다리지 않음.

결과: **p50 TTFT ~60% 감소, p95 TTFT 90%+ 감소**. 많은 brain 확장 = stateless 하네스를 많이 시작하고, hands는 필요 시에만 연결.

### 6.2 Many Hands — 다중 실행 환경

Claude가 여러 실행 환경에 대해 추론하고 어디에 작업을 보낼지 결정하는 것 — 단일 셸보다 어려운 인지 과제. 초기 모델은 이것이 불가능했기에 단일 컨테이너로 시작. 지능이 확장되면서 **단일 컨테이너가 한계**.

분리 후: 각 hand가 도구 `execute(name, input) → string`. 커스텀 도구, MCP 서버, 자체 도구 모두 지원. 하네스는 샌드박스가 **컨테이너인지, 폰인지, 포켓몬 에뮬레이터인지** 모른다. hand가 brain에 결합되지 않으므로 **brain끼리 hand를 주고받을 수 있다**.

## 7. 결론: Meta-Harness

Managed Agents는 **메타-하네스**: Claude가 미래에 어떤 특정 하네스를 필요로 할지에 대해 의견이 없다. 대신 **다양한 하네스를 수용할 수 있는 범용 인터페이스** 를 가진 시스템.

인터페이스에 대한 의견:
- Claude는 **상태를 조작**(세션)하고 **계산을 수행**(샌드박스)할 능력이 필요
- **다수의 brain과 hand**로 확장할 능력이 필요
- 이것이 장기 시간대에 걸쳐 **안정적이고 안전**하게 실행되어야

의견이 없는 것:
- Claude가 필요로 할 brain과 hand의 **수나 위치**

## Soo에게 의미 있는 이유

1. **에이전트 인프라 설계의 교과서**: Pets vs Cattle 패턴을 에이전트 아키텍처에 적용하는 완전한 사례
2. **"메타-하네스" 개념**: 특정 하네스가 아닌, 하네스를 수용하는 시스템 설계 — Starforge 프로젝트의 설계 방향과 직결
3. **보안 설계 패턴**: 자격증명을 샌드박스에서 분리하는 두 가지 구체적 패턴 (Git, MCP 프록시)
4. **하네스 가정의 유통기한**: 모델 발전 시 하네스 코드가 죽은 무게가 될 수 있다는 실전 교훈
5. **TTFT 60-90% 개선**: brain/hands 분리의 직접적 성능 효과 — 고객 제안서에 인용 가능
