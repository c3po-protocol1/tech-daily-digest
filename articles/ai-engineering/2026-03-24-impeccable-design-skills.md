## 1. 문제 정의 — "AI Slop"이란 무엇인가

모든 LLM은 **같은 제네릭 템플릿**에서 학습했다. 그래서 가이드 없이 프론트엔드 코드를 생성하면, 예측 가능하게 같은 실수를 반복한다:

- **Inter 폰트** — 어디서나 보이는, 개성 없는 기본 선택
- **보라색-파란색 그라데이션** — "AI가 만들었습니다"를 광고하는 색상 조합
- **카드 안에 카드** — 모든 것을 카드로 감싸고, 그 카드를 다시 카드 안에 넣는 패턴
- **회색 텍스트 + 유색 배경** — 세탁된 것처럼 보이는 대비 부족
- **순수 검정(#000)과 순수 흰색(#fff)** — 자연에는 존재하지 않는 극단적 색상
- **네온 악센트 + 다크 모드** — "쿨해 보이지만" 실제 디자인 결정을 회피한 것
- **동일한 카드 그리드 반복** — 아이콘 + 제목 + 텍스트가 같은 크기로 끝없이 반복

이것이 **"AI Slop(AI 찌꺼기)"**이다. 기술적으로 작동하지만, 한눈에 "AI가 만들었구나"라는 인상을 주는 제네릭한 디자인. Impeccable은 이 문제를 체계적으로 해결하기 위해 만들어졌다.

## 2. 구조 — 1 스킬, 7 레퍼런스, 20 커맨드

Impeccable은 Anthropic이 만든 `frontend-design` 스킬을 기반으로, 더 깊은 전문성과 더 세밀한 제어를 추가한 확장판이다.

### 핵심 스킬: `frontend-design`

하나의 종합 디자인 스킬과 7개의 도메인별 레퍼런스 파일로 구성된다:

| 레퍼런스 | 다루는 영역 |
|---|---|
| **typography** | 타입 시스템, 폰트 페어링, 모듈러 스케일, OpenType, 가독성 |
| **color-and-contrast** | OKLCH 색상 공간, 틴티드 뉴트럴, 다크 모드, 접근성 대비 |
| **spatial-design** | 간격 시스템, 그리드, 시각적 리듬, 컨테이너 쿼리 |
| **motion-design** | 이징 곡선, 스태거링, 감소 모션(prefers-reduced-motion) |
| **interaction-design** | 폼, 포커스 상태, 로딩 패턴 |
| **responsive-design** | 모바일 퍼스트, 플루이드 디자인, 컨테이너 쿼리 |
| **ux-writing** | 버튼 레이블, 에러 메시지, 빈 상태 문구 |

## 4. 레퍼런스의 깊이 — 왜 이것이 "프롬프트 모음"이 아닌가

Impeccable의 진짜 가치는 7개 레퍼런스 파일에 담긴 **전문적 디자인 지식의 깊이**에 있다. 몇 가지 예를 보면:

### 타이포그래피 레퍼런스
- **"Inter, Roboto, Open Sans, Lato, Montserrat를 피하라"** — 이 폰트들은 어디서나 보여서 디자인이 제네릭해진다
- **대안 제시:** Inter 대신 → Instrument Sans, Plus Jakarta Sans, Outfit. Roboto 대신 → Onest, Figtree, Urbanist
- **"두 번째 폰트가 꼭 필요한 건 아니다"** — 하나의 폰트 패밀리를 여러 가중치로 사용하는 것이 두 개의 충돌하는 서체보다 깔끔하다
- **다크 모드에서 line-height를 0.05-0.1 늘려라** — 밝은 배경 대비 어두운 텍스트보다 지각적 가중치가 가벼워서 더 많은 여백이 필요

### 색상 레퍼런스
- **"HSL을 버리고 OKLCH를 써라"** — HSL에서 50% 밝기의 노란색과 50% 밝기의 파란색은 실제로 같은 밝기로 보이지 않는다. OKLCH는 지각적으로 균일(perceptually uniform)
- **"순수 회색은 죽었다"** — 모든 뉴트럴에 브랜드 색상의 미세한 힌트(chroma 0.01)를 추가하면 무의식적 일관성이 생긴다
- **밝기 극단에서 채도를 줄여라** — 극단적 밝기에서 높은 채도는 저렴해 보인다

### 모션 레퍼런스
- **100/300/500 규칙:** 즉각 피드백(100-150ms), 상태 변화(200-300ms), 레이아웃 변화(300-500ms)
- **"`ease`를 쓰지 마라"** — 절충안이라 거의 최적이 아니다. ease-out(진입), ease-in(퇴장), ease-in-out(토글)을 구분하라
- **"바운스/일래스틱 이징을 피하라"** — 2015년에 트렌디했지만 지금은 촌스럽다. 실제 물체는 멈출 때 튀지 않고 부드럽게 감속한다

## 5. Context Gathering Protocol — "코드만 봐서는 모른다"

Impeccable의 가장 스마트한 설계 결정 중 하나는 **Context Gathering Protocol**이다. 디자인 스킬은 프로젝트 컨텍스트 없이는 제네릭한 결과만 낸다. 따라서 세 가지 필수 컨텍스트를 반드시 확보한 후에만 디자인 작업을 시작한다:

1. **대상 사용자(Target audience):** 누가, 어떤 맥락에서 사용하는가?
2. **유스케이스(Use cases):** 어떤 작업을 수행하려 하는가?
3. **브랜드 성격/톤(Brand personality/tone):** 인터페이스가 어떤 느낌이어야 하는가?

**핵심 원칙: "코드를 읽어서는 이 컨텍스트를 추론할 수 없다. 코드는 무엇이 만들어졌는지 알려주지, 누구를 위한 것인지나 어떤 느낌이어야 하는지는 알려주지 않는다."**

컨텍스트 확보 순서:
1. 현재 로드된 instruction에 Design Context가 있으면 → 즉시 진행
2. `.impeccable.md` 파일이 있으면 → 읽고 진행
3. 둘 다 없으면 → `/teach-impeccable`을 **반드시** 먼저 실행 (건너뛸 수 없음)

## 7. `/audit` 스킬의 깊이 — 4차원 품질 검사

`/audit`는 단순한 "잘 만들었는지 확인"이 아니라, 4가지 차원에서 체계적 검사를 수행한다:

1. **접근성(A11y):** 대비율 < 4.5:1 확인, ARIA 누락, 키보드 내비게이션, 시맨틱 HTML, 대체 텍스트, 폼 레이블
2. **성능:** 레이아웃 스레싱(읽기/쓰기 루프), `width/height` 대신 `transform/opacity` 애니메이션, 이미지 레이지 로딩, 번들 크기
3. **테마:** 하드코딩된 색상(토큰 미사용), 다크 모드 깨짐, 테마 스위칭 문제
4. **반응형:** 고정 너비, 터치 타겟 < 44x44px, 수평 스크롤 발생

**`/audit`는 수정하지 않고 리포트만 생성한다.** 수정은 `/normalize`, `/polish` 등 다른 커맨드의 역할이다. 이 역할 분리가 gstack의 철학과도 일치한다 — 분석과 실행을 분리하여 각 단계의 품질을 보장한다.

## 8. Design Direction 철학 — "대담함과 의도성"

Impeccable의 스킬에 포함된 Design Direction 가이드라인은 에이전트에게 **"대담한 미학적 방향에 커밋하라"**고 지시한다:

- **톤을 극단적으로 선택하라:** 잔혹한 미니멀리즘, 맥시멀리즘 카오스, 레트로-퓨처리스틱, 유기적/자연적, 럭셔리/세련, 장난스러운, 에디토리얼/매거진, 브루탈리스트, 아르데코, 소프트/파스텔, 산업적/실용적 등
- **핵심 원칙:** "대담한 맥시멀리즘과 세련된 미니멀리즘 모두 유효하다 — 핵심은 강도(intensity)가 아니라 **의도성(intentionality)**이다"
- **차별화 질문:** "이것을 잊을 수 없게 만드는 것은 무엇인가? 누군가가 기억할 한 가지는?"

이것이 AI Slop과의 근본적 차이이다. AI Slop은 **의도 없이 안전한 기본값**을 선택한 결과이고, 좋은 디자인은 **의도적으로 대담한 결정**을 내린 결과이다.

## 9. Soo에게 의미 있는 이유

### AInD 컨설팅에서의 활용

**첫째, "AI가 만든 것은 품질이 떨어진다"는 우려에 대한 직접적 반론.** Impeccable을 적용한 before/after를 보여주면, 스킬(하네스)이 얼마나 극적으로 AI 출력물의 품질을 변화시키는지 증명할 수 있다. impeccable.style에 실제 사례가 공개되어 있다.

**둘째, 하네스 엔지니어링의 가장 직관적인 데모 소재.** 기술적이지 않은 경영진에게 "하네스란 무엇인가"를 설명할 때, 코드 품질보다 **디자인 품질의 차이**가 훨씬 직관적이다. 같은 AI에게 같은 요청을 하되, Impeccable 유/무의 결과를 비교하면 하네스의 가치가 한눈에 보인다.

**셋째, gstack과의 조합.** gstack의 `/plan-design-review`와 Impeccable의 `/audit`, `/critique`를 결합하면, 디자인 품질 관리의 완전한 파이프라인이 구성된다. gstack이 프로세스(Think→Ship)를 제공하고, Impeccable이 디자인 품질의 기준을 제공한다.

**넷째, 기업 맞춤 디자인 스킬의 템플릿.** 기업에 AI를 도입할 때, 해당 기업의 디자인 시스템(색상, 타이포그래피, 컴포넌트 규칙)을 Impeccable 형식의 스킬로 변환하면, AI가 기업의 브랜드 가이드라인을 준수하는 코드를 생성하게 된다. 이것이 구체적인 컨설팅 딜리버리가 된다.
