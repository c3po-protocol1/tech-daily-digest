# 데이터만이 유일한 해자다 — AI 시대 소프트웨어 비즈니스의 경쟁 우위

## 1. 핵심 질문 — "소프트웨어를 만드는 것이 쉬워지면, 비즈니스의 기반은 무엇인가?"

저자 Arvid Kahl이 던지는 질문은 단순하지만 근본적이다:

> **"소프트웨어를 만드는 것이 계속 쉬워지고 있다. 그렇다면 우리는 대체 무엇 위에 비즈니스를 세우고 있는가?"**

LLM 기반 도구(Claude Code, Cursor, Codex 등)로 복잡한 소프트웨어 개발이 획기적으로 쉬워졌다. 아직 완전히 해결되지는 않았지만, 의미 있는 것을 만드는 데 10명이 필요하던 것이 3명, 2명, 혹은 1명으로 줄어들고 있다. 소프트웨어 제품을 쉽게 만들고, 배포하고, 유지할 수 있게 되면 — 전통적으로 "해자(moat)"라고 불리던 것들이 무너진다:

- **개발 난이도?** → AI가 대체
- **도메인 지식의 제품화?** → AI가 학습하여 대체
- **기술적 복잡성?** → AI가 구현하여 대체

그럼 무엇이 남는가? Kahl의 답: **실세계의 인간 생성 데이터만이 유일한 해자이다.**

## 2. The Great Data Bifurcation — 데이터의 대분기

Kahl은 데이터 세계가 두 갈래로 **양분(bifurcation)**되고 있다고 분석한다:

### 인간 생성 데이터 (가치 상승)
- 팟캐스트 에피소드, 동영상, 소셜 미디어 글, 블로그 포스트
- 사람이 직접 만든 모든 콘텐츠
- **희소성과 고유성**으로 인해 가치가 계속 상승
- 해당 데이터를 생성할 수 있는 유일한 주체는 그 데이터를 만든 사람 자신
- 생성자 본인만이 가진 지식, 창의성, 노력, 전문성, 배타성이 담겨 있음

### AI 생성 데이터 (범용재로 전락)
- AI 이미지, TTS 합성 음성, 완전 AI 제작 영상, 에이전트가 작성한 이메일
- 모델이 빠르고 저렴해질수록 **범용재(commodity)**로 전락
- AI 생성 데이터도 가치가 있을 수 있지만, 그것은 commodity 수준의 가치

핵심 논리: **AI는 정의상 인간 생성 데이터를 만들 수 없다.** 따라서 실세계의 인간 생성·검증·정제된 데이터가 향후 10년간 소프트웨어 창업자의 유일하게 신뢰할 수 있는 해자이다.

## 3. Podscan 사례 — 데이터 해자의 실제 작동

Kahl은 자신이 운영하는 **Podscan**(팟캐스트 모니터링 서비스)을 통해 데이터 해자가 실제로 어떻게 작동하는지 설명한다.

### Podscan의 가치는 어디에 있는가?

가치가 **있는** 것:
- **5,000만 건의 팟캐스트 에피소드** 전사(transcription) 및 AI 분석(키워드, 테마, 감성 분석) 데이터
- 최신성(freshness) 데이터, 차트 랭킹, 다양한 소스의 추가 메타데이터
- 대부분의 사람이 접근 방법조차 모르는 소스들에서 수집된 데이터
- **시스템 오브 레코드(System of Record)**로서의 포지션

가치가 **없는** 것:
- RSS 피드 수집 속도
- API 응답 속도
- UI/UX 품질

Kahl이 발견한 놀라운 사실: **"UI가 불편하거나 API가 제한적이어도 고객은 데이터에 접근하는 방법을 찾아낸다. 데이터 자체가 관건이지, 접근 수단이 아니다."**

### 만약 Podscan이 데이터 수집 없이 "URL을 주면 전사·분석해 주는 기능만" 제공한다면?

> **"Claude Code 안의 잘 작성된 스킬 하나로 2시간 내에 대체될 것이다."**

이것이 "변환형 소프트웨어(transformative software)"의 취약성이다. 입력을 받아 처리하고 출력하는 순수 변환 기능은 에이전트 AI가 직접 수행할 수 있다.

그러나 **하루 5만 건의 에피소드를 수집·전사·분석하는 대규모 지속적 데이터 수집**은 에이전트가 대체하기 어렵다 — 토큰 소비가 막대하여 하루 수만 달러의 API 비용이 발생하기 때문이다.

## 4. 변환형 소프트웨어의 취약성 (Transformative Software)

Kahl이 가장 강력하게 경고하는 것은 **순수 변환형 소프트웨어 비즈니스의 소멸**이다.

예시: "ChatGPT야, 이 Excel 파일로 리포트 만들어서 PDF로 내보내고 이메일 보내줘"

에이전트 AI는 이것을 자율적으로 수행할 수 있다:
1. Excel 파싱 → 분석 쿼리 실행 → 결과 생성
2. PDF 렌더링
3. 이메일 발송

외부 서비스 없이, 사람 입력 없이, 에이전트 혼자서. **이런 워크플로를 위한 SaaS 비즈니스는 더 이상 필요하지 않다.**

위험한 비즈니스 유형:
- 파일 형식 변환 서비스 (PDF→Word, 이미지 리사이징 등)
- 단순 데이터 변환 파이프라인 (ETL without unique data)
- 템플릿 기반 문서 생성 서비스
- 단일 기능의 마이크로 SaaS

안전한 비즈니스 유형:
- **대규모 지속적 데이터 수집** (에이전트의 토큰 비용 한계)
- **시스템 오브 레코드** (축적된 데이터의 가치)
- **네트워크 효과가 있는 데이터** (사용자가 많을수록 데이터 가치 상승)
- **메타데이터 기반 인사이트** (사용 패턴에서 추출된 고유 데이터)

## 5. 에이전트의 구조적 한계 — "상시 에이전트"의 비용 문제

Kahl의 분석에서 특히 통찰력 있는 부분은 **에이전트의 "상시 작동(always-on)" 한계**에 대한 분석이다.

> "에이전트의 일시적(ephemeral) 특성을 생각해보라. Cursor 세션을 열거나, Claude Code 인스턴스를 생성하거나, 브라우저에서 ChatGPT와 대화할 때 — 그 짧은 상호작용 순간에만 에이전트가 존재한다. 나머지는 상태 머신의 항목에 불과하다."

**상시 스캔, 상시 작업하는 에이전트**는 토큰 소비가 막대하여 경제적으로 비현실적이다. Podscan이 하루 5만 건의 에피소드를 수집·전사·분석하는 작업을 에이전트에게 시키면, **하루 수만 달러의 토큰/API 비용**이 발생한다.

이것이 데이터 수집 비즈니스가 에이전트에 의해 대체되기 어려운 구조적 이유이다. 에이전트는 세션 기반이고, 데이터 수집은 상시 기반이다. 이 간극이 해자가 된다.

## 6. API 퍼스트 전략 — "사용 퍼리티(Usage Parity)"

Kahl은 데이터 해자를 가진 비즈니스가 취해야 할 전략으로 **API 퍼스트**를 제안한다.

핵심 개념: **세 가지 사용자 유형에 대한 기능 동등성(parity)**

| 사용자 유형 | 접근 방식 |
|---|---|
| **인간 사용자** | Web UI |
| **컴퓨터 사용자** | REST API |
| **에이전트 사용자** | MCP (Model Context Protocol) |

UI에서 할 수 있는 모든 것을 API로도 동일하게 할 수 있어야 하고, API로 할 수 있는 모든 것을 MCP로도 할 수 있어야 한다.

Kahl은 실제로 **Claude Code의 서브에이전트**를 며칠마다 돌려서 "platform parity tracking file"을 업데이트한다. 모든 기능이 UI, REST API, MCP 세 곳에서 동일하게 사용 가능한지 추적하고, 누락된 것을 구현 대상으로 관리한다.

이 접근법은 **"에이전트가 고객이 되는 시대"**를 대비한 것이다. 고객의 에이전트가 자사 제품에 자동으로 접근할 수 있어야 하며, 이를 위해 API/MCP의 완전한 기능 지원이 필수적이다.

## 7. 메타데이터가 곧 해자 — "어떤 비즈니스에서든 적용 가능"

마지막으로 Kahl은 데이터 해자가 팟캐스트 데이터에만 국한되지 않는다고 강조한다:

> **"사람들이 당신의 플랫폼을 사용할 때, 또는 당신의 제품 사용의 결과를 관찰할 때 수집되는 메타데이터 — 그것이 당신의 고유한 데이터 해자이다."**

예시:
- **소셜 미디어 관리 도구** → 대부분의 사람이 언제 포스팅하는지, 어떤 콘텐츠가 가장 많은 인게이지먼트를 유발하는지의 메타데이터
- **프로젝트 관리 도구** → 팀의 작업 패턴, 병목 지점, 생산성 트렌드의 메타데이터
- **CRM** → 고객 행동 패턴, 구매 주기, 이탈 신호의 메타데이터
- **코드 에디터** → 개발자의 코딩 패턴, 가장 흔한 에러, 생산성 최고점 시간의 메타데이터

이 메타데이터는 **사용자가 만들어주는 데이터**이며, AI가 생성할 수 없고, 시간이 지날수록 축적되어 가치가 증가한다. 그리고 이 데이터를 활용하여 **사용자의 경험을 개선하는 피드백 루프**를 만들면, 네트워크 효과와 결합된 가장 견고한 해자가 된다.

## 8. Soo에게 의미 있는 이유

### AInD 컨설팅에 대한 직접적 함의

**첫째, 컨설팅의 근본적 질문이 바뀐다.** "AI로 무엇을 만들 것인가?"보다 **"AI 시대에 당신의 경쟁 우위는 무엇인가?"**를 먼저 질문해야 한다. 기업의 기존 비즈니스가 "변환형 소프트웨어"에 해당하면, AI에 의해 대체될 위험을 먼저 경고하고, 데이터 기반 해자를 구축하는 전략을 제안해야 한다.

**둘째, "데이터 해자 진단"이 컨설팅 서비스가 된다.** 기업이 보유한 데이터 자산을 감사하고, 어떤 데이터가 AI에 의해 복제 불가능한 고유 가치를 가지는지 식별하며, 그 데이터를 활용한 비즈니스 전략을 수립하는 것이 하나의 컨설팅 패키지가 될 수 있다.

**셋째, API 퍼리티 전략이 기술 컨설팅 딜리버리가 된다.** 기업의 제품에 "인간/컴퓨터/에이전트" 세 유형의 사용자가 동등하게 접근할 수 있는 API를 설계하는 것. Kahl이 실제로 하고 있는 "platform parity tracking" 방법론을 기업에 도입하는 것도 구체적 딜리버리이다.

**넷째, "에이전트가 고객이 되는 시대"를 대비하라.** 기업 고객의 에이전트가 자사 제품을 직접 사용하는 시나리오가 현실화되고 있다. MCP 통합, API 완전성, 에이전트 친화적 데이터 구조 — 이것들이 향후 제품 경쟁력의 핵심이 된다.

**다섯째, Cohen의 "Creating Space"와 연결된다.** 어제 정리한 "여백 만들기" 글과 결합하면 강력한 메시지가 된다: "모든 것을 AI로 하려 하지 말고, **AI가 대체할 수 없는 한 가지(데이터)**에 불균형적으로 집중하라."
