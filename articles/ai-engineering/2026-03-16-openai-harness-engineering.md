# OpenAI: Harness Engineering — 코드 한 줄 안 쓰고 백만 라인 코드베이스 구축하기

> **원문:** [Harness Engineering - OpenAI](https://openai.com/index/harness-engineering/)
> **날짜:** 2026년 3월
> **출처:** OpenAI 공식 블로그

---

## 📌 핵심 요약

OpenAI의 소규모 팀이 **5개월 동안 사람이 직접 코드를 한 줄도 작성하지 않고** 실제 내부 소프트웨어 제품을 개발·출시했다. 모든 코드(애플리케이션 로직, 테스트, CI 설정, 문서, 관측 도구, 내부 스크립트)를 **Codex가 작성**했으며, 약 **100만 라인의 코드**를 **1,500개의 PR**로 머지했다. 피크 시 3명의 엔지니어가 하루 평균 3.5개의 PR을 처리했고, 팀이 7명으로 늘었을 때 처리량은 오히려 증가했다. 기존 대비 **약 1/10의 시간**으로 제품을 완성했다.

---

## 🔧 엔지니어의 역할 재정의

- 코드를 직접 작성하지 않는 제약이 **엔지니어링의 의미 자체를 근본적으로 변화**시켰다
- 초기에는 Codex의 능력 부족이 아닌 **환경의 명세 부족**이 병목이었다 — 에이전트에게 도구, 추상화, 구조가 부족했음
- 엔지니어들은 **깊이 우선 문제 분해(depth-first problem decomposition)** 방식으로 전환
  - 목표를 작은 빌딩 블록으로 분해 → Codex가 구축 → 이를 기반으로 더 복잡한 작업 수행
- 실패 시 "더 열심히"가 아닌 **"어떤 기능(capability)이 빠져있는가?"** 를 질문
- 인간은 거의 **프롬프트를 통해서만** 시스템과 상호작용
- 코드 리뷰의 대부분은 **에이전트 간 리뷰(agent-to-agent)** 로 이전

## 👁️ 에이전트에게 눈과 귀를 제공

- 코드 처리량이 늘어나면서 병목이 **인간의 QA 능력**으로 이동
- 대응 방법:
  - 앱을 **git worktree 단위로 부팅 가능**하게 만들어 Codex가 변경 사항별로 격리된 인스턴스를 실행
  - **Chrome DevTools Protocol**을 에이전트 런타임에 연결 → DOM 스냅샷, 스크린샷, 네비게이션 가능
  - Codex가 직접 버그를 재현하고, 수정을 검증하고, UI 동작을 추론
  - **관측(Observability) 도구**: worktree별 임시 로컬 스택 → LogQL/PromQL로 쿼리 가능
- "서비스 시작이 800ms 이내에 완료되도록 보장하라" 같은 프롬프트가 **직접 실행 가능**
- 단일 Codex 실행이 **최대 6시간** 동안 하나의 작업을 수행 (인간이 자는 동안에도)

## 📚 저장소가 유일한 진실의 원천 (Single Source of Truth)

컨텍스트 관리가 **가장 어려운 과제 중 하나**였다:

- 처음에는 큰 단일 `AGENTS.md` 파일을 시도 → 실패
  - 컨텍스트는 희소하고, 가이드가 너무 많으면 오히려 무가이드
  - 단일 파일은 빠르게 부패하며, 기계적 검증이 어려움
- **해결책**: `AGENTS.md`를 **목차(~100줄)**로 축소하고, 실제 지식 기반은 구조화된 `docs/` 디렉토리에 저장
  - 검증 상태가 있는 설계 문서
  - 도메인·패키지 레이어링이 있는 아키텍처 맵
  - 진행 상황·결정 로그가 있는 실행 계획
  - 각 제품 도메인별 품질 등급 문서
  - 기술 부채 트래커
- **전용 린터와 CI 작업**이 지식 기반의 최신성, 상호 링크, 구조 정확성을 강제
- **"문서 정원사(doc-gardening)" 에이전트**가 주기적으로 오래된 문서를 스캔하고 수정 PR을 자동 생성

## 🏗️ 회의가 아닌 기계로 강제하는 아키텍처

- 각 비즈니스 도메인을 고정된 레이어 세트로 분할, **의존성 방향을 엄격히 검증**
- 의존성 흐름: `Types → Config → Repo → Service → Runtime → UI` (단방향만 허용)
- 교차 관심사(cross-cutting concerns)는 단일 명시적 인터페이스를 통해서만 진입
- **커스텀 린터와 구조 테스트**(Codex가 생성)로 기계적으로 강제
- 구조화된 로깅, 네이밍 컨벤션, 파일 크기 제한, 플랫폼별 신뢰성 요구사항 등을 정적으로 강제
- 린트 에러 메시지에 **수정 지침을 에이전트 컨텍스트에 주입**하도록 작성

> *"이것은 보통 수백 명의 엔지니어가 있을 때까지 미루는 종류의 아키텍처입니다. 코딩 에이전트와 함께라면, 이것은 초기 필수 조건입니다."*

## 🚀 처리량이 기존 머지 철학을 뒤집다

- **최소한의 차단 머지 게이트**, 짧은 수명의 PR, 테스트 실패는 후속 실행으로 해결
- 에이전트 처리량이 인간 주의력을 크게 초과하는 시스템에서, **수정은 저렴하고 기다림은 비싸다**
- 저처리량 환경에서는 무책임하겠지만, 여기서는 올바른 트레이드오프

## 🧹 엔트로피, 가비지 컬렉션, "AI 슬롭"

- Codex는 저장소에 이미 존재하는 패턴을 복제 — 비균일하거나 차선의 패턴 포함
- 초기에는 **매주 금요일(주의 20%)을 "AI 슬롭" 정리**에 투입 → 확장 불가
- **해결책**: "골든 원칙(golden principles)"을 저장소에 직접 인코딩
  - 백그라운드 Codex 작업이 주기적으로 편차를 스캔
  - 품질 등급 업데이트 및 타겟 리팩토링 PR 생성
  - 대부분 1분 이내에 리뷰 가능, 자동 머지 대상
- **기술 부채를 고금리 대출로 프레이밍** — 고통스러운 한 번이 아닌 지속적인 소액 상환
- 인간의 취향(taste)을 한 번 캡처하고, 모든 코드 라인에 지속적으로 강제

## 🔮 아직 배우고 있는 것들

- 수백 명의 내부 사용자(매일 사용하는 파워 유저 포함)에게 출시 완료
- 아직 미지의 영역:
  - 완전 에이전트 생성 시스템에서 **수년간 아키텍처 일관성**이 어떻게 진화하는지
  - **인간 판단이 가장 큰 레버리지를 발휘**하는 지점
  - 기반 모델이 계속 개선됨에 따라 시스템이 어떻게 변화하는지

> **핵심 교훈:** 소프트웨어 구축에는 여전히 규율이 필요하지만, 그 규율은 이제 **코드가 아닌 스캐폴딩(도구, 추상화, 피드백 루프)**에서 나타난다. 코드베이스를 일관되게 유지하는 이러한 스캐폴딩이 바로 **희소하고 복리로 성장하는 투자**다.

---

## 💡 왜 중요한가

이 글은 단순한 "AI가 코딩한다" 사례가 아니다. **AI 네이티브 개발(AInD)의 실전 청사진**이다. 엔지니어가 코드 작성자에서 **시스템 설계자·환경 구축자·품질 관리자**로 역할이 변화하는 구체적 과정을 보여준다. 에이전트의 성공은 에이전트 자체의 능력이 아닌 **에이전트가 일할 수 있는 환경(harness)의 품질**에 달려 있다는 것이 핵심이다.
