## 핵심 기법: Frequent Intentional Compaction (빈번한 의도적 압축)

### 개념

**전체 개발 워크플로우를 컨텍스트 관리 중심으로 설계.** 컨텍스트 사용률을 **40~60% 범위**에 유지하고, 파이프라인의 정확한 지점에 고레버리지 인간 리뷰를 구축.

### 왜 컨텍스트에 집착해야 하는가

> "LLM은 무상태 함수다. 출력의 품질에 영향을 미치는 유일한 것은 (모델 자체를 훈련/튜닝하지 않는 한) 입력의 품질이다."

에이전트의 어느 주어진 시점에서의 턴은 **무상태 함수 호출**: 컨텍스트 윈도우 들어가면, 다음 스텝 나옴. **컨텍스트 윈도우가 출력 품질에 영향을 미칠 수 있는 유일한 레버.**

컨텍스트 윈도우 최적화 우선순위:
1. **정확성(Correctness)** — 잘못된 정보가 최악
2. **완전성(Completeness)** — 빠진 정보가 두 번째로 나쁨
3. **크기(Size)** — 너무 많은 노이즈가 세 번째
4. **궤적(Trajectory)** — 올바른 방향으로의 진행

## Research → Plan → Implement 워크플로우

### Step 1: Research (연구)
코드베이스를 이해. 이슈에 관련된 파일, 정보 흐름, 문제의 잠재적 원인을 파악.

- 서브에이전트를 사용하여 **신선한 컨텍스트 윈도우에서** 검색/요약 수행
- 부모 에이전트의 컨텍스트를 Glob/Grep/Read 호출로 오염시키지 않음
- 결과: 구조화된 연구 문서
- [실제 프롬프트 공개](https://github.com/humanlayer/humanlayer/blob/main/.claude/commands/research_codebase.md)

**핵심 교훈: 첫 연구가 잘못될 수 있다.** BAML 예시에서 Claude가 "이 버그는 유효하지 않고 코드베이스가 맞다"고 결론. **연구를 버리고 더 많은 조향(steering)으로 새 연구를 시작.** → 이것이 "인간 리뷰의 고레버리지 포인트"

### Step 2: Plan (계획)
이슈 수정을 위한 정확한 단계 개요. 편집할 파일과 방법, 각 단계의 테스트/검증 단계를 **매우 구체적으로**.

- [실제 프롬프트 공개](https://github.com/humanlayer/humanlayer/blob/main/.claude/commands/create_plan.md)
- 연구 없는 계획 vs 연구 기반 계획 비교: 둘 다 "작동"하지만, 연구 기반 계획이 **최적의 위치에서 문제를 수정**하고 코드베이스 관습에 맞는 테스트를 지정

### Step 3: Implement (구현)
계획을 단계별로 실행. 복잡한 작업의 경우, 각 구현 단계가 검증된 후 **현재 상태를 원래 계획 파일로 압축(compact)**.

- [실제 프롬프트 공개](https://github.com/humanlayer/humanlayer/blob/main/.claude/commands/implement_plan.md)
- Git worktree가 필요한 것은 이 단계만. 연구/계획은 main에서 수행

## 서브에이전트 = 컨텍스트 제어

> "서브에이전트는 역할극하고 의인화하는 것이 아니다. **컨텍스트 제어에 관한 것**이다."

가장 일반적 사용 사례: 신선한 컨텍스트 윈도우에서 찾기/검색/요약을 수행하여, 부모 에이전트가 중간 노이즈 없이 바로 작업에 착수할 수 있게 함.

## 인간 레버리지의 핵심 원칙

> **나쁜 코드 한 줄은... 나쁜 코드 한 줄이다.**
> 하지만 **계획의 나쁜 한 줄**은 수백 줄의 나쁜 코드로 이어질 수 있다.
> 그리고 **연구의 나쁜 한 줄** — 코드베이스의 작동 방식이나 특정 기능의 위치에 대한 오해 — 은 **수천 줄의 나쁜 코드**로 이어질 수 있다.

따라서 **파이프라인의 가장 높은 레버리지 부분에 인간의 노력과 주의를 집중**해야 한다.

연구와 계획을 리뷰할 때 코드를 리뷰할 때보다 **더 높은 레버리지**를 얻는다.

## 코드 리뷰의 재정의

> "2,000줄의 복잡한 Go 코드를 매일 주의 깊게 읽는 것은 지속 가능하지 않았다."

문제는 코드의 정확성이 아니라 **팀의 정신적 정렬(mental alignment)**: "우리 제품이 무엇이고 어떻게 작동하는지에 대한 감각을 잃어가고 있었다."

해결: 코드 리뷰 → 스펙/계획/연구 리뷰로 전환
- 2,000줄의 Go 코드를 매일 읽을 수는 없다
- **200줄의 잘 작성된 구현 계획은 읽을 수 있다**
- 40개 이상 파일의 데몬 코드를 1시간 이상 탐색할 수는 없다 (할 수는 있지만 하고 싶지 않다)
- **연구 프롬프트를 조향하여 어디를 봐야 하고 왜인지의 속성 파악은 할 수 있다**

## 실패 사례도 공유

**Parquet Java에서 Hadoop 의존성 제거** 시도 — 7시간을 투자했지만 실패:
- 연구 단계가 의존성 트리를 **충분히 깊게** 들어가지 않음
- 클래스를 upstream으로 이동하면 깊이 중첩된 hadoop 의존성이 도입된다는 것을 가정하지 못함
- **코드베이스 전문가가 최소 1명은 필요** — 이 경우 둘 다 전문가가 아니었음

> "이것은 마법이 아니다. 7시간 동안 깊이 몰입해야 한다. 하나의 마법 프롬프트로 모든 문제를 해결할 것을 찾는 사람은 실망할 것이다."

2. **"인간 레버리지"** — 코드가 아닌 연구/계획 리뷰에 집중하라는 원칙. 팀 생산성 교육의 핵심
3. **300K LOC Rust + 35K LOC in 7 hours** — 구체적 수치로 "오늘의 모델로 충분하다"를 증명
4. **실패 사례(Parquet Java)의 솔직한 공유** — 한계를 인정하면서 기법의 신뢰성을 높임
5. **$12K/월** — 3명 팀의 구체적 비용. ROI 계산에 사용 가능
6. **"스펙이 새로운 코드"** — 우리의 SPEC.md 기반 워크플로우가 바로 이 철학
