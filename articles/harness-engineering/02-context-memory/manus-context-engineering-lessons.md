## 배경

Manus는 에이전트 프레임워크를 **4번 재구축**했다. 매번 컨텍스트를 더 잘 관리하는 방법을 발견한 후였다. 이 과정을 내부에서 **"Stochastic Graduate Descent (확률적 대학원생 하강법)"**이라 부른다. 우아하지는 않지만 작동한다. 이 글은 그 과정에서 도달한 최적 지점을 공유한다.

핵심 설계 결정: Manus는 end-to-end 에이전트 모델을 훈련하는 대신 **프론티어 모델의 in-context learning 능력 위에 에이전트를 구축**하기로 했다. 이유: 모델 훈련은 반복에 수 주가 걸리지만, context engineering은 **수 시간 내에 개선을 배포**할 수 있다.

> "모델 발전이 밀물이라면, Manus는 해저에 박힌 기둥이 아니라 배가 되고 싶다."

## 원칙 1: KV-Cache를 중심으로 설계하라

> "단 하나의 메트릭을 선택해야 한다면, **KV-cache 히트율**이 프로덕션 AI 에이전트의 가장 중요한 메트릭이라고 주장하겠다."

### 왜 KV-Cache가 중요한가

에이전트의 동작 패턴: 매 반복마다 모델이 현재 컨텍스트를 기반으로 행동을 선택 → 환경에서 실행 → 관찰 결과를 컨텍스트에 추가 → 반복. **컨텍스트는 매 단계마다 성장**하지만, 출력(보통 구조화된 함수 호출)은 상대적으로 짧다.

Manus의 평균 입력 대 출력 토큰 비율: **100:1**

동일한 접두사를 가진 컨텍스트는 KV-cache를 활용 → TTFT(첫 토큰까지의 시간)와 추론 비용을 **극적으로** 줄임.

**구체적 비용 차이:** Claude Sonnet에서 캐시된 입력 토큰은 **0.30 USD/MTok**, 캐시되지 않은 것은 **3 USD/MTok** — **10배 차이**.

### KV-Cache 히트율을 높이는 실전 기법

**1. 프롬프트 접두사를 안정적으로 유지:**
- LLM의 autoregressive 특성상 **단일 토큰 차이**도 그 토큰 이후의 모든 캐시를 무효화
- **흔한 실수:** 시스템 프롬프트 시작부에 **초 단위 정밀도의 타임스탬프** 포함 → 시간을 알려주는 것은 좋지만 캐시 히트율을 죽임

**2. 컨텍스트를 append-only로 만들기:**
- 이전 행동이나 관찰을 수정하지 말 것
- **직렬화가 결정론적**이어야 함. 많은 프로그래밍 언어/라이브러리가 JSON 객체 직렬화 시 안정적인 키 순서를 보장하지 않아 **조용히 캐시를 깨뜨릴 수 있음**

**3. 캐시 브레이크포인트 명시적 표시:**
- 일부 모델 프로바이더/추론 프레임워크가 자동 점진적 접두사 캐싱을 지원하지 않음
- 최소한 시스템 프롬프트 끝에 브레이크포인트 포함

## 원칙 2: 제거하지 말고 마스킹하라 (Mask, Don't Remove)

### 문제: 도구 폭발

에이전트가 더 많은 능력을 갖추면 행동 공간이 복잡해진다. MCP의 인기로 더 가속됨.

> "사용자 설정 가능한 도구를 허용하면, 누군가는 반드시 신중하게 큐레이션한 행동 공간에 수백 개의 의문스러운 도구를 연결할 것이다."

결과: 모델이 잘못된 행동을 선택하거나 비효율적 경로를 택할 확률 증가. **중무장한 에이전트가 더 멍청해진다.**

### 동적 도구 공간의 실패

RAG 같은 방식으로 도구를 온디맨드로 로딩하는 시도를 했으나, 실험 결과 명확한 규칙 도출:

> "절대적으로 필요하지 않는 한, 반복 중간에 도구를 동적으로 추가하거나 제거하지 마라."

두 가지 이유:
1. **대부분의 LLM에서 도구 정의가 컨텍스트 앞부분에 위치** → 변경 시 모든 후속 행동/관찰의 KV-cache 무효화
2. **이전 행동/관찰이 더 이상 정의되지 않은 도구를 참조**하면 모델이 혼란 → 스키마 위반이나 환각된 행동

### 해결: 컨텍스트 인식 상태 머신 + 토큰 로짓 마스킹

Manus는 도구를 제거하는 대신 **디코딩 중 토큰 로짓을 마스킹**하여 현재 컨텍스트에 기반한 특정 행동의 선택을 방지(또는 강제).

**응답 프리필을 활용한 3가지 함수 호출 모드:**
- **Auto:** 모델이 함수 호출 여부를 선택 → 응답 접두사만 프리필
- **Required:** 반드시 함수를 호출하지만 선택은 자유 → 도구 호출 토큰까지 프리필
- **Specified:** 특정 하위 집합에서만 선택 → 함수 이름 시작까지 프리필 (e.g., `"name": "browser_"`)

**도구 이름 설계:** 의도적으로 일관된 접두사 사용 — 모든 브라우저 관련 도구는 `browser_`로, 명령줄 도구는 `shell_`로 시작. 이렇게 하면 상태별 로짓 프로세서 없이도 에이전트가 특정 도구 그룹에서만 선택하도록 쉽게 강제 가능.

## 원칙 3: 파일 시스템을 컨텍스트로 사용

128K+ 토큰의 컨텍스트 윈도우도 에이전트 시나리오에서는 부족하거나 오히려 **부담**인 세 가지 이유:
1. 관찰이 거대할 수 있음 (웹 페이지, PDF와 상호작용 시)
2. 일정 컨텍스트 길이를 넘으면 모델 성능 저하
3. 긴 입력은 접두사 캐싱이 있어도 비쌈

### 왜 단순 압축이 위험한가

> "에이전트는 본질적으로 모든 이전 상태를 기반으로 다음 행동을 예측해야 하며, 10단계 후에 어떤 관찰이 중요해질지 신뢰할 수 있게 예측할 수 없다. 논리적으로 **모든 비가역적 압축은 위험**을 수반한다."

### 해결: 파일 시스템 = 궁극의 컨텍스트

Manus는 파일 시스템을 **무한 크기, 영구적, 에이전트가 직접 조작 가능한** 궁극의 컨텍스트로 취급.

**항상 복원 가능한 압축:**
- 웹 페이지 내용은 URL이 보존되면 컨텍스트에서 제거 가능
- 문서 내용은 파일 경로가 샌드박스에서 사용 가능하면 생략 가능
- 컨텍스트 길이를 줄이되 **영구적으로 정보를 잃지 않음**

## 원칙 4: 암송을 통한 주의력 조작 (Attention Manipulation Through Recitation)

Manus 사용 시 눈에 띄는 행동: 복잡한 작업을 처리할 때 **todo.md 파일을 만들고 단계별로 업데이트**하며 완료된 항목을 체크.

> "귀여운 행동이 아니라 — 주의력을 조작하는 **의도적 메커니즘**이다."

Manus의 일반적 작업은 평균 **약 50번의 도구 호출** 필요. 긴 루프에서 모델이 주제에서 벗어나거나 초기 목표를 잊는 데 취약.

**todo 리스트를 지속적으로 재작성하면:** 글로벌 계획을 컨텍스트 끝(모델의 최근 주의 범위)으로 밀어넣음. "Lost-in-the-middle" 문제를 피하고 목표 정렬 오류를 줄임. **자연어로 자신의 초점을 작업 목표 쪽으로 편향시키는 것** — 특별한 아키텍처 변경 없이.

## 원칙 5: 잘못된 것을 남겨두라 (Keep the Wrong Stuff In)

에이전트는 실수한다. 모델이 환각하고, 환경이 에러를 반환하고, 외부 도구가 오작동하고, 예상치 못한 에지 케이스가 나타난다. 멀티 스텝 작업에서 **실패는 예외가 아니라 루프의 일부**다.

> "흔한 충동은 이 에러를 숨기는 것: 트레이스를 정리하고, 행동을 재시도하고, 모델 상태를 리셋. 더 안전하고 통제되어 보이지만 대가가 있다: **실패를 지우면 증거가 사라진다.** 증거 없이 모델은 적응할 수 없다."

실패한 행동(그리고 결과 관찰이나 스택 트레이스)을 모델이 보면, **암묵적으로 내부 신념을 업데이트**한다. 유사한 행동에서 사전확률(prior)이 이동하여 같은 실수를 반복할 확률이 감소.

> "에러 복구는 진정한 에이전트 행동의 가장 명확한 지표 중 하나다. 하지만 대부분의 학술 연구와 공개 벤치마크에서 아직 과소대표되어 있다."

## 원칙 6: 자기 자신에게 Few-shot하지 마라

Few-shot 프롬프팅은 일반적 기법이지만, 에이전트 시스템에서는 **미묘한 방식으로 역효과**를 낼 수 있다.

LLM은 컨텍스트의 행동 패턴을 모방하는 데 탁월하다. 유사한 과거 행동-관찰 쌍으로 컨텍스트가 채워지면, 모델은 **더 이상 최적이 아닌데도** 그 패턴을 따르는 경향.

**예시:** 20개 이력서를 일괄 검토할 때, 에이전트가 리듬에 빠져 유사한 행동을 반복 — 단순히 컨텍스트에서 본 것이기 때문. 이는 드리프트, 과일반화, 때로는 환각으로 이어짐.

**해결:** 다양성을 도입. Manus는 행동과 관찰에 **구조화된 작은 변형**을 도입 — 다른 직렬화 템플릿, 대안적 표현, 순서나 포맷의 작은 노이즈. 이 통제된 무작위성이 패턴을 깨고 모델의 주의를 조정.

## Soo에게 의미 있는 이유

이 글은 **수백만 사용자 규모의 프로덕션 에이전트**에서 얻은 교훈이다. 이론이 아닌 실전.

**AInD 컨설팅 핵심 — 대부분의 가이드에서 다루지 않는 인프라 수준 최적화:**
1. **KV-cache 10배 비용 차이** — JSON 키 순서, 타임스탬프 위치 같은 "사소한" 것이 비용을 10배 변화시킴
2. **도구 마스킹 > 도구 제거** — 캐시를 깨지 않으면서 행동 공간을 제어하는 구체적 기법
3. **파일시스템 = 무한 컨텍스트** — 모든 압축은 복원 가능하게 설계
4. **todo.md로 주의력 조작** — LLM의 "lost in the middle" 문제를 자연어로 해결하는 창의적 패턴
5. **실패를 컨텍스트에 남겨라** — 반직관적이지만 검증된 교훈
6. **자기 자신에게 few-shot하지 마라** — 반복 작업에서의 패턴 고착 방지
