## 전제: LLM은 (대부분) 무상태

LLM은 무상태 함수. 가중치는 추론 시 고정되어 시간이 지나도 학습하지 않음. **모델이 코드베이스에 대해 아는 것은 당신이 넣은 토큰뿐.**

`CLAUDE.md`는 기본적으로 에이전트와의 **모든 단일 대화에 들어가는 유일한 파일**이다.

**세 가지 중요한 함의:**
1. 코딩 에이전트는 각 세션 시작 시 코드베이스에 대해 **아무것도 모름**
2. 코드베이스에 대해 중요한 것은 **매번** 에이전트에게 알려줘야 함
3. `CLAUDE.md`가 이를 위한 **선호되는 방법**

## CLAUDE.md는 Claude를 코드베이스에 온보딩한다

세 가지 차원으로 커버:
- **WHAT:** 기술, 스택, 프로젝트 구조. 코드베이스의 지도 제공. 모노레포에서 특히 중요 — 앱이 무엇이고, 공유 패키지가 무엇이고, 각각이 무엇을 위한 것인지
- **WHY:** 프로젝트의 **목적**과 리포지토리의 각 부분이 하는 일
- **HOW:** 프로젝트에서 어떻게 작업해야 하는지. 예: `bun`을 `node` 대신 사용하는지? 변경사항 검증 방법? 테스트, 타입체크, 컴파일 단계?

> "하지만 하는 방식이 중요하다! Claude가 가능하게 필요한 모든 명령을 CLAUDE.md에 채우려 하지 마라 — 차선의 결과를 얻을 것이다."

## Claude가 CLAUDE.md를 종종 무시하는 이유

Claude Code의 시스템 리마인더 분석:
```
IMPORTANT: this context may or may not be relevant to your tasks.
You should not respond to this context unless it is highly relevant to your task.
```

Claude는 CLAUDE.md 내용이 현재 작업에 관련이 없다고 판단하면 **무시**한다. **보편적으로 적용 가능하지 않은** 지시가 파일에 많을수록 Claude가 지시를 무시할 확률이 높아짐.

**왜 Anthropic이 이렇게 했는가?** 대부분의 CLAUDE.md 파일이 광범위하게 적용되지 않는 지시들 — 마음에 들지 않는 행동에 대한 "핫픽스"를 계속 추가 — 을 포함하고 있어, 나쁜 지시를 무시하도록 하면 harness가 실제로 **더 나은 결과**를 생산한다고 판단한 것으로 추정.

## 좋은 CLAUDE.md 작성 원칙

### 원칙 1: 지시는 적을수록 좋다 (Less Instructions is More)

연구 결과:
- **프론티어 thinking LLM은 약 150~200개의 지시를 합리적 일관성으로 따를 수 있음**
- 작은 모델은 **지수적 감쇠** (지시 수 증가에 따라 급격히 악화)
- 큰 모델은 **선형 감쇠** (더 완만한 악화)
- **지시 수가 증가하면 지시 따르기 품질이 균일하게 감소** — 새 지시만 무시하는 것이 아니라 **모든 지시를 균일하게 무시**하기 시작

**Claude Code의 시스템 프롬프트에 이미 ~50개의 개별 지시가 포함.** 모델에 따라 이것이 이미 에이전트가 안정적으로 따를 수 있는 지시의 **거의 1/3**. Rules, plugins, skills, 사용자 메시지가 추가되기 전에 이미.

→ **CLAUDE.md는 가능한 한 적은 지시를 포함해야 한다. 이상적으로는 보편적으로 적용 가능한 것만.**

### 원칙 2: 간결하고 보편적으로 적용 가능하게

`CLAUDE.md`는 **모든 단일 세션에 들어가므로**, 내용이 가능한 한 보편적으로 적용 가능해야 한다.

**피해야 할 예:** "새 데이터베이스 스키마를 구성하는 방법" — 관련 없는 작업을 할 때 모델을 **산만하게 하고 방해**

길이 권장: 공식 추천은 없지만 일반적 합의는 **300줄 미만**. 짧을수록 더 좋다.

> **HumanLayer의 루트 CLAUDE.md 파일은 60줄 미만.**

### 원칙 3: 점진적 공개 (Progressive Disclosure)

모든 지시를 CLAUDE.md에 넣는 대신, 작업/프로젝트별 지시를 **자기 설명적 이름의 별도 마크다운 파일**에 보관:

```
agent_docs/
├── building_the_project.md
├── running_tests.md
├── code_conventions.md
├── service_architecture.md
├── database_schema.md
└── service_communication_patterns.md
```

CLAUDE.md에는 이 파일들의 **목록과 간략한 설명**을 포함하고, Claude에게 관련 있는 것을 결정하여 작업 시작 전 읽도록 지시.

> **"복사본보다 포인터를 선호하라."** 이 파일들에 코드 스니펫을 포함하지 마라 — 빠르게 구식이 된다. 대신 `file:line` 참조로 Claude를 권위 있는 컨텍스트로 안내.

### 원칙 4: Claude는 비싼 린터가 아니다

> "절대 린터의 일을 하라고 LLM을 보내지 마라."

코드 스타일 가이드라인을 CLAUDE.md에 넣으면:
- 많은 지시와 대부분 무관한 코드 스니펫이 컨텍스트 윈도우에 추가
- LLM의 성능과 지시 따르기가 **저하**
- 컨텍스트 윈도우를 **소모**

**LLM은 in-context 학습자!** 코드가 특정 스타일 가이드라인이나 패턴을 따르면, 코드베이스 검색 몇 번으로 **에이전트가 기존 코드 패턴과 규칙을 따르는 경향**이 있다 — 알려주지 않아도.

**대안:**
- 자동 수정 가능한 린터 사용 (예: Biome)
- Claude Code `Stop` hook으로 포매터/린터를 실행하고 에러를 Claude에 제시
- Slash command로 코드 가이드라인을 포함하되, 변경사항에 대해서만 적용 → 구현과 포맷팅을 **분리하면 둘 다 더 나은 결과**

### 원칙 5: 자동 생성하지 마라

`/init`이나 자동 생성 사용을 **비추천**.

> "CLAUDE.md는 harness의 가장 높은 레버리지 포인트 중 하나다. 코드의 나쁜 한 줄은 나쁜 한 줄이다. 구현 계획의 나쁜 한 줄은 많은 나쁜 코드를 만들 수 있다. 하지만 CLAUDE.md는 워크플로우의 모든 단계와 모든 산출물에 영향을 미친다."

→ 모든 한 줄을 **신중하게 생각하여** 넣어야 한다.

## 결론 요약

1. CLAUDE.md는 코드베이스에 Claude를 온보딩하는 것. **WHY, WHAT, HOW**를 정의
2. **지시는 적을수록 좋다.** 필요한 지시를 빠뜨리지 않되, 합리적으로 가능한 **최소한의 지시**
3. **간결하고 보편적으로 적용 가능하게** 유지
4. **점진적 공개** — 모든 정보를 넣지 말고, 중요한 정보를 **찾는 방법**을 알려줘 필요할 때만 사용
5. **Claude는 린터가 아니다.** 린터와 포매터를 사용하고, Hooks와 Slash Commands를 필요에 따라
6. **CLAUDE.md는 harness의 가장 높은 레버리지 포인트.** 자동 생성을 피하고 내용을 **신중하게** 작성

## Soo에게 의미 있는 이유

이 글은 **CLAUDE.md/AGENTS.md 작성의 결정판**이다. 우리의 SPEC.md, ARCHITECTURE.md, PRINCIPLES.md 패턴은 이 원칙의 점진적 공개(progressive disclosure)를 직접 적용한 것.

**AInD 컨설팅 핵심:**
1. **"지시 예산"** — Claude Code 시스템 프롬프트에 이미 ~50개 지시, 남은 예산은 100~150개뿐
2. **60줄 미만** — "많이 넣을수록 좋다"는 상식에 반하는 구체적 증거
3. **"Claude가 CLAUDE.md를 무시하는 이유"** — 관련 없는 지시가 많으면 모두 무시. 고객이 "왜 규칙을 안 따르냐"고 물을 때의 답
4. **점진적 공개** — 대규모 프로젝트의 에이전트 도입 전략의 핵심 패턴
5. **"린터의 일을 LLM에 시키지 마라"** — 비용과 효과 양쪽에서 중요한 원칙
