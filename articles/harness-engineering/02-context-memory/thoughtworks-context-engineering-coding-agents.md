## 배경

코딩 에이전트의 컨텍스트를 구성하고 풍부하게 하는 옵션이 지난 몇 달간 **폭발적으로 증가**했다. Claude Code가 이 공간의 혁신을 주도하고 있고, 다른 코딩 어시스턴트들이 빠르게 따라오고 있다. **강력한 context engineering이 이 도구들의 개발자 경험에서 거대한 부분**이 되고 있다.

> Context engineering의 단순한 정의 (Bharani Subramaniam): "더 나은 결과를 얻기 위해 모델이 보는 것을 큐레이션하는 것."

## 코딩 에이전트에서의 컨텍스트란 무엇인가

### 재사용 가능한 프롬프트 — 두 가지 의도

모든 형태의 AI 코딩 context engineering은 궁극적으로 **프롬프트가 담긴 마크다운 파일들**이다. 두 가지 주요 카테고리:

| 의도 | 정의 | 예시 |
|------|------|------|
| **Instructions** | 에이전트에게 무언가를 하라고 지시 | "E2E 테스트를 다음과 같이 작성하라: ..." |
| **Guidance** (규칙, 가드레일) | 에이전트가 따라야 할 일반적 규칙 | "항상 서로 독립적인 테스트를 작성하라" |

이 두 카테고리는 자주 혼합되지만, 구분하는 것이 유용하다.

### 컨텍스트 인터페이스 — 에이전트가 더 많은 컨텍스트를 가져오는 방법

- **Tools:** 내장 기능 (bash 명령, 파일 검색 등)
- **MCP Servers:** 에이전트에게 데이터 소스와 행동에 대한 접근을 제공하는 프로그램/스크립트
- **Skills:** 가장 새로운 진입자. LLM이 작업에 관련이 있다고 판단할 때 **온디맨드로 로드**할 수 있는 추가 리소스, 지시, 문서, 스크립트 등의 설명

> "이것들을 많이 구성할수록 컨텍스트에서 차지하는 공간이 커진다. 특정 작업에 어떤 컨텍스트 인터페이스가 필요한지 **전략적으로 생각**하는 것이 현명하다."

**가장 기본이자 강력한 컨텍스트 인터페이스:** 파일 읽기와 검색. 기존 코드가 컨텍스트로서 **얼마나 잘 역할하는지** 되돌아볼 가치가 있다 — 사실상 **AI 친화적 코드베이스 설계**.

## 세 가지 핵심 차원

### 차원 1: 누가 컨텍스트 로딩을 결정하는가?

| 결정자 | 특징 | 예시 |
|--------|------|------|
| **LLM** | 비지도 실행의 전제조건. 하지만 LLM이 실제로 로드할지 **불확실성** 존재 | Skills |
| **Human** | 통제력 있지만 자동화 수준 감소 | Slash commands |
| **Agent 소프트웨어** | 결정론적 시점에 트리거 | Claude Code hooks |

### 차원 2: 컨텍스트 크기 관리

> "컨텍스트 윈도우가 기술적으로 매우 커졌다고 해서, 정보를 무차별적으로 넣는 것이 좋은 생각은 아니다."

- 에이전트의 효과는 **너무 많은 컨텍스트를 받으면 저하**
- 비용 문제도 물론 존재
- **규칙 파일을 점진적으로** 구축하는 것을 추천 — 처음부터 너무 많이 넣지 말 것
- 모델이 상당히 강력해졌으므로, 6개월 전에 넣어야 했던 것이 **이제는 불필요할 수 있음**
- 도구의 **투명성** 중요: 컨텍스트가 얼마나 차 있는지, 무엇이 얼마나 공간을 차지하는지

## Claude Code의 컨텍스트 설정 기능 — 전체 분류

### 1. CLAUDE.md
- **무엇:** Guidance
- **로딩 결정:** Claude Code가 세션 시작 시 항상 사용
- **용도:** 전체 프로젝트에 적용되는 가장 자주 반복되는 일반적 규칙
- **예시:** "yarn을 사용, npm 아님", "가상 환경을 활성화한 후 실행", "리팩토링 시 하위 호환성 불필요"
- **다른 도구:** 거의 모든 코딩 어시스턴트가 이 기능 보유. `AGENTS.md`로 표준화 시도 중

### 2. Rules (파일 경로 기반)
- **무엇:** Guidance (모듈화, 범위 한정)
- **로딩 결정:** Claude Code가 해당 경로의 파일이 로드될 때
- **용도:** CLAUDE.md를 모듈화하고 크기를 제한. 파일에 범위 지정 가능 (e.g., `*.ts` → TypeScript 파일에만)
- **예시:** "bash 스크립트 작성 시 변수를 `${var}`로 참조, `$var` 아님" → paths: `**/*.sh`
- **다른 도구:** GH Copilot, Cursor 등도 지원

### 3. Skills
- **무엇:** Guidance + Instructions + 문서 + 스크립트 등 무엇이든
- **로딩 결정:** LLM (스킬 설명 기반) 또는 Human
- **용도:** 작업에 관련될 때만 **"lazy load"**하고 싶은 가이던스나 지시. 스킬 폴더에 추가 리소스와 스크립트를 넣고 `SKILL.md`에서 참조
- **예시:** JIRA 접근 (CLI 사용법 설명), "React 컴포넌트 규칙", "XYZ API 연동법"
- **다른 도구:** Cursor도 Claude Code 스타일 Skills로 전환 중

### 4. Subagents
- **무엇:** Instructions + 모델/도구 설정. 자체 컨텍스트 윈도우에서 실행, 병렬화 가능
- **로딩 결정:** LLM 또는 Human
- **용도:**
  - 자체 컨텍스트에서 실행할 가치가 있는 대규모 공통 작업
  - 기본 모델과 다른 모델을 사용하고 싶은 작업
  - 기본 컨텍스트에 항상 제공하고 싶지 않은 특정 도구/MCP가 필요한 작업
- **예시:** E2E 테스트 생성, 다른 모델로의 코드 리뷰("세컨드 오피니언"), claude-flow나 Gas Town 같은 스웜 실험의 기반
- **다른 도구:** Roo Code(오래전부터 "modes"로), Cursor(최근), GH Copilot(인간 트리거만)

### 5. MCP Servers
- **무엇:** 에이전트에게 API나 로컬 도구 접근 제공
- **로딩 결정:** LLM
- **참고:** 일부 MCP 서버 기능을 **스킬과 CLI 사용법으로 대체**하는 트렌드
- **예시:** JIRA API, Playwright 브라우저, 로컬 지식 베이스

### 6. Hooks
- **무엇:** 스크립트 (결정론적 실행)
- **로딩 결정:** Claude Code 생명주기 이벤트
- **용도:** 파일 편집, 명령 실행, MCP 호출 **매번** 결정론적으로 실행하고 싶을 때
- **예시:** 커스텀 알림, JS 파일 편집 시 prettier 자동 실행, 모든 실행 명령 로깅
- **다른 도구:** Hooks는 아직 드문 기능. Cursor가 최근 지원 시작

### 7. Plugins
- **무엇:** 위의 것들을 배포하는 방법
- **예시:** 조직 내 팀들에게 공통 명령, 스킬, 훅 세트를 배포

## 컨텍스트 설정 공유의 과제

좋은 설정을 만드는 데 시간이 걸린다 — 설정이 잘 작동하는지 판단하려면 **한동안 사용해봐야** 한다. Context engineering에는 유닛 테스트가 없다.

**공유의 어려움:**
- 공유자와 수신자의 **컨텍스트가 최대한 유사**해야 함 — 팀 내에서가 인터넷 낯선 사람 사이보다 훨씬 효과적
- 불필요하게 복사/붙여넣기된 지시로 **과도 엔지니어링하는 경향** — 반복적으로 구축하는 것이 최선
- 다른 경험 수준은 **다른 규칙/지시**가 필요할 수 있음
- 낯선 사람에게서 많이 복사하면 **지시 반복이나 모순**, 또는 에이전트가 *당신의* 지시를 따르는 것인데 에이전트를 비난하는 상황

## 경고: 통제의 환상

> "이름에도 불구하고, 이것은 진정한 의미의 engineering이 아니다... 에이전트가 이 모든 지시와 가이던스를 받으면, 실행은 여전히 **LLM이 얼마나 잘 해석하느냐**에 달려있다!"

Context engineering은 코딩 에이전트를 더 효과적으로 만들고 유용한 결과의 확률을 상당히 높일 수 있다. 하지만 때때로 사람들이 "X를 **보장**한다", "환각을 **방지**한다"고 말한다. **LLM이 관여하는 한, 우리는 아무것도 *확실*할 수 없다. 여전히 확률로 사고**하고 작업에 적절한 수준의 인간 감독을 선택해야 한다.

## Soo에게 의미 있는 이유

이 글은 **코딩 에이전트의 컨텍스트 설정 기능에 대한 가장 체계적인 분류**다. Claude Code의 7가지 기능(CLAUDE.md, Rules, Skills, Subagents, MCP, Hooks, Plugins)을 **누가 결정하는가, 언제 사용하는가** 차원으로 정리한다.

**AInD 컨설팅 핵심:**
1. **7가지 기능의 분류표** — 고객에게 "어떤 기능을 언제 사용해야 하는가" 교육 시 직접 사용 가능
2. **Instructions vs Guidance 구분** — "하라"와 "따르라"의 차이를 인식하는 것이 효과적 컨텍스트의 시작
3. **"통제의 환상" 경고** — 고객에게 현실적 기대치를 설정할 때 중요한 메시지
4. **컨텍스트 공유의 과제** — 팀 내 표준화 시 주의사항
5. **Skills가 Slash commands와 Rules를 흡수하는 트렌드** — 기능 수렴의 방향
