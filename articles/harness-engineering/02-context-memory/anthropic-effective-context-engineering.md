## Context Engineering의 정의

프롬프트 엔지니어링이 "적절한 단어와 구문 찾기"에 집중했다면, **context engineering은 "어떤 컨텍스트 구성이 모델의 원하는 행동을 생성할 가능성이 가장 높은가?"**라는 더 넓은 질문에 답하는 것이다.

- **Context** = LLM에서 샘플링할 때 포함되는 토큰 집합
- **Engineering** = 원하는 결과를 일관되게 달성하기 위해 LLM의 내재적 제약에 대해 그 토큰의 유용성을 최적화하는 것

> "LLM을 효과적으로 다루려면 종종 *컨텍스트로 사고(thinking in context)*해야 한다 — 주어진 시점에 LLM에 사용 가능한 **전체 상태**와 그 상태가 산출할 수 있는 잠재적 행동을 고려하는 것."

## 왜 Context Engineering이 중요한가: Context Rot

LLM은 인간처럼 특정 지점에서 **초점을 잃거나 혼란**을 경험한다. "Needle-in-a-haystack" 스타일 벤치마킹 연구가 **Context Rot** 개념을 발견: 컨텍스트 윈도우의 토큰 수가 증가할수록, 그 컨텍스트에서 정보를 정확히 회상하는 모델의 능력이 **감소**한다.

**기술적 원인:**
- Transformer 아키텍처에서 모든 토큰이 모든 다른 토큰에 attend → n개 토큰에 대해 **n² 쌍별 관계**
- 컨텍스트 길이가 증가하면 이 쌍별 관계 포착 능력이 분산
- 모델이 더 짧은 시퀀스가 일반적인 훈련 데이터 분포에서 주의 패턴을 개발 → 긴 컨텍스트 전반의 의존성을 위한 특화 파라미터가 적음
- Position encoding interpolation 기법으로 더 긴 시퀀스를 처리할 수 있지만, 토큰 위치 이해에서 약간의 저하
- **절벽(hard cliff)이 아닌 성능 경사(gradient)**: 모델이 긴 컨텍스트에서도 매우 유능하지만, 짧은 컨텍스트에서의 성능 대비 정보 검색과 장거리 추론에서 **정밀도가 감소**

따라서 컨텍스트는 **체감수익이 감소하는 유한한 자원**으로 취급해야 한다. 인간이 제한된 작업 기억 용량을 가진 것처럼, LLM은 큰 볼륨의 컨텍스트를 파싱할 때 끌어쓰는 **"주의 예산(attention budget)"**을 가지고 있다.

## 효과적인 컨텍스트의 해부학

> **좋은 context engineering = 원하는 결과의 가능성을 최대화하는 가장 작은 고신호 토큰 집합을 찾는 것**

### 시스템 프롬프트: 적절한 고도(Right Altitude)

두 가지 흔한 실패 모드 사이의 골디락스 존:

| 극단 | 문제 |
|------|------|
| **과도한 하드코딩** | 복잡하고 취약한 if-else 로직. 취약성 증가, 유지보수 복잡성 증가 |
| **과도한 일반화** | 모호하고 고수준의 가이던스. 구체적 신호 부재, 공유 컨텍스트를 잘못 가정 |
| **적절한 고도** ✅ | 행동을 효과적으로 안내할 만큼 구체적이면서, 강한 휴리스틱을 제공할 만큼 유연 |

**최선의 모델로 최소한의 프롬프트를 테스트**하는 것에서 시작하여, 초기 테스트에서 발견된 실패 모드에 기반하여 명확한 지시와 예시를 추가.

### 도구: 효율적인 계약

도구는 에이전트와 정보/행동 공간 사이의 **계약**을 정의. 가장 흔한 실패 모드: **기능이 너무 많거나 모호한 결정 지점을 만드는 비대한 도구 세트**.

> "인간 엔지니어가 주어진 상황에서 어떤 도구를 사용해야 하는지 확정적으로 말할 수 없다면, AI 에이전트가 더 잘할 것이라 기대할 수 없다."

### 예시(Few-shot): 에지 케이스 목록이 아닌 표본 집합

팀들이 종종 LLM이 따라야 할 모든 가능한 규칙을 프롬프트에 **에지 케이스 세탁 목록**으로 넣으려 한다. **이것을 권장하지 않는다.**

대신 예상되는 에이전트 행동을 효과적으로 보여주는 **다양하고 정규적인 예시 집합**을 큐레이션할 것을 권장.

> "LLM에게 예시는 천 마디 말의 가치가 있는 '그림'이다."

## 컨텍스트 검색과 에이전트 검색: Just-In-Time 접근

### 전통적 접근 vs 에이전트 접근

- **전통적:** 임베딩 기반 사전 추론 시간 검색 (모든 관련 데이터를 미리 처리)
- **에이전트:** **"Just-In-Time" 전략** — 경량 식별자(파일 경로, 저장된 쿼리, 웹 링크)를 유지하고, 런타임에 도구를 사용하여 동적으로 데이터를 로드

**Claude Code의 예시:** 대규모 데이터베이스에 대한 복잡한 데이터 분석을 수행할 때, 모델이 타겟된 쿼리를 작성하고, 결과를 저장하고, `head`와 `tail` 같은 Bash 명령을 활용하여 전체 데이터 객체를 컨텍스트에 로드하지 않고도 대량의 데이터를 분석.

이 접근은 **인간의 인지를 반영**: 우리는 전체 정보 말뭉치를 암기하지 않고, 파일 시스템, 받은편지함, 북마크 같은 **외부 조직/인덱싱 시스템**으로 관련 정보를 온디맨드로 검색한다.

### 점진적 공개 (Progressive Disclosure)

에이전트가 탐색을 통해 관련 컨텍스트를 **점진적으로 발견**:
- 파일 크기 → 복잡성 암시
- 네이밍 규칙 → 목적 암시
- 타임스탬프 → 관련성의 프록시
- 에이전트가 **필요한 것만** 작업 메모리에 유지하고, 메모 작성 전략으로 추가 지속성 활용

### 하이브리드 전략

가장 효과적인 에이전트는 **하이브리드 전략**을 사용:
- 일부 데이터를 속도를 위해 **미리 검색**
- 추가 자율 탐색은 **재량에 따라**

**Claude Code의 하이브리드:** CLAUDE.md 파일은 컨텍스트에 **즉시 투입**되고, glob과 grep 같은 프리미티브는 환경을 탐색하고 파일을 just-in-time으로 검색할 수 있게 하여, 낡은 인덱싱과 복잡한 구문 트리 문제를 효과적으로 우회.

## 장기 과제를 위한 Context Engineering

장기 과제는 에이전트가 토큰 수가 컨텍스트 윈도우를 초과하는 행동 시퀀스에 걸쳐 **일관성, 컨텍스트, 목표 지향 행동**을 유지해야 한다. 수십 분에서 수 시간의 연속 작업을 위한 세 가지 기법:

### 1. Compaction (압축)

컨텍스트 윈도우 한계에 가까운 대화를 요약하고, **요약으로 새 컨텍스트 윈도우를 시작**.

**Claude Code의 구현:** 메시지 히스토리를 모델에 전달하여 가장 중요한 세부사항을 요약/압축. 아키텍처 결정, 미해결 버그, 구현 세부사항을 보존하면서 중복 도구 출력이나 메시지를 폐기. 에이전트는 이 압축된 컨텍스트 + **최근 접근한 5개 파일**로 계속.

**Compaction의 기술:**
- 회상(recall)을 최대화하는 것에서 시작 — 트레이스에서 모든 관련 정보를 포착
- 정밀도(precision)를 개선하기 위해 반복 — 불필요한 콘텐츠 제거
- 가장 안전한 경량 형태: **도구 호출 결과 지우기** — 메시지 히스토리 깊은 곳의 도구 호출 결과를 다시 볼 이유가 있는가?

### 2. 구조화된 메모 작성 (Structured Note-Taking)

에이전트가 컨텍스트 윈도우 외부 메모리에 정기적으로 **메모를 지속**. 이 메모가 나중에 컨텍스트 윈도우로 다시 가져와짐.

**Claude가 포켓몬을 플레이한 사례:** 에이전트가 수천 게임 스텝에 걸쳐 정확한 집계를 유지 — "지난 1,234 스텝 동안 루트 1에서 포켓몬을 훈련. 피카츄가 목표 10 레벨 중 8 레벨 상승." 메모리 구조에 대한 프롬프팅 없이 탐험 지역 맵, 주요 성취, 전투 전략 메모를 자체 개발. **컨텍스트 리셋 후 자신의 메모를 읽고** 수 시간의 훈련이나 던전 탐험을 계속.

### 3. 서브에이전트 아키텍처

하나의 에이전트가 전체 프로젝트의 상태를 유지하려 하지 않고, **특화된 서브에이전트가 깨끗한 컨텍스트 윈도우로 집중된 작업을 처리**. 메인 에이전트는 고수준 계획으로 조정하고 서브에이전트는 깊은 기술 작업이나 정보 검색을 수행. 각 서브에이전트는 수만 토큰을 광범위하게 탐색할 수 있지만, **압축된 요약(종종 1,000~2,000 토큰)만 반환**.

**관심사의 명확한 분리:** 상세한 검색 컨텍스트는 서브에이전트 안에 격리, 리드 에이전트는 결과 종합과 분석에 집중.

### 접근법 선택 가이드

| 기법 | 적합한 상황 |
|------|-----------|
| **Compaction** | 광범위한 왕복이 필요한 작업의 대화 흐름 유지 |
| **메모 작성** | 명확한 마일스톤이 있는 반복적 개발 |
| **멀티에이전트** | 병렬 탐색이 효과적인 복잡한 연구/분석 |

## 결론

> "모델이 더 capable해져도, 확장된 상호작용에 걸쳐 일관성을 유지하는 과제는 더 효과적인 에이전트를 구축하는 데 중심적인 것으로 남을 것이다."

> "모델이 더 스마트해지면서 덜 처방적인 엔지니어링이 필요해지고, 에이전트가 더 많은 자율성으로 운영될 수 있게 된다. 하지만 능력이 확장되어도, **컨텍스트를 귀중하고 유한한 자원으로 취급하는 것**은 신뢰할 수 있고 효과적인 에이전트를 구축하는 데 중심적으로 남을 것이다."

## Soo에게 의미 있는 이유

이 글은 **context engineering의 결정판**이다. "prompt engineering → context engineering"이라는 패러다임 전환을 선언한 글.

**AInD 컨설팅 핵심:**
1. **Context Rot의 기술적 설명** — n² 쌍별 관계, 훈련 데이터 분포 → "왜 컨텍스트를 무작정 크게 하면 안 되는가"의 근거
2. **"적절한 고도" 프레임워크** — 프롬프트가 너무 구체적이거나 너무 일반적인 두 극단 사이의 균형
3. **Just-In-Time 검색** — 사전 로딩이 아닌 필요 시점에 동적 로드. Claude Code의 실제 구현 방식
4. **Compaction + 메모 + 서브에이전트** — 장기 과제의 3가지 기법, 상황별 선택 가이드
5. **"가능한 가장 작은 고신호 토큰 집합"** — 모든 context engineering의 제1원칙
