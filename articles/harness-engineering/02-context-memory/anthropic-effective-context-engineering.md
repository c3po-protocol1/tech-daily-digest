## 배경

컨텍스트 윈도우를 **"쓰레기 투기장"이 아닌 "작업 메모리 예산"**으로 관리하라는 Anthropic의 핵심 가이드. 컨텍스트 엔지니어링이라는 용어를 업계에 확립시킨 글.

## 핵심 원칙

### 컨텍스트는 유한한 자원이다

200K 토큰 윈도우가 크다고 느껴질 수 있지만, **신호 대 잡음 비율(SNR)**이 중요하다. 관련 없는 정보로 윈도우를 채우면:
- 모델이 중요한 정보를 놓침 ("lost in the middle" 현상)
- 지시 따르기(instruction following) 품질 저하
- 불필요한 토큰 비용 발생

### "무엇을 넣지 않을까"가 "무엇을 넣을까"만큼 중요

정보 과부하는 정보 부족만큼 해로움:
- 10,000줄의 로그를 통째로 넣으면 → 핵심 에러를 놓침
- 모든 프로젝트 파일을 넣으면 → 현재 작업과 무관한 파일이 혼란 유발
- **선택적 포함 + 요약 + 필터링**이 핵심

## 실용적 기법

### 1. 프롬프트 계층화
컨텍스트를 세 계층으로 관리:

| 계층 | 특성 | 예시 |
|------|------|------|
| **시스템 프롬프트** | 영구, 변경 드뭄 | 역할, 규칙, 아키텍처 |
| **대화 컨텍스트** | 세션 내 변동 | 이전 대화, 에이전트 결정 |
| **도구 결과** | 일시적, 매 턴 변동 | 파일 내용, 명령 출력 |

시스템 프롬프트는 **안정적으로** 유지하여 KV-cache 효율 극대화.

### 2. 선택적 포함
- 작업 관련 파일만 컨텍스트에 포함
- `tree` 출력으로 전체 구조를 보여주되, **파일 내용은 필요할 때만**
- 에이전트에게 "먼저 구조를 확인하고, 필요한 파일만 읽어라"고 지시

### 3. 도구 출력 요약
긴 출력은 그대로 넣지 말고:
- 에러 로그 → 핵심 에러 메시지만 추출
- 테스트 결과 → 실패한 테스트만 포함
- API 응답 → 관련 필드만 추출

### 4. 컨텍스트 청소
더 이상 관련 없는 정보는 적극적으로 제거:
- 이전 작업의 중간 결과 → 최종 결과만 유지
- 디버깅 로그 → 문제 해결 후 제거
- 이전 에러 메시지 → 수정 후 제거

## "컨텍스트 = 작업 메모리" 멘탈 모델

인간의 작업 메모리처럼 생각하라:
- 동시에 7±2개의 정보만 활성 유지
- 새 정보가 들어오면 덜 중요한 정보는 배경으로
- **가장 관련 있는 정보가 가장 잘 보이는 위치에**
- 시스템 프롬프트 직후와 대화 최하단이 모델의 주의가 가장 높은 위치

## Soo에게 의미 있는 이유

컨텍스트 엔지니어링은 AI 에이전트 활용의 **가장 기본이자 가장 중요한 기술**이다. AInD 컨설팅에서 "왜 같은 모델인데 결과가 다른가"를 설명할 때, 컨텍스트 관리의 차이가 핵심 답변이 된다. 이 글의 SNR 프레임워크와 4가지 기법은 실무 교육에 바로 사용 가능.
