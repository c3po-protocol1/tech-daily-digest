# 2. Context, Memory & Working State — 컨텍스트 엔지니어링

> 에이전트의 컨텍스트 윈도우를 효과적으로 관리하는 기법들.

---

## 2.1 Anthropic — Effective Context Engineering for AI Agents

**출처:** [anthropic.com](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents)

### 핵심 내용

컨텍스트 윈도우를 **"쓰레기 투기장"이 아닌 "작업 메모리 예산"**으로 관리하라는 Anthropic의 가이드.

**핵심 원칙:**
- 컨텍스트 윈도우는 유한한 자원 — 무엇을 넣을지 **의도적으로 설계**해야 함
- 모든 정보를 넣는 것은 정보를 안 넣는 것만큼 해로움 (노이즈가 신호를 묻음)
- **필요한 정보를, 필요한 시점에, 적절한 형태로** 제공

**실용적 기법:**
- **프롬프트 계층화** — 시스템 프롬프트(영구) + 대화 컨텍스트(변동) + 도구 결과(일시)
- **선택적 포함** — 모든 파일이 아닌 관련 파일만 컨텍스트에 포함
- **요약과 압축** — 긴 도구 출력은 요약하여 포함
- **컨텍스트 청소** — 더 이상 관련 없는 정보는 적극적으로 제거

### 핵심 교훈
> "컨텍스트 윈도우가 클수록 좋은 것이 아니다. 신호 대 잡음 비율(SNR)이 중요하다."

## 2.2 Manus — Context Engineering: Lessons from Building Manus

**출처:** [manus.im](https://manus.im/blog/Context-Engineering-for-AI-Agents-Lessons-from-Building-Manus)

### 핵심 내용

Manus(AI 에이전트 제품) 개발 경험에서 얻은 컨텍스트 엔지니어링 실전 교훈.

**KV-Cache 지역성:**
- LLM의 KV-cache는 프롬프트 접두사가 같으면 캐시 히트
- 시스템 프롬프트와 도구 정의를 **안정적으로 유지**하면 추론 비용 절감
- 매번 도구 목록을 다르게 넣으면 캐시 미스 → 비용 + 지연 증가

**도구 마스킹(Tool Masking):**
- 매 턴마다 에이전트가 사용할 수 있는 도구를 **동적으로 제한**
- 모든 도구를 항상 노출하면 혼란 → 현재 상태에 적합한 도구만 보이게
- KV-cache를 깨지 않으면서 도구를 마스킹하는 기법

**파일시스템 메모리:**
- 컨텍스트 윈도우 외부에 상태를 저장하는 수단으로 **파일시스템** 활용
- 에이전트가 작업 중간 결과를 파일에 저장하고 나중에 다시 읽음
- 컨텍스트 윈도우의 한계를 우회하는 실용적 패턴

**유용한 실패를 컨텍스트에 유지:**
- 에이전트의 실패 시도도 가치 있는 정보 — 같은 실수 반복 방지
- 성공만 남기고 실패를 제거하면 에이전트가 같은 시도를 반복

### 핵심 교훈
> "컨텍스트 엔지니어링은 단순히 프롬프트를 잘 쓰는 것이 아니라, 캐시, 메모리, 도구 가시성까지 포함하는 시스템 설계다."

## 2.3 Thoughtworks — Context Engineering for Coding Agents

**출처:** [martinfowler.com](https://martinfowler.com/articles/exploring-gen-ai/context-engineering-coding-agents.html)

### 핵심 내용

코딩 에이전트가 **기반을 유지(grounded)**하면서 생산적으로 작업할 수 있도록 환경을 설계하는 방법.

**"기반 유지(Grounding)"의 의미:**
- 에이전트가 프로젝트의 실제 상태를 정확히 이해하고 있는 것
- 파일 구조, 기존 패턴, 의존성, 테스트 상태를 파악한 상태에서 코딩

**실용적 접근:**
- **리포 구조 문서화** — AGENTS.md, ARCHITECTURE.md 등으로 프로젝트 맥락 제공
- **관련 파일 자동 발견** — 에이전트가 작업 시작 전 관련 코드를 자동으로 탐색
- **테스트 피드백 루프** — 코드 작성 → 테스트 → 결과를 컨텍스트에 포함 → 수정
- **점진적 컨텍스트 구축** — 한 번에 모든 것을 넣지 않고, 작업 진행에 따라 확장

### 핵심 교훈
> "코딩 에이전트의 품질은 모델 능력보다 얼마나 잘 기반이 잡혀있느냐에 더 많이 좌우된다."

## 2.4 HumanLayer — Advanced Context Engineering for Coding Agents

**출처:** [humanlayer.dev](https://www.humanlayer.dev/blog/advanced-context-engineering)

### 핵심 내용

컨텍스트 드리프트(drift)를 줄이고, 코딩 세션을 쉽게 재개할 수 있는 고급 패턴.

**컨텍스트 드리프트:**
- 장기 세션에서 초기 지시가 점점 희미해지는 현상
- 에이전트가 원래 목표에서 서서히 벗어남
- **주기적 리마인더, 구조화된 체크포인트**로 완화

**세션 재개 패턴:**
- 세션 중단 시 **상태 스냅샷**을 파일로 저장
- 새 세션 시작 시 스냅샷을 컨텍스트에 주입
- "지금까지 X를 했고, Y가 남았다"는 명확한 핸드오프

**구조화된 메모리:**
- 자유 형식 대화보다 **구조화된 형식**(JSON, 체크리스트, 테이블)이 세션 간 전달에 유리
- 에이전트가 파싱하기 쉬운 형태로 상태를 유지

### 핵심 교훈
> "장기 코딩 세션의 적은 컨텍스트 한계가 아니라 컨텍스트 드리프트다. 주기적 앵커링이 해결책이다."

## 2.5 HumanLayer — Context-Efficient Backpressure for Coding Agents

**출처:** [humanlayer.dev](https://www.humanlayer.dev/blog/context-efficient-backpressure)

### 핵심 내용

에이전트가 **노이즈나 저가치 작업에 컨텍스트를 낭비하지 않도록** 하는 백프레셔 패턴.

**문제:**
- 에이전트가 불필요한 파일을 읽거나, 관련 없는 에러를 디버깅하거나, 이미 해결된 문제를 반복 탐색
- 이 모든 것이 컨텍스트 예산을 소모

**백프레셔 기법:**
- **도구 출력 필터링** — 도구 결과에서 관련 부분만 컨텍스트에 포함
- **실패 예산** — 같은 접근을 N회 이상 실패하면 다른 전략 강제
- **컨텍스트 비용 인식** — 에이전트에게 "컨텍스트가 제한적이니 효율적으로 사용하라"고 명시
- **불필요한 탐색 차단** — 작업과 무관한 디렉토리/파일 접근 제한

### 핵심 교훈
> "컨텍스트는 예산이다. 백프레셔 없이 에이전트를 돌리면 예산은 노이즈에 소모된다."

## 2.6 HumanLayer — Writing a Good CLAUDE.md

**출처:** [humanlayer.dev](https://www.humanlayer.dev/blog/writing-a-good-claude-md)

### 핵심 내용

에이전트가 반복적으로 따를 수 있는 **리포 로컬 지시서(CLAUDE.md)** 작성 가이드.

**좋은 CLAUDE.md의 특징:**
- **구체적** — "좋은 코드를 작성하라"가 아닌 "모든 함수에 JSDoc 주석을 포함하라"
- **검증 가능** — 에이전트가 따랐는지 자동으로 확인할 수 있는 규칙
- **안정적** — 매번 바뀌면 에이전트가 혼란. 프로젝트 전체에 걸쳐 일관되게 유지
- **계층적** — 프로젝트 전체 규칙 + 디렉토리별 규칙

**피해야 할 것:**
- 모순되는 지시
- 너무 많은 규칙 (에이전트가 모두 따르기 어려움)
- 암묵적 가정 (에이전트는 "당연한 것"을 모름)

**실용 팁:**
- 빌드 명령어, 테스트 명령어, 린트 규칙을 명시
- 프로젝트 구조와 핵심 파일의 역할 설명
- 금지 사항을 명확히 (e.g., "이 파일은 수정하지 마라")

### 핵심 교훈
> "CLAUDE.md는 에이전트의 온보딩 문서다. 새 팀원에게 설명하듯 구체적이고 검증 가능하게 작성하라."
