## 핵심 주장

> "지난 1년간 코딩 에이전트가 상상 가능한 모든 방식으로 실패하는 것을 지켜봤다: 지시를 무시하고, 위험한 명령을 무단 실행하고, 가장 단순한 작업에서 빙글빙글 돌았다."

매번 본능적 반응은 같았다:
- "더 좋은 모델이 필요해, GPT-6가 해결할 거야"
- "instruction-following만 개선되면 될 거야"
- "내가 쓰는 라이브러리가 훈련 데이터에 들어가면 될 거야"

하지만 수십 개의 프로젝트와 수백 번의 에이전트 세션을 거치며 **같은 결론에 반복적으로 도달**:

> **모델 문제가 아니다. 설정(configuration) 문제다.**

모델은 더 똑똑해지고 일부 실패 모드는 사라질 것이다. 하지만 더 똑똑해지면 더 크고 어려운 문제를 주게 되고, **예상치 못한 방식으로 계속 실패**할 것이다. 비결정적 시스템의 예상치 못한 실패 모드는 **근본적 문제**다.

그래서 `gpt-6.4-codex-ultrahigh_extended`를 기도하는 대신, **"오늘의 모델에서 최대한의 성능을 끌어내는 방법"**에 집중한다.

## Harness Engineering의 정의

```
coding agent = AI model(s) + harness
```

Skills, MCP 서버, 서브에이전트, 메모리, AGENTS.md — 이것들은 모두 기술적으로 별도 개념이지만, 모두 코딩 에이전트의 **설정 표면(configuration surface)**의 일부다. 이것을 **에이전트의 harness**라고 부른다.

Mitchell Hashimoto의 정의:
> "에이전트가 실수할 때마다, 에이전트가 그 실수를 **다시는 하지 않도록** 솔루션을 엔지니어링하는 데 시간을 투자하는 것."

### Harness Engineering ⊂ Context Engineering

HumanLayer는 harness engineering을 **context engineering의 하위 집합**으로 본다. Context engineering은 "prompt engineering"의 상위 개념으로, AI 에이전트의 신뢰성을 체계적으로 개선하는 다양한 기법의 총칭.

Harness engineering이 답하는 질문들:
- 코딩 에이전트에게 **새로운 능력**을 어떻게 부여하는가?
- 훈련 데이터에 없는 **코드베이스 지식**을 어떻게 가르치는가?
- 시스템 메시지의 `CRITICAL: always do XYZ` **이상의 결정론**을 어떻게 추가하는가?
- 에이전트의 행동을 우리의 **특정 코드베이스에 맞게** 어떻게 조정하는가?
- "매직 프롬프트" **이상으로** 작업 성공률을 어떻게 높이는가?

### 두 가지 추가 레버 (LangChain이 강조하지 않는 것)

1. **Hooks** — 자동화된 통합과 결정론적 제어 흐름
2. **Skills** — 지식의 점진적 공개 ("Instruction Modules")

### 서브에이전트: 가장 강력한 레버

> "복잡한 브라운필드 엔터프라이즈 규모 코드베이스에서 어려운 문제를 해결한 수개월 후, 서브에이전트가 특히 강력한 레버임을 발견했다."

서브에이전트가 중요한 이유: 많은 컨텍스트 윈도우에 걸친 어려운 문제를 해결할 때, 서브에이전트는 **"컨텍스트 방화벽"** 역할을 한다.

- 개별 작업이 **격리된 컨텍스트 윈도우**에서 실행
- 중간 노이즈가 **오케스트레이션을 담당하는 부모 스레드에 축적되지 않음**
- **훨씬 더 오래** 일관성을 유지할 수 있음

## CLAUDE.md & AGENTS.md

다른 harness 설정을 건드리기 전에 **CLAUDE.md/AGENTS.md를 먼저 커스터마이즈**하는 것이 보통 가치가 있다. 이 파일들은 harness가 에이전트의 시스템 프롬프트에 **결정론적으로 주입**하는 마크다운 파일이다.

### ETH Zurich 연구에 대한 반박

ETH Zurich가 138개의 agentfile을 테스트한 연구에서 대부분의 agentfile이 "무용지물이거나 더 나쁘다"는 결과가 나왔을 때, 많은 사람이 "CLAUDE.md는 시간 낭비"라고 했다.

**HumanLayer의 분석:** 연구 결과는 사실 HumanLayer가 원래 주장한 것을 확인해준다:

| 연구 발견 | HumanLayer의 원래 조언 |
|-----------|----------------------|
| LLM 생성 파일이 성능을 해침 + 20%+ 비용 증가 | "자동 생성을 피하라. 내용을 신중하게 작성하라" |
| 특정 도구 사용을 과도하게 유도하는 파일이 나쁜 결과 | "지시는 적을수록 좋다. 합리적으로 가능한 한 적은 지시를 포함하라" |
| 무관한 컨텍스트 포함 | "점진적 공개를 사용하라" |
| 인간 작성 파일도 조건부 규칙이 너무 많아 거의 도움 안 됨 | "CLAUDE.md 내용을 간결하고 보편적으로 유지하라" |

> "우리의 CLAUDE.md는 **60줄 미만**이다."

## MCP 서버: 도구를 위한 것

MCP 서버의 도구 설명이 **에이전트의 시스템 프롬프트에 주입**되므로:

**경고:** 신뢰하지 않는 MCP 서버에 절대 연결하지 마라. 프롬프트 인젝션의 위험한 벡터가 될 수 있다.

### "너무 많은 도구는 해로움"

너무 많은 MCP 도구를 연결하면:
- 컨텍스트 윈도우가 도구 설명으로 채워짐
- **"dumb zone"에 훨씬 빨리** 진입
- 모든 무관한 도구 설명이 에이전트가 혜택 없이 처리해야 하는 지시

Anthropic이 이 문제를 위해 **MCP 도구 검색의 실험적 지원**을 출시한 이유.

### 실전 교훈: CLI가 MCP보다 나을 때

> "MCP 서버가 훈련 데이터에 잘 표현된 CLI의 기능을 복제하면, 에이전트에게 그냥 CLI를 사용하도록 프롬프트하는 것이 더 좋다."

GitHub, Docker, 대부분의 데이터베이스의 경우, 에이전트는 **적절한 CLI와 셸 명령을 그냥 사용**하면 된다. 모델이 훈련 중 이 도구들을 충분히 봤으므로 이미 사용 방법을 알고 있으며, `grep`과 `jq` 같은 도구와의 **조합 가능성(composability)** + 컨텍스트 효율성이라는 추가 이점도 있다.

**HumanLayer의 사례:** Linear MCP 서버를 사용하다가 실제로는 도구의 작은 하위 집합만 사용한다는 것을 깨닫고, Linear API를 감싸는 **작은 CLI**를 작성. 매우 컨텍스트 효율적인 응답을 제공하고 CLAUDE.md에 6개 사용 예시를 포함.

## 모델-Harness 과적합

프론티어 코딩 모델이 harness 위에서 사후 훈련되므로 (Claude → Claude Code, GPT-5 → Codex), 일부는 "최적의 harness = 모델이 훈련된 harness"라고 주장.

**예시:** Codex 모델이 Codex harness의 `apply_patch` 도구에 너무 밀접하게 결합되어, OpenCode가 Codex 모델 성능 개선을 위해 Codex harness의 `apply_patch`를 모방하는 도구를 추가해야 했다.

하지만 **양날의 검:**
> "Terminal Bench 2.0에서 Opus 4.6이 Claude Code에서는 33위이지만, 사후 훈련 시 보지 못한 다른 harness에서는 **5위** (±4 포지션)."

## Soo에게 의미 있는 이유

이 글은 **실전 harness 엔지니어링의 종합 가이드**다. 이론이 아닌 "수백 번의 에이전트 세션"에서 나온 교훈.

**AInD 컨설팅 핵심 메시지:**
1. **"모델 문제가 아니라 설정 문제"** — 고객이 "AI가 안 된다"고 할 때의 답변
2. **서브에이전트 = 컨텍스트 방화벽** — 장기 작업의 일관성을 위한 핵심 아키텍처 패턴
3. **CLAUDE.md 60줄 미만** — 적을수록 좋다는 구체적 증거
4. **CLI > MCP** — 훈련 데이터에 있는 도구는 MCP로 감싸지 않는 것이 나음
5. **Terminal Bench 33위 → 5위** — harness 변경만으로 순위 역전
