# Skill Issue: Harness Engineering for Coding Agents

- **출처:** [HumanLayer](https://www.humanlayer.dev/blog/skill-issue-harness-engineering-for-coding-agents)
- **태그:** `#HumanLayer` `#SkillIssue` `#CodingAgents` `#Debugging`

---

## 배경

코딩 에이전트의 약한 결과를 모델의 한계로 돌리기 전에, **harness가 충분히 좋은지 먼저 확인하라**는 실용적 주장.

## 핵심 주장: "It's a Skill Issue" (= Harness Issue)

개발자들이 흔히 하는 말: "AI가 만든 코드는 별로다." HumanLayer의 반론: **같은 모델이라도 harness에 따라 결과가 극적으로 달라진다.** 에이전트가 나쁜 코드를 만들면, 모델을 탓하기 전에 harness를 점검하라.

## Harness가 해결하는 5가지 문제

### 1. 컨텍스트 부족
- **증상:** 에이전트가 프로젝트 규칙을 무시하고 자기 방식으로 코딩
- **원인:** 관련 파일, 규칙, 패턴이 컨텍스트에 포함되지 않음
- **해결:** 작업 시작 시 관련 파일을 자동으로 컨텍스트에 포함하는 harness 설계

### 2. 구조 불일치
- **증상:** 기존 코드와 다른 스타일, 패턴, 디렉토리 구조 사용
- **원인:** 아키텍처 규칙이 에이전트에게 전달되지 않음
- **해결:** CLAUDE.md/AGENTS.md에 규칙 명시 + 참조 모듈 지정

### 3. 검증 부재
- **증상:** 코드가 컴파일은 되지만 실제로는 동작하지 않음
- **원인:** 자동 테스트, 빌드 검증이 루프에 포함되지 않음
- **해결:** 코드 생성 → 빌드 → 테스트 → 결과를 에이전트에 피드백하는 루프

### 4. 핸드오프 실패
- **증상:** 새 세션에서 이전 작업을 이어가지 못하고 처음부터 다시 시작
- **원인:** 세션 간 상태 전달이 구조화되지 않음
- **해결:** 핸드오프 아티팩트 (완료 목록, 현재 상태, 다음 할 일)

### 5. 도구 설계 미비
- **증상:** 에이전트가 같은 도구를 잘못된 방식으로 반복 호출
- **원인:** 도구 설명이 모호하거나, 에러 메시지가 도움이 되지 않음
- **해결:** 명확한 도구 설명 + 유용한 에러 메시지 + 예시 포함

## 진단 프로세스

에이전트가 기대에 못 미칠 때:
1. ❓ "컨텍스트에 충분한 정보가 있는가?" — 아니면 → 컨텍스트 개선
2. ❓ "아키텍처 규칙이 전달되었는가?" — 아니면 → CLAUDE.md 개선
3. ❓ "자동 검증이 돌고 있는가?" — 아니면 → 빌드/테스트 루프 추가
4. ❓ "에이전트의 도구가 잘 설계되었는가?" — 아니면 → 도구 개선
5. 위 모두 OK인데도 결과가 나쁘면 → **그때** 모델의 한계를 의심

## Soo에게 의미 있는 이유

AInD 컨설팅의 핵심 메시지로 사용 가능: **"AI가 안 되는 게 아니라, 환경이 안 된 것입니다."** 고객사가 "AI 코딩 도구를 도입했는데 효과가 없다"고 할 때, harness 진단 프로세스를 제안하는 것이 컨설턴트의 가치.
