## 핵심 정의

> **Agent = Model + Harness. 모델이 아니면, 당신은 harness다.**

Harness는 모델 자체가 아닌 **모든 코드, 설정, 실행 로직**이다. 원시 모델은 에이전트가 아니다. Harness가 상태, 도구 실행, 피드백 루프, 강제 가능한 제약을 제공할 때 에이전트가 된다.

구체적으로 harness가 포함하는 것:
- 시스템 프롬프트
- 도구, 스킬, MCP 및 그 설명
- 번들된 인프라 (파일시스템, 샌드박스, 브라우저)
- 오케스트레이션 로직 (서브에이전트 스폰, 핸드오프, 모델 라우팅)
- Hooks/미들웨어 (compaction, 연속, 린트 체크)

## 왜 Harness가 필요한가 — 모델 관점

모델은 (대부분) 텍스트, 이미지, 오디오, 비디오를 입력받아 텍스트를 출력한다. **그게 전부다.** 기본적으로 모델이 할 수 없는 것:

- 상호작용 간 영구 상태 유지
- 코드 실행
- 실시간 지식 접근
- 작업 완료를 위한 환경 설정 및 패키지 설치

이 모든 것이 **harness 레벨 기능**이다. LLM의 구조 자체가 유용한 작업을 위해 감싸는 기계장치(machinery)를 필요로 한다.

## Harness 구성 요소 — 원하는 행동에서 역으로 도출

LangChain은 "원하는 에이전트 행동 → harness 설계"라는 패턴으로 각 구성 요소를 도출한다:

### 1. 파일시스템 — 가장 기초적인 Harness 프리미티브

**원하는 행동:** 에이전트가 실제 데이터와 상호작용하고, 컨텍스트에 맞지 않는 정보를 오프로드하고, 세션 간 작업을 지속하는 것.

**파일시스템이 해제하는 것:**
- 에이전트가 데이터, 코드, 문서를 읽는 작업 공간
- 모든 것을 컨텍스트에 담는 대신 점진적으로 추가/오프로드
- **자연스러운 협업 표면** — 여러 에이전트와 인간이 공유 파일을 통해 조정
- Git = 파일시스템에 버전 관리 추가 → 작업 추적, 오류 롤백, 실험 분기

> "파일시스템은 아마도 가장 기초적인 harness 프리미티브다. 다른 필요한 기능들의 핵심 프리미티브로 계속 재등장하기 때문이다."

### 2. Bash + 코드 실행 — 범용 도구

**원하는 행동:** 인간이 모든 도구를 사전 설계하지 않아도 에이전트가 자율적으로 문제를 해결.

모든 가능한 행동에 대한 도구를 만드는 대신, bash 같은 **범용 도구**를 제공. 모델이 사전 구성된 고정 도구 세트에 제약받지 않고 **코드를 통해 자체 도구를 즉석에서 설계**할 수 있다.

### 3. 샌드박스 — 안전한 실행 환경

**원하는 행동:** 에이전트가 안전하게 행동하고, 결과를 관찰하고, 진전을 만드는 환경.

에이전트가 생성한 코드를 로컬에서 실행하는 것은 위험하고, 단일 로컬 환경은 대규모 에이전트 작업에 확장되지 않는다.

샌드박스의 역할:
- 보안: 격리된 코드 실행
- 확장: 환경을 온디맨드로 생성, 여러 작업에 분산, 완료 후 폐기
- 기본 도구: 언어 런타임, 패키지, git CLI, 테스팅, 브라우저 사전 설치
- **자기 검증 루프**: 코드 작성 → 테스트 → 로그 검사 → 오류 수정

### 4. 메모리와 검색 — 지속적 학습

**원하는 행동:** 에이전트가 보았던 것을 기억하고, 훈련 시점에 존재하지 않던 정보에 접근.

- 파일시스템이 다시 핵심 프리미티브: AGENTS.md 같은 **메모리 파일 표준**
- 에이전트가 이 파일을 추가/편집 → harness가 업데이트된 파일을 컨텍스트에 로드
- 이것은 **지속적 학습(continual learning)의 한 형태** — 한 세션의 지식을 미래 세션에 주입
- 웹 검색 + MCP 도구(Context7 등)로 지식 컷오프 이후 정보 접근

### 5. Context Rot 대응

**원하는 행동:** 작업 과정에서 에이전트 성능이 저하되지 않는 것.

**Context Rot:** 컨텍스트 윈도우가 채워질수록 추론과 작업 완수 능력이 **악화**되는 현상. 컨텍스트는 귀중하고 희소한 자원.

> "오늘날의 harness는 대부분 **좋은 컨텍스트 엔지니어링을 위한 전달 메커니즘**이다."

**세 가지 대응 전략:**

1. **Compaction** — 컨텍스트 윈도우가 거의 찰 때 기존 컨텍스트를 지능적으로 오프로드/요약하여 작업 계속
2. **도구 호출 오프로딩** — 임계 토큰 수를 초과하는 도구 출력의 앞/뒤만 유지하고 전체 출력은 파일시스템으로 오프로드. 필요 시 모델이 접근 가능
3. **스킬을 통한 점진적 공개(Progressive Disclosure)** — 너무 많은 도구/MCP 서버가 시작 시 로드되면 작업 시작 전부터 성능 저하. 스킬이 이를 해결

### 6. 장기 자율 실행

**원하는 행동:** 에이전트가 복잡한 작업을 자율적으로, 정확하게, 긴 시간에 걸쳐 완수.

앞선 harness 프리미티브들이 **복합적으로 작용**하는 지점:

**Ralph Loop 패턴:**
모델의 종료 시도를 hook으로 가로채고, 깨끗한 컨텍스트 윈도우에서 원래 프롬프트를 재주입하여 **완료 목표에 대해 작업을 계속하도록 강제**. 파일시스템 덕분에 가능 — 각 반복이 신선한 컨텍스트로 시작하지만 이전 반복의 상태를 파일에서 읽음.

**계획 + 자기 검증:**
- 계획: 모델이 목표를 단계로 분해. Harness가 좋은 프롬프팅과 파일시스템의 계획 파일 사용 리마인더로 지원
- 자기 검증: 각 단계 완료 후 정확성 확인. Harness의 hook이 사전 정의된 테스트 스위트를 실행하고 실패 시 에러 메시지와 함께 모델로 루프백

## Harness의 미래

### 모델 훈련과 Harness 설계의 결합

Claude Code, Codex 같은 에이전트 제품은 **모델과 harness를 함께 포함하여 사후 훈련**된다. 이는 모델이 harness 설계자가 중요하다고 생각하는 행동(파일시스템 작업, bash 실행, 계획, 서브에이전트 병렬화)에 네이티브하게 능숙해지도록 한다.

**흥미로운 부작용 — 일반화 문제:**
- 도구 로직을 바꾸면 모델 성능이 나빠질 수 있음
- 예: Codex-5.3의 apply_patch 도구 — 진정으로 지능적인 모델은 패치 방식 전환에 어려움이 없어야 하지만, harness와 함께 훈련하면 이런 과적합(overfitting)이 발생

**하지만 최적의 harness가 반드시 모델이 훈련된 harness는 아니다:**
- Terminal Bench 2.0 리더보드에서 **Opus 4.6이 Claude Code에서는 다른 harness보다 훨씬 낮은 점수**
- LangChain은 harness만 변경하여 **Terminal Bench 2.0에서 Top 30 → Top 5**로 향상

### Harness 엔지니어링의 방향

모델이 더 capable해지면 오늘날 harness에 있는 것 중 일부가 모델에 흡수될 것. 모델이 계획, 자기 검증, 장기 일관성에서 네이티브하게 더 능숙해져 컨텍스트 주입이 덜 필요해질 것.

> "이는 harness가 시간이 지나면 덜 중요해질 것을 시사한다. 하지만 프롬프트 엔지니어링이 오늘날에도 여전히 가치 있듯이, harness 엔지니어링도 좋은 에이전트를 만드는 데 계속 유용할 것이다."

LangChain이 탐구 중인 열린 문제:
- 수백 개의 에이전트가 공유 코드베이스에서 병렬로 작업하는 오케스트레이션
- 에이전트가 자신의 trace를 분석하여 harness 레벨 실패 모드를 식별하고 수정
- 작업에 맞춰 올바른 도구와 컨텍스트를 **just-in-time으로 동적 조립**하는 harness

## Soo에게 의미 있는 이유

이 글은 harness의 구성 요소를 **"모델이 할 수 없는 것"에서 역으로 도출**하는 사고방식을 제공한다. AInD 컨설팅에서 "harness란 무엇인가"를 설명할 때 가장 체계적인 프레임워크.

**특히 주목할 발견들:**
1. **Terminal Bench에서 harness만 변경으로 Top 30 → Top 5** — harness의 가치를 수치로 증명
2. **파일시스템이 가장 기초적 프리미티브** — 메모리, 협업, 상태 관리 모두 파일시스템 기반
3. **Ralph Loop** — 에이전트가 작업을 조기 종료하지 않도록 하는 구체적 패턴
4. **Context Rot** — 컨텍스트가 채워질수록 성능이 저하되는 현상과 3가지 대응 전략
5. **모델-harness 과적합** — 모델이 훈련된 harness와 다른 harness에서 성능이 달라질 수 있음
