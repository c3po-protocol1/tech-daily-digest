## 배경

"에이전트 = 모델 + Harness"라는 간결한 정의에서 출발하여, harness의 구성 요소를 체계적으로 분해한 LangChain의 아키텍처 글.

## Harness의 5가지 레이어

### Layer 1: Prompts (프롬프트)
- 시스템 프롬프트 — 에이전트의 역할, 규칙, 제약을 정의
- 대화 컨텍스트 — 이전 대화 내용
- 도구 결과 — 도구 호출 결과를 컨텍스트에 포함
- **핵심:** 프롬프트가 harness의 "영혼". 같은 모델이라도 프롬프트에 따라 전혀 다른 에이전트가 됨

### Layer 2: Tools (도구)
- 에이전트가 세상과 상호작용하는 인터페이스
- 파일 읽기/쓰기, 명령 실행, API 호출, 웹 검색 등
- 도구의 **설명(description)**이 에이전트의 도구 선택을 결정
- 도구가 너무 많으면 혼란, 너무 적으면 능력 제한

### Layer 3: Middleware (미들웨어)
- 입력 전처리 — 사용자 입력을 정규화, 검증
- 출력 후처리 — 에이전트 출력을 포맷팅, 필터링
- 로깅 — 모든 도구 호출과 결정을 기록
- 재시도 — 도구 실패 시 자동 재시도 로직
- 검증 — 출력이 제약 조건을 만족하는지 확인

### Layer 4: Orchestration (오케스트레이션)
- 멀티 에이전트 조정 — 여러 에이전트의 역할 분배
- 작업 분해 — 큰 작업을 작은 단위로 나누기
- 핸드오프 — 에이전트 간 상태 전달
- 병렬 실행 — 독립적 작업의 동시 처리
- **단일 에이전트에서는 불필요하지만, 복잡한 시스템에서는 핵심**

### Layer 5: Runtime Infrastructure (런타임 인프라)
- 실행 환경 — 컨테이너, 샌드박스, 클라우드
- 상태 관리 — 세션 상태의 영속성
- 내구성(durability) — 프로세스 재시작에도 작업 유지
- 모니터링 — 실행 상태 추적, 알림

## 프레임워크 vs 런타임 vs Harness

| 개념 | 정의 | 특징 |
|------|------|------|
| **프레임워크** | 에이전트 구축 라이브러리 | 범용, 재사용 가능, 여러 프로젝트에 적용 |
| **런타임** | 실행 인프라 | 상태, 재시도, 내구성, 동시성 |
| **Harness** | 특정 작업용 맞춤 환경 | 프롬프트 + 도구 + 검증의 구체적 조합 |

**핵심 구분:** 프레임워크는 "에이전트를 어떻게 만드는가", harness는 "이 에이전트를 이 작업에서 어떻게 잘 동작시키는가". 좋은 harness는 프레임워크 없이도 만들 수 있다 (셸 스크립트 + 프롬프트로도 가능).

## Soo에게 의미 있는 이유

Harness의 구성 요소를 처음 접하는 사람에게 **가장 좋은 입문 자료**. AInD 컨설팅에서 "harness란 무엇인가"를 설명할 때 이 5-레이어 모델을 프레임워크로 사용할 수 있다.
