# Your Agent Needs a Harness, Not a Framework

## 배경

Inngest(이벤트 드리븐 인프라 회사)의 주장: 에이전트에게 필요한 것은 범용 프레임워크가 아니라 **인프라 수준의 harness**.

## 프레임워크의 한계

대부분의 에이전트 프레임워크가 집중하는 것:
- ✅ 프롬프트 체이닝
- ✅ 도구 등록
- ✅ 대화 관리
- ✅ RAG 통합

프레임워크가 **놓치는 것** (하지만 프로덕션에서 가장 문제되는 것):
- ❌ 상태의 영속성 (프로세스 재시작 시 상태 소실)
- ❌ 실패 시 자동 재시도 (에이전트가 API 429를 만나면?)
- ❌ 실행 경로 추적 (디버깅)
- ❌ 동시성 제어 (같은 리소스에 여러 에이전트 접근)
- ❌ 비용 제한 (에이전트가 무한 루프에 빠지면?)

## Harness가 제공해야 하는 5가지 인프라

### 1. State (상태 관리)
- 에이전트의 현재 상태를 **영속적으로** 관리
- 프로세스가 죽어도 상태가 유지됨
- 체크포인트에서 재개 가능

### 2. Retries (재시도)
- API 실패, 타임아웃 시 **자동 재시도**
- 지수 백오프 (exponential backoff)
- 최대 재시도 횟수 제한
- 재시도 불가능한 에러 구분 (4xx vs 5xx)

### 3. Traces (추적)
- 전체 실행 경로를 **타임라인으로** 기록
- 각 도구 호출, 결정, 에러를 시간순으로 추적
- "에이전트가 왜 이런 결정을 했는가"를 사후에 분석 가능
- **Holocron이 제공하려는 것과 정확히 같은 목적**

### 4. Concurrency (동시성)
- 여러 에이전트가 같은 파일/리소스에 동시 접근할 때의 충돌 관리
- 잠금(locking), 큐잉, 우선순위
- 에이전트 간 작업 분배와 결과 수집

### 5. Durability (내구성)
- **프로세스 재시작에도 작업이 유지**
- 장기 실행 작업이 서버 재부팅에도 중단되지 않음
- 이벤트 소싱 패턴으로 상태 복원

## 프레임워크 vs Harness 비교

| 관심사 | 프레임워크 | Harness |
|--------|-----------|---------|
| 초점 | 에이전트 구축 방법 | 에이전트 안정 운영 |
| 수명 | 프로토타입에서 프로덕션까지 | **프로덕션 전용** |
| 관리 대상 | 프롬프트, 도구, 체인 | 상태, 재시도, 추적, 동시성 |
| 실패 시 | 에러 반환 | **자동 복구** |

## Soo에게 의미 있는 이유

AInD 컨설팅에서 "프로덕션에서 에이전트를 운영하려면 무엇이 필요한가"를 설명할 때 핵심 자료. 많은 기업이 프레임워크를 도입하면 프로덕션 준비가 끝났다고 생각하지만, 실제로는 **인프라 레이어의 harness가 없으면 프로덕션에서 실패**한다.
