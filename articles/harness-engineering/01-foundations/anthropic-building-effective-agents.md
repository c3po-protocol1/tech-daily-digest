## 배경

Anthropic이 수십 개의 팀과 업계 전반에서 LLM 에이전트를 구축한 경험을 바탕으로 작성한 에이전트 설계 총론. **업계에서 가장 많이 인용되는 에이전트 가이드**이며, 모든 harness 논의의 출발점이 되는 글.

핵심 발견:
> "가장 성공적인 구현들은 복잡한 프레임워크나 전문 라이브러리를 사용하지 않았다. 대신 **단순하고 조합 가능한 패턴**으로 구축했다."

## 핵심 원칙: 가장 단순한 솔루션부터

> "LLM으로 애플리케이션을 구축할 때, 가능한 가장 단순한 솔루션을 찾고, 필요할 때만 복잡도를 높여라."

이것은 에이전트를 **아예 만들지 않는 것**을 의미할 수도 있다. 에이전트 시스템은 종종 더 나은 작업 성능을 위해 **지연 시간과 비용을 교환**하며, 이 트레이드오프가 언제 의미 있는지를 고려해야 한다.

## 워크플로우 vs 에이전트

| 개념 | 정의 | 특성 |
|------|------|------|
| **워크플로우** | LLM과 도구가 **사전 정의된 코드 경로**로 오케스트레이션 | 예측 가능, 일관적, 디버깅 쉬움 |
| **에이전트** | LLM이 **자율적으로 프로세스와 도구 사용을 지시** | 유연하지만 비결정적 |

많은 애플리케이션에서 검색과 in-context 예시로 **단일 LLM 호출을 최적화하는 것만으로 충분**하다.

## 5가지 워크플로우 패턴 (단순 → 복잡)

### 1. Prompt Chaining (프롬프트 체이닝)

작업을 순차적 단계로 분해. 각 LLM 호출이 이전의 출력을 처리. 중간에 프로그래밍적 체크("게이트")를 추가하여 프로세스가 올바른 궤도에 있는지 확인 가능.

**언제 사용:** 작업이 고정된 하위 작업으로 쉽고 깔끔하게 분해될 때. 지연 시간을 감수하고 더 높은 정확도를 얻기 위해 각 LLM 호출을 더 쉬운 작업으로 만듦.

**예시:**
- 마케팅 카피 생성 → 다른 언어로 번역
- 문서 아웃라인 작성 → 기준 충족 확인 → 아웃라인 기반 문서 작성

### 2. Routing (라우팅)

입력을 분류하고 전문화된 후속 작업으로 지시. 관심사 분리와 더 전문화된 프롬프트 구축을 가능하게 함.

**언제 사용:** 별도로 처리하면 더 나은 뚜렷한 카테고리가 있고, 분류가 정확하게 이루어질 수 있는 복잡한 작업.

**예시:**
- 고객 서비스 쿼리를 유형별로 라우팅 (일반 질문 / 환불 / 기술 지원)
- 쉬운 질문 → Haiku(저비용), 어려운 질문 → Sonnet(고성능)으로 라우팅

### 3. Parallelization (병렬화)

LLM이 작업에 동시에 작업하고 출력을 프로그래밍적으로 집계. 두 가지 변형:
- **Sectioning:** 독립적 하위 작업으로 나누어 병렬 실행
- **Voting:** 같은 작업을 여러 번 실행하여 다양한 출력 확보

**예시:**
- Sectioning: 한 모델이 쿼리 처리, 다른 모델이 부적절한 콘텐츠 스크리닝 (같은 LLM에서 둘 다 하는 것보다 성능이 더 좋음)
- Voting: 코드 취약점 리뷰를 여러 프롬프트로 실행하여 발견 시 플래그

### 4. Orchestrator-Workers (오케스트레이터-워커)

중앙 LLM이 **동적으로** 작업을 분해하고, 워커 LLM에 위임하고, 결과를 종합.

**핵심 차이 (병렬화와):** 하위 작업이 사전 정의되지 않고, **오케스트레이터가 입력에 따라 결정**.

**예시:**
- 여러 파일을 복잡하게 변경하는 코딩 제품 (변경 파일 수와 성격이 작업에 따라 다름)
- 여러 소스에서 정보를 수집하고 분석하는 검색 작업

### 5. Evaluator-Optimizer (평가자-최적화자)

한 LLM이 응답을 생성하고, 다른 LLM이 **루프에서** 평가와 피드백을 제공.

**적합 조건 두 가지:**
1. LLM 응답이 인간의 피드백으로 **시연 가능하게 개선**될 수 있을 때
2. LLM이 그러한 **피드백을 제공할 수 있을 때**

이것이 Anthropic의 후속 글(harness-design-long-running-apps)에서 **GAN-inspired Generator-Evaluator 루프**의 기초가 되는 패턴.

**예시:**
- 문학 번역 (뉘앙스를 번역자 LLM이 처음에 포착 못할 수 있지만, 평가자 LLM이 유용한 비평 제공)
- 포괄적 정보를 위해 여러 라운드의 검색/분석이 필요한 복잡한 검색 작업

## 에이전트: 자율 시스템

에이전트가 프로덕션에 등장하는 것은 LLM이 핵심 능력에서 성숙하기 때문:
- 복잡한 입력 이해
- 추론과 계획
- 도구를 **안정적으로** 사용
- 오류에서 복구

에이전트의 실행 패턴:
1. 인간으로부터 명령 또는 대화로 시작
2. 작업이 명확해지면 **독립적으로 계획하고 운영**
3. 추가 정보나 판단이 필요하면 인간에게 복귀
4. 매 단계에서 환경으로부터 **"ground truth"를 획득** (도구 결과, 코드 실행)
5. 체크포인트나 차단점에서 인간 피드백을 위해 일시 중지
6. 완료 시 종료, 또는 **최대 반복 횟수** 같은 정지 조건

## 도구 설계 원칙

> "모델이 올바르게 사용할 수 있는 도구 = 좋은 도구"

- **도구 이름을 명확하고 기술적으로** 만들어 모델의 의사결정을 도움
- **가능하면 도구 수를 줄여** 혼란 방지
- **필수 파라미터만 포함** — 선택적 파라미터는 모델이 혼란스러워할 수 있음
- **유용한 에러 메시지** — "왜 실패했고 어떻게 하면 성공할 수 있는지"를 포함

## 프레임워크 사용에 대한 조언

> "개발자가 **LLM API를 직접** 사용하여 시작할 것을 제안한다: 많은 패턴이 몇 줄의 코드로 구현될 수 있다."

프레임워크를 사용하더라도 **내부 코드를 이해**하라. "후드 아래에 무엇이 있는지"에 대한 잘못된 가정이 **고객 오류의 가장 흔한 원인**.

프레임워크의 장점: 표준 저수준 작업(LLM 호출, 도구 정의/파싱, 호출 체이닝) 단순화
프레임워크의 위험: 추상화 레이어가 기저 프롬프트/응답을 **불투명**하게 만듦 → 디버깅 어려움

## Soo에게 의미 있는 이유

이 글은 **에이전트 설계의 바이블**이다. AInD 컨설팅에서 모든 논의의 출발점.

**핵심 활용 포인트:**
1. **"워크플로우 vs 에이전트" 프레임워크** — 고객에게 "에이전트가 항상 답은 아니다"를 설명할 때 핵심
2. **5가지 패턴의 점진적 복잡도** — 고객의 현재 수준에 맞는 패턴 추천 가능
3. **"가장 단순한 솔루션부터"** — 과도한 복잡도를 경계하는 원칙
4. **Evaluator-Optimizer** — GAN-inspired harness의 이론적 기초
5. **도구 설계 원칙** — MCP 도구 설계 교육에 바로 사용 가능
6. **프레임워크 경고** — 프레임워크 도입 시 "내부를 이해하라"는 메시지
