## 배경

Anthropic의 에이전트 설계 총론이자, **가장 널리 인용되는** 에이전트 가이드. 업계 전체의 에이전트 설계 사고방식에 영향을 미친 글.

## 핵심 원칙: 가장 단순한 솔루션부터

> "에이전트를 만들기 전에, 워크플로우로 충분한지 확인하라."

- **워크플로우:** 사전 정의된 코드 경로. 결정적(deterministic). 예측 가능. 디버깅 쉬움.
- **에이전트:** 모델이 자율적으로 도구를 선택하고 경로를 결정. 비결정적. 유연하지만 예측 어려움.

복잡도를 정당화할 수 있을 때만 에이전트를 선택해야 한다.

## 워크플로우 패턴 (단순 → 복잡)

### 1. Prompt Chaining (프롬프트 체이닝)
작업을 순차적 단계로 분해. 각 단계의 출력이 다음 단계의 입력.
- 중간에 **게이트(gate)** 추가 가능 — 품질 확인 후 다음 단계로 진행
- 가장 단순하고 예측 가능한 패턴

### 2. Routing (라우팅)
입력을 분류하고 적절한 처리 경로로 분배.
- 고객 문의 분류 → 기술/결제/일반으로 라우팅
- 각 경로는 특화된 프롬프트와 도구를 가짐

### 3. Parallelization (병렬화)
독립적인 작업을 동시에 실행.
- 여러 파일을 동시에 분석
- 검색 + 코드 생성을 병렬로

### 4. Orchestrator-Worker (오케스트레이터-워커)
중앙 조정자가 작업을 분배하고 결과를 수집.
- 오케스트레이터: 전체 계획 수립, 작업 할당
- 워커: 할당된 작업 실행, 결과 반환

### 5. Evaluator-Optimizer (평가자-최적화자)
생성과 평가의 반복 루프.
- Generator가 출력 생성
- Evaluator가 품질 평가 + 피드백
- Generator가 피드백 반영하여 재생성
- **GAN-inspired harness**의 기초가 되는 패턴

## 도구 설계 원칙

**모델이 올바르게 사용할 수 있는 도구 = 좋은 도구**

1. **명확한 설명** — 언제, 왜, 어떻게 사용하는지를 도구 설명에 포함
2. **단순한 스키마** — 파라미터가 너무 많으면 에이전트가 혼란
3. **예측 가능한 출력** — 같은 입력에 일관된 출력 형식
4. **유용한 에러 메시지** — "실패"가 아니라 "왜 실패했고 어떻게 하면 성공할 수 있는지"
5. **적절한 범위** — 하나의 도구가 한 가지 일만. Unix 철학.

## Soo에게 의미 있는 이유

에이전트 설계의 **바이블**. AInD 컨설팅에서 가장 먼저 읽어야 할 자료. 특히 "워크플로우 vs 에이전트" 구분은 고객사에게 "에이전트가 항상 답은 아니다"를 설명할 때 핵심 프레임워크.
