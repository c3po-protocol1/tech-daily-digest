# Harness Engineering (Thoughtworks / Martin Fowler)

- **태그:** `#Thoughtworks` `#MartinFowler` `#Entropy` `#ContextEngineering`

## 배경

Thoughtworks(Martin Fowler 팀)가 정의한 harness 엔지니어링 프레이밍. 특히 **"엔트로피 대응"**이라는 독특한 관점이 핵심.

## Harness 작업의 3가지 축

### 축 1: Context Engineering (컨텍스트 엔지니어링)
에이전트에게 올바른 정보를, 올바른 시점에, 적절한 형태로 제공하는 것.

- 프로젝트 구조, 기존 패턴, 의존성 정보를 **선택적으로** 컨텍스트에 포함
- "모든 것을 넣지 마라" — 신호 대 잡음 비율이 핵심
- 작업 진행에 따라 컨텍스트를 **동적으로 조절**

### 축 2: Architectural Constraints (아키텍처 제약)
에이전트가 만들 수 있는 구조적 결정의 범위를 제한하는 것.

- 디렉토리 구조 규칙
- 의존성 방향 규칙 (어떤 모듈이 어떤 모듈을 import할 수 있는지)
- 금지 패턴 (e.g., "전역 상태 사용 금지", "직접 DB 쿼리 금지")
- 강제 패턴 (e.g., "모든 API는 이 미들웨어를 거쳐야 함")

### 축 3: Garbage Collection / 엔트로피 대응
**에이전트가 만든 코드의 품질 저하를 지속적으로 정리하는 것.**

이것이 Thoughtworks 프레이밍의 가장 독특한 기여다.

**엔트로피의 구체적 형태:**
- 중복 코드 누적 — 에이전트가 기존 유틸리티를 모르고 새로 만듦
- 일관성 깨짐 — 세션마다 다른 스타일, 네이밍, 패턴 사용
- 불필요한 복잡도 — 간단한 문제에 과도한 추상화
- 죽은 코드 — 리팩토링 후 남은 사용되지 않는 코드
- 테스트 부채 — 기능은 추가되지만 테스트가 따라가지 못함

**대응 전략:**
- **자동 린트/포매팅** — 스타일 일관성 자동 유지
- **아키텍처 테스트** — 의존성 규칙 위반 자동 탐지 (e.g., ArchUnit)
- **중복 탐지** — 유사한 코드 블록을 자동으로 식별
- **정기적 리팩토링 세션** — 에이전트에게 "코드를 정리하라"는 별도 작업 부여
- **메트릭 모니터링** — 코드 복잡도, 파일 크기, 의존성 깊이 추적

## 핵심 통찰

> "Harness 엔지니어링의 절반은 에이전트를 돕는 것이고, 나머지 절반은 에이전트가 만든 엔트로피를 청소하는 것이다."

대부분의 harness 논의가 "에이전트를 더 잘 동작시키기"에 집중하는 반면, Thoughtworks는 **"에이전트가 만든 쓰레기를 어떻게 치우는가"**에 동등한 비중을 둔다. 이것이 장기적으로 코드베이스의 건강을 유지하는 핵심이다.

## Soo에게 의미 있는 이유

"한국에서 AI 프로젝트가 망하는 이유"와 직접 연결된다. 한국 개발 문화의 문제 — 코드를 쌓기만 하고 정리하지 않는 것 — 이 AI 시대에 **엔트로피 폭증**으로 이어진다. Thoughtworks의 "garbage collection" 프레이밍은 이 문제의 해결책을 제시한다.
