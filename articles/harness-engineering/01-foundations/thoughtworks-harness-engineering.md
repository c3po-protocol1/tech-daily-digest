## 배경

OpenAI의 harness engineering 글에 대한 Thoughtworks의 분석과 독자적 관점을 제시한 글. 저자 Birgitta Böckeler는 20년 이상의 소프트웨어 개발 경험을 가진 Distinguished Engineer이며 AI 보조 개발 전문가다.

## OpenAI Harness의 3가지 카테고리 분류

저자는 OpenAI 팀의 harness 구성 요소를 **결정론적(deterministic)과 LLM 기반의 혼합**으로 분류한다:

### 1. Context Engineering (컨텍스트 엔지니어링)
- 코드베이스 내 지속적으로 향상되는 지식 베이스
- 관측성 데이터, 브라우저 네비게이션 같은 **동적 컨텍스트**에 대한 에이전트 접근

### 2. Architectural Constraints (아키텍처 제약)
- LLM 기반 에이전트뿐 아니라 **결정론적 커스텀 린터와 구조 테스트**로도 모니터링
- 이것이 핵심: 아키텍처 규칙은 LLM에 의존하지 않고 **코드로 강제**

### 3. "Garbage Collection" (엔트로피 대응)
- 주기적으로 실행되는 에이전트들이 문서의 불일치나 아키텍처 제약 위반을 찾음
- **엔트로피와 부패에 맞서 싸우는 것**

저자가 인용한 핵심 문구:
> "에이전트가 어려움을 겪으면, 우리는 이를 신호로 취급한다: 빠진 것이 무엇인지 — 도구, 가드레일, 문서 — 를 식별하고, 항상 Codex 자체에 수정을 작성하게 하여 리포에 피드백한다."

## 저자의 독자적 관점과 질문들

### "Harness가 미래의 서비스 템플릿이 될 것인가?"

대부분의 조직은 2~3개의 주요 기술 스택만 사용한다. 저자는 미래에 팀이 **일반적인 애플리케이션 토폴로지를 위한 harness 세트에서 선택**하여 시작하는 세계를 상상한다.

이것은 오늘날의 **서비스 템플릿**을 연상시킨다 — 팀이 "골든 패스" 위에서 새 서비스를 인스턴스화하는 것. Harness가 커스텀 린터, 구조 테스트, 기본 컨텍스트/지식 문서, 추가 컨텍스트 프로바이더를 포함한 **새로운 서비스 템플릿**이 될 것인가?

**미해결 질문:** 서비스 템플릿에서 팀들이 경험을 쌓으면서 기여하지만, 다른 팀들은 업데이트를 통합하는 데 어려움을 겪는다. Harness에서도 **유사한 포크와 동기화 문제**를 겪게 될 것인가?

### "더 많은 AI 자율성을 위해 런타임을 제약해야 하는가?"

초기 AI 코딩 과대광고는 LLM이 무한한 유연성을 줄 것이라고 가정했다. **어떤 언어, 어떤 패턴으로든** 제약 없이 생성 — LLM이 알아서 할 것이다.

하지만 대규모로 신뢰할 수 있는 AI 생성 코드를 위해, **무언가를 포기해야 한다.**

> harness는 신뢰와 신뢰성을 높이기 위해 **솔루션 공간을 제약**해야 했다: 특정 아키텍처 패턴, 강제된 경계, 표준화된 구조. 이는 "무엇이든 생성" 유연성의 일부를 프롬프트, 규칙, 기술적 세부사항으로 가득한 harness에게 양보하는 것이다.

### "제한된 수의 기술 스택과 토폴로지로 수렴될 것인가?"

코딩이 코드 타이핑보다 생성 조정에 가까워지면서, AI가 우리를 **더 적은 기술 스택**으로 밀어낼 수 있다.

- 프레임워크/SDK의 사용성은 여전히 중요 — **인간에게 좋은 것이 AI에게도 좋다**
- 하지만 개발자 취향은 세부 수준에서 덜 중요해짐
- 인터페이스의 작은 비효율성은 직접 다루지 않으므로 덜 신경쓰임
- **좋은 harness가 있는 스택을 선택**하고 "AI 친화성"을 우선시할 수 있음
- 데이터 구조를 안정적으로 유지하고 모듈 경계를 정의/강제하는 것이 핵심 초점

### "Pre-AI vs Post-AI 애플리케이션 유지보수의 두 세계?"

AI 자율성을 9까지 높이는 좋은 harness 기법을 개발한다고 가정하면:
- 어떤 기법은 기존 애플리케이션에 적용 가능
- 어떤 기법은 **처음부터 harness를 염두에 두고 구축된 애플리케이션에만 작동**

기존 코드베이스에 harness를 후속 적용하는 것이 가치가 있을지 고려해야 한다. AI가 이를 더 빠르게 할 수 있지만, 오래된 애플리케이션은 종종 **너무 비표준적이고 엔트로피로 가득**하여 가치가 없을 수 있다.

> "정적 코드 분석 도구를 한 번도 사용하지 않은 코드베이스에 처음 돌린 후, 경고에 익사하는 것과 비슷하다."

### 저자의 비판: 기능/동작 검증의 부재

> "내가 이 글에서 빠졌다고 느끼는 것은 **기능과 동작의 검증**이다."

OpenAI 글은 장기적 내부 품질과 유지보수성에 초점을 맞추지만, 에이전트가 만든 것이 **실제로 사용자가 원하는 대로 동작하는지** 검증하는 부분이 약하다.

### "오늘 당신의 Harness는 무엇인가?"

> "이 팀이 5개월 동안 harness를 작업했다는 것은 **이것이 빠른 결과를 위해 뛰어들 수 있는 것이 아님**을 보여준다."

하지만 오늘 당신의 harness가 무엇인지 되돌아볼 가치가 있다:
- pre-commit hook이 있는가? 무엇이 들어있는가?
- 커스텀 린터 아이디어가 있는가?
- 코드베이스에 어떤 아키텍처 제약을 부과하고 싶은가?
- ArchUnit 같은 구조 테스트 프레임워크를 실험해봤는가?

## 최종 관찰

> "그들이 설명하는 것은 Markdown 규칙 파일 몇 개를 생성하고 유지하는 것보다 **훨씬 더 많은 작업**처럼 들린다."

결정론적 harness 부분을 위한 광범위한 도구 구축. 컨텍스트 엔지니어링은 지식 베이스 큐레이션뿐 아니라 중대한 설계 작업을 포함 — **코드 설계 자체가 컨텍스트의 거대한 부분**.

> "마지막으로, 이 분야에서 처음으로 용어가 마음에 든다. 아직 2주밖에 안 됐지만 — 누군가 단일 프롬프트, LLM 기반 코드 리뷰 에이전트를 'harness'라고 부를 때까지 숨을 참을 수 있을 것 같다..."

## Soo에게 의미 있는 이유

이 글의 가치는 **비판적 시각**에 있다. OpenAI의 주장을 그대로 받아들이지 않고, 실무자 관점에서 질문을 던진다.

**AInD 컨설팅에서 핵심적인 통찰들:**
1. **"Harness = 미래의 서비스 템플릿"** — 기업에 harness를 "표준화된 AI 개발 환경"으로 포지셔닝 가능
2. **Pre-AI vs Post-AI 코드베이스** — 기존 시스템에 AI를 도입할 때의 현실적 한계를 인정
3. **"솔루션 공간 제약 = 신뢰성"** — "AI가 아무거나 생성"이 아닌 "제약 안에서 정확히 생성"이 프로덕션의 길
4. **5개월의 투자** — harness 구축이 빠른 승리(quick win)가 아님을 솔직히 인정
5. **기능 검증의 부재 비판** — harness가 구조적 품질은 보장하지만 사용자 관점 검증은 별도로 필요
