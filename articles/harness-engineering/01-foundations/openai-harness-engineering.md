## 실험의 전제

OpenAI 팀이 5개월간 실행한 실험: **수동으로 작성된 코드 0줄**로 소프트웨어 제품을 구축하고 출시하는 것. 이 제품은 내부 일일 사용자와 외부 알파 테스터를 가진 실제 제품이다. 배포되고, 장애가 나고, 수정된다. 다른 점은 **모든 코드 — 애플리케이션 로직, 테스트, CI 설정, 문서, 관측성, 내부 도구 — 를 Codex가 작성**했다는 것이다.

결과: 수동 코딩 대비 **약 1/10의 시간**으로 구축. 인간은 조종(steer)하고, 에이전트가 실행(execute)한다.

## 규모와 수치

- 2025년 8월 말 첫 커밋 (빈 리포지토리)
- 5개월 후: **약 100만 줄의 코드** (애플리케이션, 인프라, 도구, 문서, 유틸리티 포함)
- **약 1,500개의 PR** 생성 및 머지
- 3명의 엔지니어 → 이후 7명으로 확장
- **엔지니어당 평균 3.5 PR/일** (팀이 커져도 throughput 증가)
- 단일 Codex 실행이 **6시간 이상** 연속으로 작업하는 경우가 정기적으로 관찰 (종종 인간이 자는 동안)
- 수백 명의 내부 사용자가 실제 사용 중

**인간은 코드를 직접 작성하지 않았다.** 이것이 팀의 핵심 철학이 되었다.

## 엔지니어의 역할 재정의

초기 진행이 예상보다 느렸다. Codex의 능력이 부족해서가 아니라 **환경이 불충분하게 명세**되어 있었기 때문이다. 에이전트에게 고수준 목표를 향해 진전할 수 있는 도구, 추상화, 내부 구조가 부족했다.

**엔지니어의 주요 업무 = 에이전트가 유용한 작업을 할 수 있도록 하는 것(enabling)**

실제 워크플로우:
1. 큰 목표를 작은 빌딩 블록으로 분해
2. 에이전트에게 그 블록을 구성하도록 프롬프트
3. 완성된 블록으로 더 복잡한 작업을 해제(unlock)
4. 무언가 실패하면 → "더 열심히 시도"가 아닌 **"어떤 능력이 빠져있고, 어떻게 에이전트에게 제공할 수 있는가"**를 질문

인간은 프롬프트를 통해 시스템과 상호작용: 작업을 설명 → 에이전트 실행 → PR 생성. Codex가 자체 변경사항을 로컬에서 리뷰하고, 추가 에이전트 리뷰를 요청하고, 피드백에 응답하고, 모든 리뷰어가 만족할 때까지 반복.

**인간의 PR 리뷰가 필수가 아니다.** 시간이 지나면서 거의 모든 리뷰 작업이 **에이전트 대 에이전트**로 처리.

## 애플리케이션 가독성(Legibility) 증가

코드 throughput이 증가하면서 **병목은 인간의 QA 용량**이 됨. 고정 제약은 인간의 시간과 주의력이므로, 에이전트에게 더 많은 능력을 추가하여 해결.

### 구체적 기법들

**Git worktree 기반 앱 실행:**
- 변경사항별로 앱을 독립적으로 부팅 가능하게 설계
- Codex가 변경사항마다 하나의 앱 인스턴스를 실행 가능

**Chrome DevTools Protocol 통합:**
- 에이전트 런타임에 CDP를 연결
- DOM 스냅샷, 스크린샷, 네비게이션 스킬 생성
- Codex가 직접 버그를 재현하고, 수정을 검증하고, UI 동작을 추론

**관측성 도구 연결:**
- 로그, 메트릭, 트레이스를 로컬 관측성 스택으로 노출
- 각 worktree별로 임시(ephemeral) 스택 → 작업 완료 후 폐기
- Codex가 **LogQL로 로그를 쿼리하고, PromQL로 메트릭을 쿼리**
- 프롬프트 예시: "서비스 시작이 800ms 이내에 완료되도록 보장하라", "이 4개 중요 사용자 여정에서 어떤 span도 2초를 초과하지 않도록 하라"

## 리포지토리 지식 = 시스템 오브 레코드

### "거대한 AGENTS.md" 접근의 실패

하나의 큰 AGENTS.md를 시도했으나 예측 가능한 방식으로 실패:

1. **컨텍스트는 희소 자원** — 거대한 지시 파일이 작업, 코드, 관련 문서를 밀어냄. 에이전트가 핵심 제약을 놓치거나 잘못된 것에 최적화.
2. **너무 많은 가이던스 = 가이던스 없음** — 모든 것이 "중요"하면 아무것도 중요하지 않음. 에이전트가 의도적으로 탐색하지 않고 지역적 패턴 매칭에 의존.
3. **즉시 부패** — 단일체 매뉴얼은 낡은 규칙의 무덤이 됨. 에이전트는 무엇이 여전히 참인지 알 수 없고, 인간은 유지보수를 중단.
4. **검증 불가** — 단일 블롭은 기계적 체크(커버리지, 신선도, 소유권, 교차 링크)에 적합하지 않음.

### 해결: AGENTS.md = 목차, docs/ = 본문

AGENTS.md(약 100줄)는 **짧은 지도 역할**만 하고, 더 깊은 진실의 원천을 가리킴:

```
AGENTS.md          ← 목차 (~100줄)
ARCHITECTURE.md    ← 아키텍처 맵
docs/
├── design-docs/   ← 설계 문서 (인덱스, 핵심 신념)
├── exec-plans/    ← 실행 계획 (active, completed, tech-debt)
├── generated/     ← 자동 생성 문서 (DB 스키마 등)
├── product-specs/ ← 제품 스펙
├── references/    ← 참조 자료 (디자인 시스템, 라이브러리 등)
├── DESIGN.md
├── FRONTEND.md
├── PLANS.md
├── PRODUCT_SENSE.md
├── QUALITY_SCORE.md
├── RELIABILITY.md
└── SECURITY.md
```

**점진적 공개(Progressive Disclosure):** 에이전트가 작고 안정적인 진입점에서 시작하여, 필요한 곳을 찾아가도록 가르침. 처음부터 압도하지 않음.

**기계적 강제:** 전용 린터와 CI 작업이 지식 베이스의 최신성, 교차 링크, 구조 정확성을 검증. **"문서 정원사(doc-gardening)" 에이전트**가 정기적으로 실제 코드 동작과 일치하지 않는 오래된 문서를 스캔하고 수정 PR을 생성.

### 에이전트 관점의 가독성

> "에이전트의 관점에서, 실행 중 in-context로 접근할 수 없는 것은 사실상 존재하지 않는다."

Google Docs, Slack 스레드, 사람의 머릿속에 있는 지식은 시스템에 접근 불가능. **리포 로컬, 버전 관리된 아티팩트(코드, 마크다운, 스키마, 실행 계획)만이 에이전트가 볼 수 있는 것.**

"팀을 정렬시킨 Slack 토론"이 에이전트에게 발견 가능하지 않으면, 3개월 후 합류하는 신입 사원에게 알려지지 않는 것과 같다.

**기술 선택에 미치는 영향:**
- "지루한(boring)" 기술을 선호 — 안정적인 API, 높은 조합성, 훈련 데이터에 잘 표현됨
- 때로는 에이전트가 **외부 라이브러리의 기능 일부를 직접 재구현**하는 것이 더 효율적
- 예: 범용 `p-limit` 패키지 대신, OpenTelemetry 계측과 통합된 자체 동시성 헬퍼를 구현 → 100% 테스트 커버리지, 런타임 기대에 정확히 맞는 동작

## 아키텍처와 "취향(Taste)" 강제

문서만으로는 완전한 에이전트 생성 코드베이스의 일관성을 유지할 수 없다. **불변량(invariant)을 강제하되 구현을 마이크로매니지하지 않음으로써**, 에이전트가 빠르게 출력하면서도 기반을 훼손하지 않도록 함.

### 엄격한 아키텍처 모델

각 비즈니스 도메인이 고정된 레이어 세트로 나뉨:

```
Types → Config → Repo → Service → Runtime → UI
```

**의존성 방향이 엄격하게 검증됨.** 교차 관심사(auth, connectors, telemetry, feature flags)는 단일 명시적 인터페이스(Providers)를 통해서만 진입. 나머지는 **기계적으로 금지.**

> "이런 종류의 아키텍처는 보통 수백 명의 엔지니어가 있을 때 미루는 것이다. 코딩 에이전트와 함께라면, 이것은 조기 전제 조건이다: 제약이 부패나 아키텍처 드리프트 없는 속도를 가능하게 한다."

### "취향 불변량(Taste Invariants)"

- 구조화된 로깅 강제
- 스키마와 타입의 네이밍 규칙 강제
- 파일 크기 제한
- 플랫폼별 신뢰성 요구사항

**커스텀 린트의 에러 메시지가 교정 지시를 포함** → 에이전트 컨텍스트에 자동으로 주입되어 즉시 수정 가능.

> "인간 중심 워크플로우에서는 이런 규칙이 지나치게 꼼꼼하게 느껴질 수 있다. 에이전트와 함께라면, 이것은 배수기가 된다: 한 번 인코딩하면 모든 곳에 동시에 적용된다."

### 인간 취향의 피드백 루프

인간의 취향은 시스템에 지속적으로 피드백됨:
- 리뷰 코멘트 → 문서 업데이트
- 리팩토링 PR → 패턴 개선
- 사용자 버그 → 도구에 직접 인코딩
- **문서가 부족하면 → 규칙을 코드로 승격**

## 머지 철학의 변화

Codex의 throughput이 증가하면서, **전통적 엔지니어링 규범이 비생산적**이 됨.

- PR은 수명이 짧음
- 테스트 플레이크는 후속 실행으로 해결 (무한 블로킹 대신)
- 최소한의 블로킹 머지 게이트
- **에이전트 throughput이 인간 주의력을 훨씬 초과하는 시스템에서, 수정은 싸고 대기는 비쌈**

> "낮은 throughput 환경에서는 이것이 무책임할 것이다. 여기서는 종종 올바른 트레이드오프이다."

## 자율성의 증가

테스트, 검증, 리뷰, 피드백 처리, 복구가 시스템에 직접 인코딩되면서, 최근 Codex가 **새 기능을 end-to-end로 드라이브**할 수 있는 임계점을 넘었다:

단일 프롬프트로:
1. 현재 코드베이스 상태 검증
2. 보고된 버그 재현
3. 실패를 보여주는 비디오 녹화
4. 수정 구현
5. 앱을 구동하여 수정 검증
6. 해결을 보여주는 두 번째 비디오 녹화
7. PR 생성
8. 에이전트/인간 피드백에 응답
9. 빌드 실패 감지 및 교정
10. 판단이 필요할 때만 인간에게 에스컬레이션
11. 변경사항 머지

## 엔트로피와 가비지 컬렉션

완전한 에이전트 자율성은 새로운 문제를 도입한다. Codex는 리포에 이미 존재하는 패턴을 복제한다 — 불균일하거나 차선인 패턴까지도. 시간이 지나면 이것은 불가피하게 드리프트로 이어진다.

### 초기 시도: 수동 정리
팀이 매주 금요일(주당 20%)을 **"AI 슬롭" 청소**에 사용. 당연히 확장되지 않음.

### 해결: "골든 원칙" + 자동 가비지 컬렉션

"골든 원칙"을 리포에 직접 인코딩하고 반복 정리 프로세스를 구축:

1. **공유 유틸리티 패키지 선호** — 수동 헬퍼 대신 불변량을 중앙화
2. **데이터를 "YOLO 방식"으로 탐색하지 않음** — 경계를 검증하거나 타입 SDK에 의존하여, 에이전트가 추측된 형태 위에 구축하지 못하도록

**정기적 가비지 컬렉션 프로세스:**
- 백그라운드 Codex 작업들이 일탈을 스캔
- 품질 등급 업데이트
- 타겟 리팩토링 PR 생성
- 대부분 1분 이내에 리뷰 가능하고 자동 머지

> "기술 부채는 고금리 대출과 같다: 소액으로 지속적으로 갚는 것이 쌓여서 고통스러운 한 번에 처리하는 것보다 거의 항상 낫다. 인간의 취향은 한 번 포착되어, 이후 모든 코드 줄에 지속적으로 강제된다."

## 아직 배우고 있는 것들

이 전략은 지금까지 OpenAI 내부 출시와 채택에서 잘 작동했다. 아직 모르는 것:

- 완전히 에이전트 생성된 시스템에서 **아키텍처 일관성이 수 년에 걸쳐 어떻게 진화**하는가
- 인간의 판단이 가장 큰 레버리지를 추가하는 지점은 어디인가
- 모델이 더 capable해질수록 이 시스템이 어떻게 진화할 것인가

> "소프트웨어 구축에는 여전히 규율이 필요하지만, 그 규율은 코드가 아닌 **스캐폴딩**에서 나타난다."

## Soo에게 의미 있는 이유

이것은 이론이 아닌 **100만 줄 코드의 실제 프로덕션 경험**이다. AInD 컨설팅의 가장 강력한 레퍼런스:

1. **"코드 0줄" 철학** — 극단적이지만, 에이전트 시대의 방향성을 보여줌
2. **엔지니어당 3.5 PR/일** — 구체적 생산성 수치
3. **AGENTS.md = 목차, docs/ = 본문** — 리포 지식 관리의 구체적 패턴
4. **아키텍처 제약이 속도의 전제 조건** — 한국 기업에서 흔히 "나중에 할 것"으로 미루는 아키텍처 규칙이, 에이전트 시대에는 첫 날부터 필요
5. **가비지 컬렉션** — "한국에서 AI 프로젝트가 망하는 이유"의 직접적 해결책
6. **6시간 연속 에이전트 실행** — Holocron이 모니터링해야 할 대상
