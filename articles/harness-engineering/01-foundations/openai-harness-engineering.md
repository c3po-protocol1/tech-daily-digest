## 배경

OpenAI가 Codex를 사용하여 **빈 Git 리포지토리에서 대규모 애플리케이션을 완전히 새로 구축**한 실전 경험을 공유한 보고서다. 단순한 코드 생성을 넘어, 에이전트 시대에 엔지니어의 역할이 어떻게 변화하는지에 대한 깊은 통찰을 제공한다.

## 핵심 내용

### 엔지니어의 역할 재정의

전통적 엔지니어는 코드를 직접 작성했다. 에이전트 시대의 엔지니어는 **에이전트가 효과적으로 작업할 수 있는 환경을 설계**하는 사람이다. 이것이 "harness engineering"의 본질이다.

구체적으로:
- **아키텍처 설계자** — 전체 시스템 구조를 정의하고, 에이전트가 따라야 할 패턴을 설정
- **품질 관문 설계자** — 에이전트의 출력을 자동으로 검증하는 체크포인트 설계
- **환경 큐레이터** — 에이전트가 참조할 문서, 예시, 규칙을 관리
- **리뷰어** — 에이전트가 생성한 코드의 최종 품질 보장

### 리포지토리 지식을 시스템 오브 레코드로

OpenAI가 발견한 가장 중요한 원칙 중 하나: **리포지토리 안의 문서가 진실의 원천(source of truth)**이어야 한다.

- 외부 대화, Slack 메시지, 구두 합의는 에이전트에게 전달되지 않음
- **AGENTS.md**, 아키텍처 문서, 코딩 규칙이 리포 안에 있어야 에이전트가 참조 가능
- 코드와 문서가 같은 리포에 공존하면 **항상 최신 상태 유지** 가능
- 리포 외부의 지식은 사실상 존재하지 않는 것과 같음

### 에이전트 가독성(Agent Legibility)

전통적 "가독성"은 인간이 읽기 좋은 코드를 의미했다. 에이전트 시대에는 **에이전트가 이해하기 좋은 구조**를 설계해야 한다.

- **명확한 디렉토리 구조** — 기능별로 분리된 폴더, 일관된 네이밍
- **자기 설명적 파일** — 파일 이름만으로 역할을 알 수 있도록
- **의존성 명시** — 파일 간 관계를 명확하게 문서화
- **패턴 일관성** — 한 모듈에서 성공한 패턴을 다른 모듈에 동일하게 적용

### 아키텍처와 "취향(Taste)" 강제

에이전트에게 자유를 주면 **코드 스타일과 구조가 세션마다 달라진다**. OpenAI는 이를 "taste"라고 부르며, 이를 강제하는 방법을 발견했다.

- **아키텍처 의사결정 기록(ADR)** — 왜 이 패턴을 선택했는지 문서화
- **금지 패턴 명시** — "이렇게 하지 마라"가 "이렇게 하라"만큼 중요
- **참조 구현** — "이 모듈처럼 만들어라"로 구체적 예시 제공
- **자동 검증** — 린트, 타입 체크, 아키텍처 테스트로 규칙 위반 자동 탐지

### 머지 철학의 변화

에이전트가 생성하는 코드의 **양과 속도**가 인간과 다르므로, 머지 철학도 달라져야 한다.

- 인간 시대: 신중한 리뷰 후 머지 (throughput 낮음)
- 에이전트 시대: **빠른 피드백 + 자동 검증 후 머지** (throughput 높음)
- 핵심: 리뷰의 초점이 "코드 한 줄"에서 "전체 기능의 동작"으로 이동

### 자율성의 단계

에이전트의 자율성은 단계적으로 높여야 한다:

| Level | 범위 | Harness 역할 |
|-------|------|-------------|
| 1 | 단일 함수/파일 수정 | 명확한 지시 + 자동 테스트 |
| 2 | 기능 단위 구현 | 아키텍처 제약 + 검증 루프 |
| 3 | 설계 결정 포함 자율 코딩 | 스펙 + 평가 에이전트 + 인간 리뷰 |

## Soo에게 의미 있는 이유

OpenAI가 직접 실천한 "에이전트 퍼스트" 개발 방법론이다. AInD 컨설팅에서 **"에이전트 시대의 엔지니어 역할"**을 설명할 때 가장 강력한 레퍼런스. 특히 "리포지토리 = 시스템 오브 레코드"와 "에이전트 가독성" 개념은 기업 교육에서 바로 사용 가능한 프레임워크다.
